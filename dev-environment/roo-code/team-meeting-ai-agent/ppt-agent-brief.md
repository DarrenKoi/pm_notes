# PPT Agent 제작 요청서

아래 내용을 기준으로 10분 발표용 PPT를 만들어줘.

## 제작 목표

- 발표 제목: 사내 LLM으로 시작하는 AI Agent 활용: Roo Code를 중심으로
- 발표 대상: 기획, 운영, 관리자
- 발표 시간: 10분
- 슬라이드 수: 본편 10장
- 목적: 회사 보안 환경에서 외부 상용 AI Agent를 그대로 쓰기 어려운 상황을 설명하고, 사내 LLM API와 Roo Code를 활용해 Agent형 작업 방식에 익숙해지는 방법을 소개한다.

## 전체 톤

- 개발자 전용 교육처럼 보이지 않게 만들어줘.
- 기획, 운영, 관리자도 이해할 수 있게 쉬운 업무 언어를 사용해줘.
- Roo Code를 만능 자동화 도구처럼 과장하지 말고, 사내 LLM API 기반으로 Agent와 일하는 방식을 익히는 현실적인 출발점으로 표현해줘.
- 보안 제약을 숨기지 말고, 오히려 안전한 사용 방식의 일부로 설명해줘.

## 회사 표준 레이아웃

각 슬라이드는 아래 구조를 기본으로 작성해줘.

1. 상단 중앙: 해당 슬라이드의 키 메시지 1문장
   - 1문장만 사용해줘.
   - 너무 길게 쓰지 말고 18~28자 수준의 짧은 요약으로 압축해줘.
   - 설명 문장을 상단에 몰아넣지 말아줘.
2. 본문 중앙: 2개 컬럼 구성
   - 왼쪽 컬럼: 현상, 문제점, As-is, 현재 한계
   - 오른쪽 컬럼: 해결책, 전략, 플랜, To-be, 적용 방향
3. 하단: 필요할 때만 짧은 take-away 또는 작은 흐름도 사용
   - 하단에는 긴 설명을 넣지 말아줘.
   - 본문 내용이 충분하면 하단은 비워도 돼.

## 디자인 방향

- 회사 내부 공유용 발표자료처럼 차분하고 전문적인 톤으로 만들어줘.
- 각 슬라이드의 상단 키 메시지는 짧고 크게 보여줘.
- 본문은 좌우 2컬럼을 기본으로 하고, 각 컬럼은 2~4개 bullet로 제한해줘.
- 왼쪽 컬럼은 회색/중립 톤, 오른쪽 컬럼은 강조색을 약하게 사용해 구분해줘.
- 장식적 이미지보다 구조도, 흐름도, 비교 표를 우선해줘.
- 너무 화려한 그라데이션이나 과한 아이콘 사용은 피하고, 읽기 쉬운 레이아웃을 우선해줘.

## 반드시 포함할 개념

- Assistant와 Agent의 차이
- Agent는 사용자를 대신해 작업을 진행하려는 주체라는 점
- Prompt Engineering에서 Context Engineering, Harness Engineering으로 발전하는 흐름
- Harness는 모델, 컨텍스트, 도구, 권한, 승인, 로그, 스킬을 묶는 실행/제어 환경이라는 점
- 회사 환경에서는 외부 상용 Agent/API 사용이 어렵고, DRM 원문 파일을 Agent가 직접 읽을 수 없다는 점
- 사내 LLM API와 Roo Code를 연결해 개인 PC에서 Agent형 작업 방식을 실험할 수 있다는 점
- 공유 가능한 txt/md 작업 메모를 기반으로 Roo Code와 협업하고, 결과를 PPT Agent에 넘기는 흐름
- `/grill`, `/grill-with-docs` 같은 스킬을 통해 발표/문서 논리를 점검할 수 있다는 점

## 주의할 표현

- "DRM 문서를 우회해서 읽는다"처럼 들리는 표현은 사용하지 말아줘.
- "Roo Code가 모든 업무를 자동화한다"는 식의 과장 표현은 피해주세요.
- "Claude Code, Codex, Gemini CLI는 쓸 수 없다"처럼 단정적으로 말하지 말고, "회사 정책상 직접 활용하기 어렵다"라고 표현해줘.
- 사내 endpoint, API Key, 모델 ID 같은 구체적인 내부 정보는 넣지 말아줘.

## 슬라이드 구성

### 1. 왜 지금 AI Agent인가

상단 키 메시지:
AI Agent는 일하는 방식의 변화다

왼쪽 컬럼 - 현상 / As-is:
- Claude Code, Codex, Gemini CLI 같은 Agent형 도구가 빠르게 등장
- 지금은 도구 이름과 기능 중심으로 관심이 집중됨
- 회사 정책상 외부 상용 Agent를 그대로 쓰기는 어려움

오른쪽 컬럼 - 대응 / To-be:
- 특정 도구보다 Agent와 일하는 방식에 먼저 익숙해져야 함
- 사내에서 가능한 범위의 실험 경로를 찾아야 함
- 작은 업무 초안부터 Agent형 작업 방식을 연습

시각화 제안:
왼쪽은 "도구 확산", 오른쪽은 "일하는 방식 적응"으로 나누는 2컬럼 구조

### 2. Assistant와 Agent는 무엇이 다른가

상단 키 메시지:
Assistant는 답변, Agent는 작업 진행

왼쪽 컬럼 - Assistant / As-is:
- 질문에 답하거나 초안을 제안
- 사용자가 결과를 복사하고 직접 적용
- 대화가 끝나면 업무 흐름도 끊기기 쉬움

오른쪽 컬럼 - Agent / To-be:
- 목표를 이해하고 필요한 정보를 확인
- 계획을 세우고 도구 사용을 제안
- 사용자 승인 아래 산출물까지 발전시킴

시각화 제안:
Assistant vs Agent 2열 비교표

### 3. Agent의 핵심은 "대신 일하기"다

상단 키 메시지:
Agent는 업무 흐름을 이어간다

왼쪽 컬럼 - 기존 방식 / As-is:
- 회의 메모, 자료, 초안이 흩어져 있음
- 목차, 질문, 수정안을 사람이 매번 다시 정리
- 산출물이 한 번에 완성되지 않으면 흐름이 끊김

오른쪽 컬럼 - Agent 협업 / To-be:
- 목표와 제약을 주면 중간 작업을 제안
- 메모 → 목차 → 예상 질문 → 슬라이드 원고로 이어감
- 반복 수정으로 산출물을 점진적으로 개선

시각화 제안:
회의 메모 → 발표 목차 → 예상 질문 → 슬라이드 원고 흐름도

### 4. Prompt에서 Context로

상단 키 메시지:
좋은 질문만으로는 부족하다

왼쪽 컬럼 - Prompt 중심 / As-is:
- 매번 배경과 제약을 다시 설명해야 함
- 질문 품질에 따라 답변 편차가 큼
- 업무 용어와 결정사항이 대화 밖에 흩어짐

오른쪽 컬럼 - Context 중심 / To-be:
- 목적, 용어, 결정사항을 재사용 가능하게 정리
- 모델이 같은 업무 맥락을 이어서 활용
- 반복 업무에서 일관된 결과를 얻기 쉬움

시각화 제안:
단발 질문과 축적된 컨텍스트의 차이를 2컬럼으로 비교

### 5. Context에서 Harness로

상단 키 메시지:
Harness는 안전한 실행 환경이다

왼쪽 컬럼 - 모델만 사용할 때 / As-is:
- 모델은 답변을 만들지만 업무 도구와 분리됨
- 파일, 승인, 로그, 권한 관리가 약함
- 결과를 실제 업무에 적용하는 책임이 사용자에게 몰림

오른쪽 컬럼 - Harness 사용 / To-be:
- 모델, 컨텍스트, 파일, 도구, 권한을 묶음
- 승인 절차와 로그로 안전성을 확보
- 답변 생성에서 실제 작업 흐름으로 확장

시각화 제안:
중앙의 LLM을 컨텍스트, 파일, 도구, 승인, 로그가 감싸는 구조도

### 6. 회사 환경의 현실적 제약

상단 키 메시지:
보안 제약 안에서 써야 한다

왼쪽 컬럼 - 제약 / As-is:
- 외부 상용 Agent/API 직접 활용이 어려움
- Local LLM API 연결은 가능해도 성능과 토큰 효율 한계가 있음
- DRM 적용 PPTX, DOCX, PDF 원문은 Agent가 직접 열람하기 어려움

오른쪽 컬럼 - 원칙 / To-be:
- 사내 LLM API 기준으로 사용 경로를 설계
- 공유 가능한 txt/md 작업 메모 중심으로 협업
- Agent 출력물은 초안으로 보고 사람이 최종 검토

시각화 제안:
"어려운 것"과 "가능한 방식"을 나누는 2컬럼 표

### 7. 가능한 구조: 사내 LLM API + Roo Code

상단 키 메시지:
Roo Code는 현실적인 Harness 예시다

왼쪽 컬럼 - 일반 도구 사용 한계 / As-is:
- 외부 모델 전제 도구는 회사 환경에 바로 맞지 않음
- 모델과 업무 파일이 분리되면 활용성이 낮음
- 토큰 정량을 금방 소모할 수 있음

오른쪽 컬럼 - Roo Code 활용 / To-be:
- 사내 OpenAI-compatible API를 연결
- VS Code 작업 폴더의 txt, md, 코드, 설정 파일을 활용
- Agent 협업 방식을 개인 PC에서 연습 가능

시각화 제안:
사내 LLM 서버 → OpenAI-compatible API → Roo Code → VS Code workspace 구조도

### 8. Roo Code로 가능한 현실적 업무 흐름

상단 키 메시지:
텍스트 메모에서 초안을 만든다

왼쪽 컬럼 - 입력 한계 / As-is:
- DRM 원문을 직접 넣는 방식은 부적절함
- 문서 전체를 넣으면 보안과 토큰 문제가 커짐
- 목적과 출력 형식이 불명확하면 결과 품질이 낮음

오른쪽 컬럼 - 권장 흐름 / To-be:
- 공유 가능한 핵심 내용을 txt/md 작업 메모로 정리
- Roo Code로 목적, 목차, 논리, 체크리스트를 발전
- Markdown 초안을 PPT Agent로 넘겨 슬라이드화

시각화 제안:
DRM 문서/회의 메모 → txt/md 작업 메모 → Roo Code → Markdown 초안 → PPT Agent → 최종 검토 흐름도

### 9. 스킬과 PPT Agent 연계

상단 키 메시지:
스킬로 생각을 점검하고 PPT로 넘긴다

왼쪽 컬럼 - 초안 작성의 문제 / As-is:
- 발표 논리의 빈틈을 스스로 발견하기 어려움
- 용어 정의와 전제가 흔들리기 쉬움
- PPT 제작까지 한 도구에 모두 맡기면 품질 관리가 어려움

오른쪽 컬럼 - 역할 분담 / To-be:
- `/grill`로 아이디어와 방향을 질문으로 검토
- `/grill-with-docs`로 작업 메모의 용어와 논리 점검
- Roo Code는 내용 구조화, PPT Agent는 슬라이드 제작 담당

시각화 제안:
Roo Code와 PPT Agent의 역할 분담 표

### 10. 결론: 지금 익숙해져야 할 일하는 방식

상단 키 메시지:
핵심은 Agent와 일하는 습관이다

왼쪽 컬럼 - 도구 중심 접근 / As-is:
- 특정 도구의 기능 목록에만 집중
- 한 번에 완성된 결과를 기대
- 모델 출력물을 검증 없이 사용하려는 위험

오른쪽 컬럼 - 업무 방식 전환 / To-be:
- 모델과 대화하며 목적과 제약을 구체화
- 컨텍스트를 쌓고 산출물을 반복 개선
- 사람이 최종 판단하며 Agent를 업무 파트너로 활용

시각화 제안:
Model과 대화 → Context 축적 → 초안 생성 → 사람 검토 → 개선 반복 순환도

## 제외할 내용

- 설치 상세 절차는 넣지 말아줘.
- 사내 LLM API 설정 상세값은 넣지 말아줘.
- `/grill`, `/grill-with-docs`의 긴 예시 프롬프트는 넣지 말아줘.
- 보안/검증 체크리스트 전체는 넣지 말아줘.
- 부록 슬라이드는 별도 요청서에서 따로 만들 예정이므로 본편 PPT에는 포함하지 말아줘.
