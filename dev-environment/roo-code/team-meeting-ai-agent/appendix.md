# 부록: Roo Code 팀 미팅 공유자료

이 부록은 발표 중 자세히 설명하지 않고, 발표 후 공유자료로 남긴다. A1~A3은 본편에서 압축한 개념을 더 풀어 쓴 보강 슬라이드이고, A4~A10은 직접 실험해보고 싶은 사람을 위한 실습/설정 자료다.

## A1. Assistant와 Agent 상세 비교

본편 2번에서 압축한 차이를 더 풀어 쓴다.

Assistant:

- 단발 대화 중심: 질문하면 답변이나 초안을 준다.
- 결과를 적용하고 검증하는 일은 사용자 몫이다.
- 대화 맥락이 누적되지 않아 매번 배경을 다시 설명해야 한다.
- 파일, 도구, 기록에 직접 접근하지 않는다.

Agent:

- 목표 수신 → 정보 확인 → 계획 → 도구 사용 → 산출물 순서로 일한다.
- 사용자 승인과 보안 경계 안에서만 동작한다.
- 필요한 중간 작업을 스스로 제안한다.
- 파일과 기록을 활용해 산출물을 점진적으로 발전시킨다.

## A2. Prompt → Context → Harness 진화 상세

본편 3번에서 압축한 3단계를 더 풀어 쓴다.

각 단계의 정의:

- **Prompt Engineering**: 질문을 잘 써서 답변 품질을 높이는 단계.
- **Context Engineering**: 목적, 용어, 의사결정, 참고자료를 모델이 다시 쓸 수 있게 축적하는 단계.
- **Harness Engineering**: 모델을 실제 작업 환경에 안전하게 연결하는 단계.

Harness가 묶는 것:

- 모델, 컨텍스트, 파일, 도구
- 권한, 승인 절차
- 로그, 스킬
- 답변 생성에서 실제 작업 흐름으로 확장

## A3. Agent 협업 업무 흐름 상세

본편 2번·5번에서 다룬 협업 흐름을 단계별로 풀어 쓴다.

사람이 하는 일:

- 공유 가능한 핵심 내용을 txt/md 작업 메모로 정리한다.
- 목적, 대상, 분량 같은 제약을 명시한다.
- 각 단계 결과를 검토하고 방향을 조정한다.
- 최종 사실관계와 보안을 검토한다.

Agent가 이어가는 일:

- 메모를 바탕으로 목적과 목차를 정리한다.
- 예상 질문을 도출하고 논리를 점검한다.
- 슬라이드별 원고나 체크리스트를 작성한다.
- 반복 수정으로 초안을 개선한다.

흐름:

```text
회의 메모
→ 공유 가능한 txt/md 작업 메모
→ Roo Code와 함께 구조화
→ Markdown 초안
→ PPT Agent로 슬라이드 제작
→ 사람이 최종 검토
```

## A4. Roo Code 설치 개요

1. VS Code를 설치한다.
2. Extensions에서 `Roo Code`를 검색해 설치한다.
3. VS Code Activity Bar에서 Roo Code 패널을 연다.
4. 사내 LLM API 설정을 연결한다.
5. 작업용 폴더를 열고 `.txt` 또는 `.md` 파일을 기준으로 시작한다.

상세 설정은 상위 문서를 참고한다.

- [사내 OpenAI-Compatible LLM 연결 가이드](../00_company_local_llm_setup.md)
- [설치와 첫 사용](../01_setup_and_first_use.md)

## A5. 사내 LLM API 설정 개요

권장 구조:

```text
Provider: OpenAI-compatible
Base URL: https://<company-llm-gateway>/v1
API Key: 사내 발급 토큰
Model: 사내 모델 ID
```

주의할 점:

- 외부 API Key를 넣지 않는다.
- 내부 endpoint, API Key, 모델 ID는 공개 자료에 쓰지 않는다.
- 토큰 정량이 있는 경우 긴 파일 전체를 무작정 넣지 않는다.
- 큰 작업은 작은 작업 메모 단위로 나눈다.

## A6. 텍스트 작업 메모 방식

DRM이 걸린 PPTX, DOCX, PDF 원문을 Agent가 직접 열람한다고 가정하지 않는다. 사용자가 공유 가능한 범위에서 핵심 내용을 텍스트 작업 메모로 정리한 뒤 사용한다.

권장 파일 형식:

- `.md`: 목차, 회의 메모, 발표 원고, 체크리스트
- `.txt`: 간단한 원문 메모, 복사 가능한 일부 내용 정리
- `.csv`: 표 형태의 비민감 데이터 요약

권장하지 않는 방식:

- 보안 문서 원문 전체를 그대로 넣기
- 내부 endpoint, 계정, 고객명, 민감 수치를 그대로 넣기
- 검증 없이 Agent 출력물을 공식 문서로 배포하기

## A7. 작업 메모 템플릿

적어야 할 내용:

- 작업 목적: 산출물, 대상자, 시간/분량
- 배경: 필요한 이유, 결정 사항, 미정 사항
- 핵심 내용: 사실, 요구사항, 참고 메모
- 원하는 출력: 슬라이드, 보고서, 체크리스트

품질을 높이는 기준:

- 보안상 언급 금지 내용을 명시한다.
- 외부 도구 사용 가능 여부를 명시한다.
- 모델이 직접 열람하면 안 되는 자료를 구분한다.
- 출력 형식을 처음부터 지정한다.

## A8. `/grill` 예시

발표 아이디어를 검토할 때:

```text
/grill
나는 팀 미팅에서 회사 환경에서 사용할 수 있는 AI Agent 활용 방식을 소개하려고 한다.
대상자는 기획, 운영, 관리자이고 발표 시간은 10분이다.
Roo Code를 사내 LLM API와 연결해 사용하는 흐름을 대표 예시로 들고 싶다.
논리의 빈틈, 보안상 위험한 표현, 청중이 오해할 지점을 질문으로 점검해줘.
```

기대 효과:

- 발표 목적이 흐릿한지 확인한다.
- 청중이 물을 질문을 미리 찾는다.
- 도구 소개가 과장되지 않도록 조정한다.

## A9. `/grill-with-docs`와 PPT Agent 연계

텍스트 작업 메모를 기준으로 발표 초안을 만들 때:

```text
/grill-with-docs
아래 작업 메모를 기준으로 10분 발표 초안을 만들고 싶다.
용어 정의가 흔들리는 부분, 보안상 조심해야 할 표현,
슬라이드 흐름에서 빠진 전제를 하나씩 질문해줘.

작업 메모:
<여기에 공유 가능한 txt/md 메모 붙여넣기>
```

역할 분담:

- Roo Code: 용어 정의 점검, 빠진 전제와 논리 흐름 점검, 슬라이드별 메시지와 발표자 노트 작성, Markdown 초안 정리
- PPT Agent: 디자인과 배치 담당, 발표용 PPTX 생성
- 사람: 보안 정책 위반 내용 제거, 사실관계 검증, 표현 수위 조정, 최종 배포 승인

## A10. 보안/검증 체크리스트

Roo Code에 입력하기 전:

- 공유 가능한 내용인지 확인했는가?
- 민감한 고객명, 계정, endpoint, API Key가 제거되었는가?
- 원문 전체가 아니라 필요한 핵심 메모만 남겼는가?
- 외부 도구 사용 가능 여부를 확인했는가?

Roo Code 결과를 사용하기 전:

- 사실관계를 사람이 확인했는가?
- 회사 정책과 어긋나는 표현은 없는가?
- Agent가 만든 내용과 사람의 판단을 구분했는가?
- PPT Agent로 넘길 때도 같은 보안 기준을 적용했는가?
