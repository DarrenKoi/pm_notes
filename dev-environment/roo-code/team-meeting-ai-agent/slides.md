# 사내 LLM으로 시작하는 AI Agent 활용: Roo Code를 중심으로

## 발표 정보

- 대상: 기획, 운영, 관리자
- 시간: 10분
- 형식: 본편 6장 + 공유용 부록 10장
- 핵심 메시지: 사내 LLM API와 Roo Code 같은 Harness를 활용해, 지금부터 Agent와 일하는 방식에 익숙해지자.

> 이 문서가 발표 내용의 source of truth다. 슬라이드 제작 명세(`ppt-agent-brief.md`)는 여기 내용을 레이아웃으로 옮긴 self-contained 사본이므로, 내용 변경은 항상 이 파일에서 먼저 한 뒤 brief에 반영한다. 본편에서 덜어낸 상세 내용은 부록(`appendix.md`)에 둔다.

## 1. 왜 지금 AI Agent인가

- **키 메시지**: 변화는 멀지 않다, 지금 익숙해지자
- **왼쪽 컬럼 — 현실 (왜 지금)**
  - Claude Code, Codex, Gemini CLI 같은 Agent형 도구가 사외에서 빠르게 확산
  - 사내 LLM은 보안 등 제약으로 사외보다 6~12개월 뒤처진 모델을 사용 중
  - 그래도 이런 AI 활용 방식이 사내에도 1년 이내 도래할 것으로 예측
  - 생산성 향상은 한 사람이 더 많은 업무를 맡는 방향으로 이어질 가능성
- **오른쪽 컬럼 — 대응 (그래서 우리는)**
  - 특정 도구보다 Agent와 일하는 방식에 먼저 익숙해져야 함
  - 사내에서 가능한 범위의 실험 경로를 찾아야 함
  - 작은 업무 초안부터 Agent형 작업 방식을 연습
- **시각 요소**: 없음 (2컬럼만)

### 발표자 노트

최근 Claude Code, Codex, Gemini CLI처럼 모델을 개발 환경이나 업무 도구에 연결하는 Agent형 도구들이 사외에서 빠르게 확산되고 있다. 우리 회사는 보안을 비롯한 여러 제약 때문에 사외보다 대략 6~12개월 뒤처진 LLM 모델을 쓰고 있다. 하지만 사외가 지금 AI를 활용하는 방식으로 나아갈 것은 분명하고, 그 흐름이 사내에도 1년 이내에 도래할 것으로 본다. 중요한 함의는 생산성 향상이다. 도구가 일을 도와줄수록 한 사람이 맡는 업무의 양과 범위는 늘어날 가능성이 높다. 그래서 도구 이름을 외우기보다, 지금부터 모델과 함께 일하는 방식 자체에 익숙해지는 것이 필요하다.

## 2. Assistant와 Agent — Agent는 "대신 일하기"다

> 기존 2번(차이)과 3번(대신 일하기)을 병합. 세부 비교는 부록 A1, 협업 흐름 상세는 부록 A3 참조.

- **키 메시지**: Assistant는 답변, Agent는 작업 진행
- **왼쪽 컬럼 — Assistant (답변 도구)**
  - 질문에 답하거나 초안을 제안
  - 결과를 복사·적용·검증하는 일은 사용자 몫
  - 대화가 끝나면 업무 흐름도 끊기기 쉬움
- **오른쪽 컬럼 — Agent (대신 일하는 주체)**
  - 목표를 이해하고 필요한 정보를 확인
  - 사용자 승인 아래 계획을 세우고 도구 사용을 제안
  - 메모 → 목차 → 예상 질문 → 원고로 산출물을 이어감
- **시각 요소**: 가로 흐름도 — 회의 메모 → 발표 목차 → 예상 질문 → 슬라이드 원고 (Agent가 이어가는 흐름)

### 발표자 노트

Assistant는 질문에 답하거나 초안을 제안하는 대화형 도구에 가깝다. 결과를 복사하고 적용하고 검증하는 일은 대부분 사용자가 직접 한다. Agent는 사용자의 목표를 받아 필요한 정보를 읽고, 계획을 세우고, 도구를 사용해 결과물까지 만들려고 한다. 단순히 더 긴 답변을 만드는 것이 아니라, 목표 달성에 필요한 중간 작업을 제안하고 실행하려 한다는 점이 핵심이다. 예를 들어 회의 메모를 바탕으로 발표 목차를 만들고, 예상 질문을 뽑고, 슬라이드별 원고를 정리하는 식이다. 회사 환경에서는 반드시 사용자 승인과 보안 경계 안에서 동작해야 한다.

## 3. Prompt → Context → Harness 진화

> 기존 4번(Prompt→Context)과 5번(Context→Harness)을 병합한 3단계 progression. 단계별 깊이는 부록 A2 참조.

- **키 메시지**: 질문에서 맥락으로, 맥락에서 실행 환경으로
- **3단 가로 구성 (예외 레이아웃 — As-is/To-be 대신 3블록 진행)**
  - **Prompt 중심**: 매번 배경·제약을 다시 설명 / 질문 품질에 답변이 좌우됨
  - **Context 중심**: 목적·용어·결정사항을 재사용 가능하게 축적 / 같은 맥락을 이어서 활용
  - **Harness 중심**: 모델·컨텍스트·파일·도구·권한·승인·로그를 묶어 안전하게 실행
- **시각 요소**: Prompt → Context → Harness 가로 화살표 진행 (각 단계 아래 핵심 1줄)

### 발표자 노트

처음에는 Prompt Engineering이 중요했다. 질문을 잘 쓰면 답변 품질이 좋아졌다. 하지만 업무가 복잡해질수록 매번 모든 배경과 제약을 설명하는 방식은 한계가 있다. 그래서 목적·용어·의사결정·참고자료를 모델이 다시 사용할 수 있는 형태로 쌓는 Context Engineering이 필요해진다. 여기서 한 걸음 더 나아간 것이 Harness다. Harness는 모델, 컨텍스트, 파일, 도구, 권한, 승인 절차, 로그, 스킬을 묶어주는 실행·제어 환경이다. 모델이 답변만 하는 것이 아니라 IDE, 파일, 터미널, 문서 초안, 체크리스트 같은 실제 작업 흐름에 안전하고 반복 가능하게 들어오려면 Harness가 필요하다.

## 4. 회사 환경의 현실적 제약

- **키 메시지**: 보안 제약 안에서 써야 한다
- **왼쪽 컬럼 — 어려운 것 (제약)**
  - 외부 상용 Agent/API 직접 활용이 어려움
  - Local LLM API 연결은 가능해도 성능과 토큰 효율 한계가 있음
  - DRM 적용 PPTX, DOCX, PDF 원문은 Agent가 직접 열람하기 어려움
- **오른쪽 컬럼 — 가능한 방식 (원칙)**
  - 사내 LLM API 기준으로 사용 경로를 설계
  - 공유 가능한 txt/md 작업 메모 중심으로 협업
  - Agent 출력물은 초안으로 보고 사람이 최종 검토
- **시각 요소**: 없음 (좌우 대비 표 역할)

### 발표자 노트

Claude Code나 Codex 같은 도구를 그대로 쓰면 좋겠지만, 회사 정책상 외부 상용 모델/API를 직접 활용하기는 어렵다. 일부 도구는 사내 Local LLM API에 연결할 수 있지만, 원래 설계된 모델과 다르기 때문에 성능, 토큰 효율, 안정성 측면에서 한계가 있다. 또한 PPTX, DOCX, PDF 같은 DRM 원문 파일은 Agent가 직접 열람하는 대상으로 보면 안 된다.

## 5. 사내 LLM API + Roo Code로 일하는 법

> 기존 7번(가능한 구조)과 8번(현실적 업무 흐름)을 병합. 설치·API 설정·작업 메모 상세는 부록 A4~A7 참조.

- **키 메시지**: Roo Code로 텍스트 메모에서 초안까지
- **왼쪽 컬럼 — 한계 (As-is)**
  - 외부 모델 전제 도구는 회사 환경에 바로 맞지 않음
  - DRM 원문을 직접 넣는 방식은 부적절하고 토큰 부담도 큼
  - 모델과 업무 파일이 분리되면 활용성이 낮음
- **오른쪽 컬럼 — Roo Code 활용 (To-be)**
  - 사내 OpenAI-compatible API를 VS Code에 연결
  - 공유 가능한 txt/md 작업 메모로 협업
  - 목적·목차·논리를 발전시켜 Markdown 초안 생성
- **시각 요소**: 가로 흐름도 (구조+흐름 통합)

```text
사내 LLM API → Roo Code (VS Code) → txt/md 작업 메모 → Markdown 초안 → PPT Agent → 사람 검토
```

### 발표자 노트

Roo Code는 모델 그 자체가 아니다. 사내 오픈소스 LLM 서버의 OpenAI-compatible API를 연결하고, VS Code 작업 폴더 안의 텍스트 파일, Markdown, 코드, 설정 파일을 바탕으로 Agent형 작업을 할 수 있게 돕는 Harness다. 보안 정책상 DRM 원문 파일을 직접 열람하지 않고, 사용자가 공유 가능한 범위의 핵심 내용을 텍스트 작업 메모로 정리해 사용한다. Roo Code는 그 메모를 바탕으로 목적을 명확히 하고, 목차를 만들고, 논리를 점검하고, 발표 초안이나 체크리스트를 만든다. 지금은 완성형 표준 도구라기보다 Agent와 함께 일하는 방식을 익히는 훈련 도구로 보는 것이 현실적이다.

---

## 6. 결론: 지금 익숙해져야 할 일하는 방식

- **키 메시지**: 핵심은 Agent와 일하는 습관이다
- **왼쪽 컬럼 — 도구 중심 접근 (As-is)**
  - 특정 도구의 기능 목록에만 집중
  - 한 번에 완성된 결과를 기대
  - 모델 출력물을 검증 없이 사용하려는 위험
- **오른쪽 컬럼 — 업무 방식 전환 (To-be)**
  - 모델과 대화하며 목적과 제약을 구체화
  - 컨텍스트를 쌓고 산출물을 반복 개선
  - 사람이 최종 판단하며 Agent를 업무 파트너로 활용
- **시각 요소**: 순환도 (화살표 루프) — 모델과 대화 → Context 축적 → 초안 생성 → 사람 검토 → (다시 대화로)

### 발표자 노트

Roo Code 하나를 완벽히 쓰는 것이 목표는 아니다. 중요한 것은 AI Agent와 일하는 방식에 익숙해지는 것이다. 지금부터 모델에게 업무 목적과 제약을 설명하고, 텍스트 컨텍스트를 쌓고, 초안을 함께 만들고, 사람이 검토해 개선하는 방식에 익숙해지면 나중에 본격적인 Agent 도구가 들어왔을 때 훨씬 빠르게 적응할 수 있다.
