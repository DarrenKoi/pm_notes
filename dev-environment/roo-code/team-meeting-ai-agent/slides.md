# 사내 LLM으로 시작하는 AI Agent 활용: Roo Code를 중심으로

## 발표 정보

- 대상: 기획, 운영, 관리자
- 시간: 10분
- 형식: 본편 10장 + 공유용 부록
- 핵심 메시지: 사내 LLM API와 Roo Code 같은 Harness를 활용해, 지금부터 Agent와 일하는 방식에 익숙해지자.

---

## 1. 왜 지금 AI Agent인가

### 슬라이드 메시지

AI Agent 도구는 이미 빠르게 등장하고 있고, 앞으로 일하는 방식에 직접 영향을 줄 가능성이 높다.

### 발표자 노트

최근 Claude Code, Codex, Gemini CLI처럼 모델을 개발 환경이나 업무 도구에 연결하는 Agent형 도구들이 빠르게 나오고 있다. 지금 당장 모든 도구를 회사에서 그대로 쓸 수 있는 것은 아니지만, 흐름 자체는 분명하다. 앞으로는 모델에게 단발성 질문을 하는 것을 넘어, 모델과 함께 업무를 진행하는 방식이 중요해진다.

---

## 2. Assistant와 Agent는 무엇이 다른가

### 슬라이드 메시지

Assistant는 답을 제안하고, Agent는 사용자를 대신해 작업을 진행하려고 한다.

### 발표자 노트

Assistant는 질문에 답하거나 초안을 제안하는 대화형 도구에 가깝다. 결과를 복사하고 적용하고 검증하는 일은 대부분 사용자가 직접 한다. Agent는 사용자의 목표를 받아 필요한 정보를 읽고, 계획을 세우고, 도구를 사용해 결과물까지 만들려고 한다. 회사 환경에서는 반드시 사용자 승인과 보안 경계 안에서 동작해야 한다.

---

## 3. Agent의 핵심은 "대신 일하기"다

### 슬라이드 메시지

Agent는 사용자의 업무 의도를 받아 파일, 도구, 기록을 활용해 산출물을 발전시킨다.

### 발표자 노트

Agent가 중요한 이유는 단순히 더 긴 답변을 만들기 때문이 아니다. 사용자가 목표와 제약을 주면, Agent는 그 목표를 달성하기 위해 필요한 중간 작업을 제안하고 실행하려고 한다. 예를 들어 회의 메모를 바탕으로 발표 목차를 만들고, 예상 질문을 뽑고, 슬라이드별 원고를 정리하는 식이다.

---

## 4. Prompt에서 Context로

### 슬라이드 메시지

좋은 질문만으로는 부족하고, 업무 배경과 제약을 모델이 다시 사용할 수 있게 정리해야 한다.

### 발표자 노트

처음에는 Prompt Engineering이 중요했다. 질문을 잘 쓰면 답변 품질이 좋아졌다. 하지만 업무가 복잡해질수록 매번 모든 배경과 제약을 설명하는 방식은 한계가 있다. 그래서 프로젝트 목적, 용어, 의사결정, 참고자료를 모델이 다시 사용할 수 있는 형태로 쌓는 Context Engineering이 필요해진다.

---

## 5. Context에서 Harness로

### 슬라이드 메시지

Harness는 모델이 실제 업무 환경에서 안전하고 반복 가능하게 일하도록 감싸는 실행 환경이다.

### 발표자 노트

Harness는 모델, 컨텍스트, 파일, 도구, 권한, 승인 절차, 로그, 스킬을 묶어주는 제어 환경이다. 모델이 답변만 하는 것이 아니라 IDE, 파일, 터미널, 문서 초안, 체크리스트 같은 실제 작업 흐름에 들어오려면 Harness가 필요하다.

---

## 6. 회사 환경의 현실적 제약

### 슬라이드 메시지

외부 상용 Agent를 그대로 쓰기 어렵고, DRM 원문 파일도 Agent가 직접 열람할 수 없다.

### 발표자 노트

Claude Code나 Codex 같은 도구를 그대로 쓰면 좋겠지만, 회사 정책상 외부 상용 모델/API를 직접 활용하기는 어렵다. 일부 도구는 사내 Local LLM API에 연결할 수 있지만, 원래 설계된 모델과 다르기 때문에 성능, 토큰 효율, 안정성 측면에서 한계가 있다. 또한 PPTX, DOCX, PDF 같은 DRM 원문 파일은 Agent가 직접 열람하는 대상으로 보면 안 된다.

---

## 7. 가능한 구조: 사내 LLM API + Roo Code

### 슬라이드 메시지

Roo Code는 사내 LLM API를 VS Code 작업 환경에 연결하는 현실적인 Agent Harness 예시다.

### 발표자 노트

Roo Code는 모델 그 자체가 아니다. 사내 오픈소스 LLM 서버의 OpenAI-compatible API를 연결하고, VS Code 작업 폴더 안의 텍스트 파일, Markdown, 코드, 설정 파일을 바탕으로 Agent형 작업을 할 수 있게 돕는 Harness다. 지금은 완성형 표준 도구라기보다 Agent와 함께 일하는 방식을 익히는 훈련 도구로 보는 것이 현실적이다.

```text
사내 오픈소스 LLM 서버
→ OpenAI-compatible API
→ Roo Code
→ VS Code workspace / txt / md / 사용자 승인
```

---

## 8. Roo Code로 가능한 현실적 업무 흐름

### 슬라이드 메시지

공유 가능한 텍스트 작업 메모를 만들고, Roo Code와 함께 초안을 발전시킨다.

### 발표자 노트

보안 정책상 DRM 원문 파일을 Agent가 직접 열람하지 않는다. 대신 사용자가 공유 가능한 범위의 핵심 내용을 텍스트 작업 메모로 정리한다. Roo Code는 그 메모를 바탕으로 목적을 명확히 하고, 목차를 만들고, 논리를 점검하고, 발표 초안이나 체크리스트를 만든다.

```text
DRM 문서 / 회의 메모
→ 사용자가 공유 가능한 핵심 내용을 txt/md 작업 메모로 정리
→ Roo Code와 함께 구조화
→ Markdown 발표 초안 생성
→ PPT Agent로 슬라이드 제작
→ 사람이 최종 검토
```

---

## 9. 스킬과 PPT Agent 연계

### 슬라이드 메시지

`/grill`, `/grill-with-docs`로 생각을 검토하고, 정리된 초안을 PPT Agent에 넘긴다.

### 발표자 노트

`/grill`은 발표 아이디어나 기획 방향을 질문으로 압박 검토하는 데 유용하다. `/grill-with-docs`는 기존 Markdown 문서나 작업 메모를 기준으로 용어, 전제, 논리 흐름을 맞추는 데 유용하다. Roo Code에서 초안의 내용과 구조를 만든 뒤, 사내에서 제공되는 PPT Agent에 넘겨 슬라이드 디자인과 배치를 맡기는 식으로 역할을 나눌 수 있다.

---

## 10. 결론: 지금 익숙해져야 할 일하는 방식

### 슬라이드 메시지

핵심은 도구가 아니라 모델과 대화하고, 컨텍스트를 쌓고, 산출물을 반복적으로 개선하는 습관이다.

### 발표자 노트

Roo Code 하나를 완벽히 쓰는 것이 목표는 아니다. 중요한 것은 AI Agent와 일하는 방식에 익숙해지는 것이다. 지금부터 모델에게 업무 목적과 제약을 설명하고, 텍스트 컨텍스트를 쌓고, 초안을 함께 만들고, 사람이 검토해 개선하는 방식에 익숙해지면 나중에 본격적인 Agent 도구가 들어왔을 때 훨씬 빠르게 적응할 수 있다.
