# PPT Agent 부록 제작 요청서

아래 내용을 기준으로 발표 후 공유할 부록 PPT를 따로 만들어줘. 본편 발표자료와 합치지 말고, 별도 파일처럼 구성해줘.

## 제작 목표

- 자료 제목: Roo Code 활용 부록: 설치, 작업 메모, 스킬 예시
- 용도: 본편 발표 후 공유하는 참고자료
- 발표 여부: 발표 중 상세 설명하지 않음
- 슬라이드 수: 부록 7장
- 대상: 기획, 운영, 관리자 중 Roo Code를 직접 실험해보고 싶은 사람
- 목적: 본편에서 다루지 않은 설치 개요, 사내 LLM API 연결 개요, txt/md 작업 메모 방식, 스킬 예시, PPT Agent 연계, 보안 체크리스트를 짧게 정리한다.

## 회사 표준 레이아웃

각 부록 슬라이드도 아래 구조를 기본으로 작성해줘.

1. 상단 중앙: 해당 슬라이드의 키 메시지 1문장
   - 1문장만 사용해줘.
   - 설명을 상단에 몰아넣지 말고 짧게 요약해줘.
2. 본문 중앙: 2개 컬럼 구성
   - 왼쪽 컬럼: 절차, 입력 항목, As-is, 확인 대상
   - 오른쪽 컬럼: 권장 방식, 주의사항, To-be, 검증 기준
3. 하단: 필요할 때만 작은 예시 또는 짧은 take-away 사용
   - 긴 예시 프롬프트는 본문 전체를 차지하지 않게 해줘.

## 전체 톤

- 매뉴얼처럼 바로 따라 할 수 있게 작성해줘.
- 개발자 전용 문서처럼 보이지 않게 쉬운 설명을 사용해줘.
- 내부 endpoint, API Key, 모델 ID 같은 실제 값은 절대 넣지 말아줘.
- 보안 정책을 우회하는 인상을 주지 말고, 정책 안에서 사용할 수 있는 흐름으로 설명해줘.

## 디자인 방향

- 본편보다 정보량은 조금 더 많아도 되지만, 각 슬라이드는 한 주제만 다뤄줘.
- 상단 키 메시지는 짧게 두고, 상세 내용은 좌우 2컬럼에 배치해줘.
- 체크리스트, 단계 목록, 2열 표, 간단한 흐름도를 우선 사용해줘.
- 발표용이라기보다 공유 후 읽는 자료이므로 각 컬럼 bullet은 3~5개까지 허용해줘.
- 너무 화려한 디자인보다 읽기 쉬운 내부 가이드 문서 톤으로 만들어줘.

## 주의할 표현

- "DRM 문서를 우회해서 읽는다"라는 표현은 사용하지 말아줘.
- "문서 원문 전체를 모델에 넣는다"는 식의 안내는 하지 말아줘.
- "외부 API를 사용한다"고 오해될 표현은 피하고, 사내 LLM API 기준으로 설명해줘.
- 구체적인 endpoint, API Key, 토큰값, 내부 모델 ID는 예시로도 넣지 말아줘.

## 부록 슬라이드 구성

### A1. Roo Code 설치 개요

상단 키 메시지:
VS Code에서 Roo Code를 시작한다

왼쪽 컬럼 - 설치 절차:
- VS Code 설치
- Extensions에서 Roo Code 설치
- Roo Code 패널 열기
- 작업용 폴더 열기

오른쪽 컬럼 - 시작 조건:
- 사내 LLM API 설정 필요
- txt/md 파일 기준으로 첫 작업 시작
- 설치 상세는 별도 가이드 참고
- 민감 정보는 설정 화면 밖에 노출하지 않음

시각화 제안:
설치 → 연결 → 작업 폴더 열기 → 첫 요청 순서의 단계형 흐름

### A2. 사내 LLM API 설정 개요

상단 키 메시지:
OpenAI-compatible Provider로 연결한다

왼쪽 컬럼 - 설정 항목:
- Provider: OpenAI-compatible
- Base URL: 사내 LLM gateway의 `/v1` endpoint
- API Key: 사내 발급 토큰
- Model: 사내 지정 모델 ID

오른쪽 컬럼 - 주의사항:
- 실제 값은 공유자료에 쓰지 않음
- 외부 API Key를 넣지 않음
- 토큰 정량이 있으면 큰 파일 투입을 피함
- 연결 후 짧은 테스트 요청으로 확인

시각화 제안:
설정 항목과 주의사항을 나눈 2열 표

### A3. txt/md 작업 메모 방식

상단 키 메시지:
공유 가능한 텍스트만 사용한다

왼쪽 컬럼 - 입력 가능한 형태:
- `.md`: 목차, 회의 메모, 발표 원고, 체크리스트
- `.txt`: 간단한 원문 메모, 일부 내용 정리
- `.csv`: 표 형태의 비민감 데이터 요약
- 사용자가 직접 정리한 작업 메모

오른쪽 컬럼 - 사용 원칙:
- DRM 원문을 직접 넣지 않음
- 원문 전체가 아니라 핵심 내용만 정리
- 고객명, 계정, endpoint 등 민감 정보 제거
- 공유 가능한 범위인지 먼저 확인

시각화 제안:
DRM 문서/회의 메모 → 공유 가능한 작업 메모 → Roo Code 작업 흐름도

### A4. 작업 메모 템플릿

상단 키 메시지:
목적과 제약을 먼저 적는다

왼쪽 컬럼 - 적어야 할 내용:
- 작업 목적: 산출물, 대상자, 시간/분량
- 배경: 필요한 이유, 결정 사항, 미정 사항
- 핵심 내용: 사실, 요구사항, 참고 메모
- 원하는 출력: 슬라이드, 보고서, 체크리스트

오른쪽 컬럼 - 품질을 높이는 기준:
- 보안상 언급 금지 내용을 명시
- 외부 도구 사용 가능 여부를 명시
- 모델이 직접 열람하면 안 되는 자료를 구분
- 출력 형식을 처음부터 지정

시각화 제안:
작업 메모 템플릿을 좌우 입력 양식으로 표현

### A5. `/grill` 예시

상단 키 메시지:
질문으로 기획의 빈틈을 찾는다

왼쪽 컬럼 - 활용 상황:
- 발표 목적이 흐릿할 때
- 청중의 예상 질문을 찾고 싶을 때
- 보안상 위험한 표현을 점검할 때
- 도구 소개가 과장되는지 확인할 때

오른쪽 컬럼 - 짧은 예시:
```text
/grill
10분 팀 미팅에서 AI Agent 활용 방식을 소개하려고 한다.
대상자는 기획, 운영, 관리자다.
논리의 빈틈과 오해할 지점을 질문으로 점검해줘.
```

시각화 제안:
아이디어 → 질문 → 보완 → 발표 초안 흐름

### A6. `/grill-with-docs`와 PPT Agent 연계

상단 키 메시지:
논리를 맞춘 뒤 PPT로 넘긴다

왼쪽 컬럼 - Roo Code 역할:
- 작업 메모 기준으로 용어 정의 점검
- 빠진 전제와 논리 흐름 점검
- 슬라이드별 메시지와 발표자 노트 작성
- Markdown 초안 정리

오른쪽 컬럼 - PPT Agent와 사람 역할:
- PPT Agent는 디자인과 배치 담당
- 사람은 보안과 사실관계 최종 검토
- 배포 전 표현 수위 조정
- 역할을 나눠 품질과 안전성 확보

시각화 제안:
Roo Code / PPT Agent / 사람의 역할 분담 표

### A7. 보안/검증 체크리스트

상단 키 메시지:
Agent 결과물은 초안이다

왼쪽 컬럼 - 입력 전 체크:
- 공유 가능한 내용인지 확인
- 고객명, 계정, endpoint, API Key 제거
- 원문 전체가 아니라 핵심 메모만 사용
- 외부 도구 사용 가능 여부 확인

오른쪽 컬럼 - 출력 후 체크:
- 사실관계 확인 후 배포
- 회사 정책과 어긋나는 표현 제거
- Agent 판단과 사람 판단을 구분
- PPT Agent로 넘길 때도 같은 기준 적용

시각화 제안:
입력 전 체크 / 출력 후 체크 2열 체크리스트
