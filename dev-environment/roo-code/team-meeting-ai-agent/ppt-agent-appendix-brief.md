# PPT Agent 부록 제작 요청서

아래 슬라이드 명세대로 부록 PPT를 만들어줘. 내용은 이미 다 정해져 있으니, 그대로 슬라이드로 옮기기만 하면 돼. 본편과 합치지 말고 별도 파일로 구성하고, 내용을 새로 구성하거나 추가하지 말 것.

- 제목: Roo Code 활용 부록: 개념 보강, 설치, 작업 메모, 스킬 예시
- 분량: 부록 10장 (본편과 별도 파일)
- 구성: A1~A3 개념 보강(본편 압축분 상세) / A4~A10 실습·설정

## 공통 레이아웃 (모든 슬라이드 동일)

- 상단 중앙: 키 메시지 1문장, 짧게. 그대로 사용.
- 본문: 좌우 2컬럼. 각 컬럼 제목과 bullet은 명세 그대로.
- 하단: 각 슬라이드에 지정된 시각 요소(흐름도/표/체크리스트)만 배치. 지정 없으면 비움.
- 부록은 읽는 자료이므로 컬럼당 bullet 3~5개 허용. 단, 명세에 적힌 문장만 쓰고 늘리지 말 것.
- 코드블록(예시 프롬프트)은 명세에 적힌 텍스트를 그대로 넣고, 본문 전체를 차지하지 않게 작게 배치.

## A1. Assistant와 Agent 상세 비교

- 키 메시지: Agent는 승인 경계 안에서 일한다
- 왼쪽 박스 제목: Assistant
  - 단발 대화 중심: 질문하면 답변·초안을 줌
  - 결과 적용·검증은 사용자 몫
  - 대화 맥락이 누적되지 않음
  - 파일·도구·기록에 직접 접근하지 않음
- 오른쪽 박스 제목: Agent
  - 목표 수신 → 정보 확인 → 계획 → 도구 사용 → 산출물
  - 사용자 승인과 보안 경계 안에서만 동작
  - 필요한 중간 작업을 스스로 제안
  - 파일·기록을 활용해 산출물을 발전
- 하단 시각 요소: 없음 (좌우 상세 비교표 역할)

## A2. Prompt → Context → Harness 진화 상세

- 키 메시지: 세 단계로 발전하는 활용 방식
- 왼쪽 박스 제목: 각 단계 정의
  - Prompt Engineering: 질문을 잘 써서 답 품질을 높임
  - Context Engineering: 목적·용어·결정을 재사용 가능하게 축적
  - Harness Engineering: 모델을 실제 작업 환경에 안전하게 연결
- 오른쪽 박스 제목: Harness가 묶는 것
  - 모델, 컨텍스트, 파일, 도구
  - 권한, 승인 절차
  - 로그, 스킬
  - 답변 생성 → 실제 작업 흐름으로 확장
- 하단 시각 요소: 중심-위성 구조도 — 중앙 LLM을 컨텍스트·파일·도구·승인·로그가 둘러쌈

## A3. Agent 협업 업무 흐름 상세

- 키 메시지: 메모에서 초안까지 단계별로 이어간다
- 왼쪽 박스 제목: 사람이 하는 일
  - 공유 가능한 핵심 내용을 txt/md 메모로 정리
  - 목적·대상·분량 등 제약 명시
  - 각 단계 결과를 검토하고 방향 조정
  - 최종 사실관계·보안 검토
- 오른쪽 박스 제목: Agent가 이어가는 일
  - 메모 기반으로 목적·목차 정리
  - 예상 질문 도출, 논리 점검
  - 슬라이드별 원고·체크리스트 작성
  - 반복 수정으로 초안 개선
- 하단 시각 요소: 6단계 가로 흐름도
  - 회의 메모 → txt/md 메모 → Roo Code 구조화 → Markdown 초안 → PPT Agent → 사람 검토

## A4. Roo Code 설치 개요

- 키 메시지: VS Code에서 Roo Code를 시작한다
- 왼쪽 박스 제목: 설치 절차
  - VS Code 설치
  - Extensions에서 Roo Code 설치
  - Roo Code 패널 열기
  - 작업용 폴더 열기
- 오른쪽 박스 제목: 시작 조건
  - 사내 LLM API 설정 필요
  - txt/md 파일 기준으로 첫 작업 시작
  - 설치 상세는 별도 가이드 참고
  - 민감 정보는 설정 화면 밖에 노출하지 않음
- 하단 시각 요소: 가로 단계 흐름도
  - 설치 → 연결 → 작업 폴더 열기 → 첫 요청

## A5. 사내 LLM API 설정 개요

- 키 메시지: OpenAI-compatible Provider로 연결한다
- 왼쪽 박스 제목: 설정 항목
  - Provider: OpenAI-compatible
  - Base URL: 사내 LLM gateway의 `/v1` endpoint
  - API Key: 사내 발급 토큰
  - Model: 사내 지정 모델 ID
- 오른쪽 박스 제목: 주의사항
  - 실제 값은 공유자료에 쓰지 않음
  - 외부 API Key를 넣지 않음
  - 토큰 정량이 있으면 큰 파일 투입을 피함
  - 연결 후 짧은 테스트 요청으로 확인
- 하단 시각 요소: 없음 (좌우 2열 표 역할)

## A6. txt/md 작업 메모 방식

- 키 메시지: 공유 가능한 텍스트만 사용한다
- 왼쪽 박스 제목: 입력 가능한 형태
  - `.md`: 목차, 회의 메모, 발표 원고, 체크리스트
  - `.txt`: 간단한 원문 메모, 일부 내용 정리
  - `.csv`: 표 형태의 비민감 데이터 요약
  - 사용자가 직접 정리한 작업 메모
- 오른쪽 박스 제목: 사용 원칙
  - DRM 원문을 직접 넣지 않음
  - 원문 전체가 아니라 핵심 내용만 정리
  - 고객명, 계정, endpoint 등 민감 정보 제거
  - 공유 가능한 범위인지 먼저 확인
- 하단 시각 요소: 가로 흐름도
  - DRM 문서/회의 메모 → 공유 가능한 작업 메모 → Roo Code

## A7. 작업 메모 템플릿

- 키 메시지: 목적과 제약을 먼저 적는다
- 왼쪽 박스 제목: 적어야 할 내용
  - 작업 목적: 산출물, 대상자, 시간/분량
  - 배경: 필요한 이유, 결정 사항, 미정 사항
  - 핵심 내용: 사실, 요구사항, 참고 메모
  - 원하는 출력: 슬라이드, 보고서, 체크리스트
- 오른쪽 박스 제목: 품질을 높이는 기준
  - 보안상 언급 금지 내용을 명시
  - 외부 도구 사용 가능 여부를 명시
  - 모델이 직접 열람하면 안 되는 자료를 구분
  - 출력 형식을 처음부터 지정
- 하단 시각 요소: 없음 (좌우 입력 양식 표 역할)

## A8. `/grill` 예시

- 키 메시지: 질문으로 기획의 빈틈을 찾는다
- 왼쪽 박스 제목: 활용 상황
  - 발표 목적이 흐릿할 때
  - 청중의 예상 질문을 찾고 싶을 때
  - 보안상 위험한 표현을 점검할 때
  - 도구 소개가 과장되는지 확인할 때
- 오른쪽 박스 제목: 짧은 예시 (코드블록으로 그대로)
  ```text
  /grill
  10분 팀 미팅에서 AI Agent 활용 방식을 소개하려고 한다.
  대상자는 기획, 운영, 관리자다.
  논리의 빈틈과 오해할 지점을 질문으로 점검해줘.
  ```
- 하단 시각 요소: 가로 흐름도
  - 아이디어 → 질문 → 보완 → 발표 초안

## A9. `/grill-with-docs`와 PPT Agent 연계

- 키 메시지: 논리를 맞춘 뒤 PPT로 넘긴다
- 왼쪽 박스 제목: Roo Code 역할
  - 작업 메모 기준으로 용어 정의 점검
  - 빠진 전제와 논리 흐름 점검
  - 슬라이드별 메시지와 발표자 노트 작성
  - Markdown 초안 정리
- 오른쪽 박스 제목: PPT Agent와 사람 역할
  - PPT Agent는 디자인과 배치 담당
  - 사람은 보안과 사실관계 최종 검토
  - 배포 전 표현 수위 조정
  - 역할을 나눠 품질과 안전성 확보
- 하단 시각 요소: 3칸 역할 분담 표
  - Roo Code: 내용/논리 점검 / PPT Agent: 디자인·배치 / 사람: 최종 검토

## A10. 보안/검증 체크리스트

- 키 메시지: Agent 결과물은 초안이다
- 왼쪽 박스 제목: 입력 전 체크
  - 공유 가능한 내용인지 확인
  - 고객명, 계정, endpoint, API Key 제거
  - 원문 전체가 아니라 핵심 메모만 사용
  - 외부 도구 사용 가능 여부 확인
- 오른쪽 박스 제목: 출력 후 체크
  - 사실관계 확인 후 배포
  - 회사 정책과 어긋나는 표현 제거
  - Agent 판단과 사람 판단을 구분
  - PPT Agent로 넘길 때도 같은 기준 적용
- 하단 시각 요소: 2열 체크리스트 (입력 전 / 출력 후)
