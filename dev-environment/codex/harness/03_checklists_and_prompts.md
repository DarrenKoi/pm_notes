# 체크리스트와 프롬프트 템플릿

> 하네스는 매번 새로 고민하는 것이 아니라, 반복 가능한 체크리스트로 고정할수록 효과가 크다.

## 공통 체크리스트

작업 전:

- `git status --short`로 기존 변경분을 확인했다.
- `-C`로 작업 루트를 좁혔다.
- 수정 금지 영역을 프롬프트에 적었다.
- 필요한 검증 명령을 프롬프트에 포함했다.
- 네트워크/외부 서비스가 필요한지 미리 판단했다.

작업 중:

- Codex가 관련 파일을 먼저 읽고 판단했다.
- 변경 범위가 요청과 맞다.
- 관련 없는 포맷팅이나 대량 리팩터링이 없다.
- 실패한 명령을 우회하지 않고 원인을 설명한다.

작업 후:

- 변경 요약이 명확하다.
- 검증 명령과 결과가 남아 있다.
- 남은 테스트 공백이나 외부 의존성 한계가 기록됐다.
- 필요하면 `codex review --uncommitted`를 실행했다.

## 기능 구현 템플릿

```text
<module-path> 안에서만 작업해줘.
목표는 <feature> 구현이야.
API/DB 계약은 <constraint>를 지켜줘.
먼저 관련 파일과 기존 패턴을 확인한 뒤 구현해줘.
완료 후 <validation-command>를 실행하고 결과를 요약해줘.
```

예:

```text
web-development/python/flask/job-scheduler 안에서만 작업해줘.
목표는 작업 목록에 상태 필터를 추가하는 거야.
기존 API 응답 필드는 깨지지 않게 유지해줘.
먼저 frontend와 backend의 관련 파일을 확인한 뒤 구현해줘.
완료 후 backend smoke test와 frontend npm run lint/typecheck를 실행해줘.
```

## 버그 수정 템플릿

```text
<module-path>에서 <symptom> 문제가 있어.
수정 전에 재현 경로, 관련 파일, 의심 원인을 정리해줘.
그 다음 최소 변경으로 고치고, 재발 방지 테스트를 추가하거나 기존 테스트를 실행해줘.
외부 서비스가 필요하면 mock 가능한 부분과 실제 검증이 필요한 부분을 분리해서 설명해줘.
```

## 리팩터링 템플릿

```text
<module-path>의 <target>만 리팩터링해줘.
동작 변경은 하지 말고, 공개 API/입출력 형식은 유지해줘.
먼저 현재 테스트와 검증 명령을 찾고, 변경 후 같은 검증을 실행해줘.
리팩터링 범위가 커질 것 같으면 구현 전에 파일별 변경 계획을 제시해줘.
```

## 코드 리뷰 템플릿

```text
현재 변경분을 코드 리뷰 관점으로 봐줘.
버그, 회귀 위험, 보안 문제, 누락된 테스트를 우선순위 순서로 지적해줘.
스타일 취향이나 큰 리팩터링 제안은 실제 위험이 있을 때만 말해줘.
```

## 문서 작성 템플릿

```text
<topic>에 대한 문서를 <folder> 아래에 작성해줘.
기존 문서 스타일과 frontmatter 형식을 맞춰줘.
상대 링크를 사용하고, 관련 README 인덱스도 업데이트해줘.
실행 명령이나 설정 예시는 복사해서 쓸 수 있게 fenced code block으로 작성해줘.
```

## 위험 작업 템플릿

```text
이 작업은 위험할 수 있으니 바로 수정하지 말고 먼저 계획만 세워줘.
영향 파일, 되돌리는 방법, 필요한 승인, 검증 명령을 정리해줘.
내가 승인하기 전에는 파일을 수정하지 마.
```

위험 작업 예:

- DB schema 변경
- 인증/권한 로직 변경
- 대량 파일 이동
- 배포 설정 변경
- secret/env 처리 변경
- 외부 API 호출 방식 변경

## 하네스 강도 선택표

| 작업 | 권장 하네스 |
|------|-------------|
| 개념 학습, 코드 구조 파악 | `read-only`, 수정 금지 프롬프트 |
| 작은 문서 수정 | `workspace-write`, 검증은 링크/README 확인 |
| 작은 버그 수정 | `workspace-write`, 관련 테스트 필수 |
| UI 변경 | `workspace-write`, lint/typecheck/browser 확인 |
| RAG/LLM 로직 변경 | `workspace-write`, 샘플 eval 또는 고정 입력 테스트 |
| 보안/배포 변경 | 먼저 `read-only` 계획, 승인 후 구현 |
| 대량 리팩터링 | 파일 단위 계획, 작은 커밋 단위, review 필수 |

## 금지에 가까운 조합

- 저장소 루트 전체 + 모호한 요청 + `danger-full-access`
- 네트워크 허용 + secret 환경 변수 전체 상속
- MCP 도구 다수 노출 + 승인 없음
- 테스트 없는 대규모 리팩터링
- 실패한 검증 명령을 무시하고 완료 처리

## 완료 조건 템플릿

```text
완료 조건:
- 요구사항을 만족하는 코드/문서 변경이 있다.
- 관련 검증 명령을 실행했다.
- 실패한 검증이 있으면 원인과 미해결 사유를 남겼다.
- 변경 범위가 요청한 폴더/파일을 벗어나지 않았다.
- 최종 답변에 변경 파일과 검증 결과를 요약했다.
```
