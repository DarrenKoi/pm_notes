# Quick Win 5선 상세 카드 (v0.1 드래프트)

> ITC AIX 로드맵 [Ch.7](./itc-aix-roadmap-outline.md#chapter-7-1년차-quick-win-5선-12p) 본문 backing 문서.
> 본문 슬라이드는 표 1장 + 핵심 메시지 1장으로 압축. 카드 상세는 Appendix B 또는 사내 비공개 문서로 격리.

## 공통 규칙 (Recap)

- **본 적용 정의 (QW #1·#2·#3, 3-gate):** G1 범위(2개 이상 stream/팀 일상 사용) + G2 행동 변화(수용 ≥70%) + G3 운영 안정성(2주 자동 운영, SLO·모니터링).
- **PoC 졸업 정의 (QW #4·#5):** 본 적용 아님. 시그널 검증·사이클 1회전 등 별도 기준.
- **Baseline 측정 의무:** Hero KPI (★1·3·5) 모두 6개월 PoC 단계에서 baseline 확보.
- **책임 표기:** AX Part / 실행 TF / 전사 TF (전사) — 3-tier 거버넌스 ([ADR-0001](./docs/adr/0001-aix-tf-ax-part-boundary.md)).

| 속성 | 값 |
|------|-----|
| **Stream** | 1 (의뢰 대응) |
| **주 능력** | 분류·선별 |
| **Hero KPI** | 의뢰 평균 TAT ↓30% (1년) — Layer 1 ★1 |
| **보조 KPI** | 분류 정확도 ≥85%, 자동 라우팅 비율 |
| **Baseline 출처** | A1 (T1, 사내 ticketing) |

### 6개월 PoC 졸업 기준
- **Baseline 측정 완료:** 최근 12개월 의뢰 TAT 분포 (평균·중앙값·p90) 산출
- **분류 모델:** [SLOT: 선정 모델 종류] 기반 다중 분류 (의뢰 유형 N개) — 정확도 ≥85% 달성
- **End-to-end demo:** 단일 팀의 의뢰 채널에서 raw 의뢰 → 자동 분류 → 담당자 추천 까지 가능

### 12개월 본 적용 기준 (3-gate)
| Gate | 충족 조건 |
|------|----------|
| **G1 범위** | 산하 6팀 중 ≥3팀의 의뢰 채널에 연결, 일상 사용 |
| **G2 행동 변화** | AI 분류 결과가 default routing — 사람 default 수용률 ≥70% |
| **G3 운영 안정성** | 월간 정확도 모니터링 정착, on-call rota 운영, 2주 무감시 자동 운영 |
| **Outcome** | Hero KPI: TAT ↓30% 측정 완료 |

### 데이터 의존성
- 사내 ticketing/협업툴 raw 의뢰 데이터 (3년치) — 학습용
- 과거 처리 사례 (의뢰 → 담당자 → 처리시간) — labeling용
- [SLOT: 시스템 접근 권한 owner — DT 또는 ITC 자체 시스템?]

### 책임 (3-tier)
- **AX Part:** 모델 선정·평가 harness 설계, 트레이닝 데이터 큐레이션, 6팀 통합 운영 설계
- **실행 TF:** 팀별 의뢰 채널 연결, 분류 라벨 검증, 사용자 피드백 수집
- **전사 TF:** LLM 스택 (Kimi/Qwen) 제공, GPU 자원, [SLOT: 의뢰 ticketing 시스템 owner라면 데이터 접근]

### 리스크
- 의뢰 유형의 long-tail (드문 유형) 처리 정확도 저하 → 사람 fallback 절차 필수
- 팀별 의뢰 양식·필드 상이 → 통합 전처리 부담

| **보조 KPI** | 초안 채택률, 사용자 만족도 |

### 6개월 PoC 졸업 기준
- **Baseline 측정 완료:** 분석 보고서 작성시간 — [SLOT: 측정 방법 — 자기보고 / 시스템 로그 / 시간기록?]. 최소 30건 sample
- **초안 모델:** RAG (사내 문헌·과거 사례) + LLM 보고서 템플릿화. 초안 채택률 ≥50%
- **End-to-end demo:** 분석 요청 → 자동 초안 → 사람 편집 → 최종 보고서 시퀀스

| Gate | 충족 조건 |
|------|----------|

| **G2 행동 변화** | 분석가의 보고서 작성이 *처음부터 → 초안 편집*으로 전환, 채택률 ≥70% |
| **G3 운영 안정성** | RAG knowledge base 정례 갱신 (월간), 모델 재평가 분기 |

- 과거 분석 보고서 N년치 — RAG 입력 + 양식 학습
- 사내 문헌·실험 데이터 (Track 3과 일부 공유)
- [SLOT: 보고서 양식 표준이 팀별로 통일되어 있나?]

### 책임 (3-tier)
- **AX Part:** RAG 파이프라인 구축 (명시지 기반, Enabler 5.3a와 연동), 초안 prompt 표준 설계
- **실행 TF:** 팀별 보고서 양식 큐레이션, 초안 채택 피드백 수집
- **전사 TF:** RAG 표준 인프라 (BGE-M3 임베딩, vector store), LLM 스택

### 리스크
- 사람마다 *"좋은 보고서"* 기준 상이 → 채택률 측정의 주관성
- 작성시간 baseline이 정직하게 측정 안 되면 ↓40% 검증 불가

## QW #3 — 검측 이미지 자동 분류·선별 ★ Hero

| 속성 | 값 |
|------|-----|
| **Stream** | 3 (계측 데이터 생산) |
| **주 능력** | 분류·선별 |
| **Hero KPI** | 검측 → 공정 의사결정 lead time ↓30% (1년) — Layer 1 ★3 |
| **보조 KPI** | 이미지 자동 분류 정확도 ≥90%, 우선순위 큐 정확도 |
| **Baseline 출처** | PoC 측정 (난이도 ↑, fallback 옵션 존재) |

### 6개월 PoC 졸업 기준
- **Baseline 측정 완료 (gate):** 검측 결과 도출 → 공정 변경 의사결정까지 cycle time. [SLOT: cross-system tracing 가능한가? 가능하지 않으면 fallback — E1 hero 승격]
- **분류 모델:** VLM 또는 specialized vision model — 검측 이미지 자동 분류 정확도 ≥90%
- **End-to-end demo:** raw 검측 이미지 → 자동 분류 → 우선순위 큐 → 분석가 검토 시퀀스

### 12개월 본 적용 기준 (3-gate)
| Gate | 충족 조건 |
|------|----------|
| **G1 범위** | DMI 메인 + AT/MASK 등 ≥1개 보조 팀 운영 라인 연결 |
| **G2 행동 변화** | 분류 결과를 분석가 작업 큐의 default로 사용, 수용률 ≥70% |
| **G3 운영 안정성** | 정확도 drift 모니터링, 월간 재평가, 데이터셋 정례 갱신 |
| **Outcome** | Hero KPI: lead time ↓30% 측정 완료 — 또는 **fallback trigger**: lead time 측정 불가 시 정확도·우선순위 큐 정확도로 대체 |

### 데이터 의존성
- 검측 이미지 데이터셋 (A2 anchor와 연결) + labeling
- 공정 변경 결정 로그 — lead time 측정의 핵심
- [SLOT: 검측 시스템 ↔ 공정 변경 시스템 cross-reference 가능 여부?]

### 책임 (3-tier)
- **AX Part:** Vision 모델 평가 설계, lead time 측정 방법론 정의 (fallback 포함)
- **실행 TF:** DMI 라벨 표준화, 분석가 작업 큐 UX 설계, 수용률 측정
- **전사 TF:** GPU 자원 (vision model training), 인프라

### 리스크
- **Stream 3 baseline 측정 자체가 PoC 과제** — 1년차 못 끝나면 hero에서 강등하고 E1 승격 (outline §3.3 fallback)
- 검측 데이터 다양성 (장비별·공정별) → 단일 모델로 일반화 어려움 → 분할 학습 필요

## QW #4 — 장비 안정성(FDC) 이상 시그널 시드 ★ Hero (조건부)

| 속성 | 값 |
|------|-----|
| **Stream** | 5 (안정성 모니터링) |
| **주 능력** | 이상·예측 |
| **Hero KPI** | 예지보전 hit rate / 알람 평균 대응시간 — Layer 1 ★5 |
| **Baseline 출처** | A3 (T2, 장비 운영 로그) |
| **본 적용 대상** | ✕ "시그널 검증" 수준 — 본 적용은 3년차 |

### 12개월 PoC 졸업 기준 (본 적용 ✕)
1. **한정 범위 baseline 확보:** [SLOT: 1~3개 장비 종류 선정]에서 12개월 FDC 시계열 + 알람·고장 라벨 정리
2. **이상 시그널 정확도 측정값 확보:** 사람 검증 동반 — precision-recall 곡선 도출
3. **운영 시나리오 ≥1개 검증:** *"이 시그널을 보고 사람이 다른 행동을 했을 때 결과가 좋아지는가"* 사례 ≥3건
4. **3년차 본 적용 조건 정의:** 어떤 정확도·범위·운영 모델이 *"진짜 예지보전"*인지 다음 라운드 의사결정 항목 명시

### 데이터 의존성
- FDC 시계열 (장비 센서·파라미터)
- 과거 알람·고장 라벨 (사후 분석된 알람 비율 — A3와 직결)
- [SLOT: 장비 종류별 FDC 데이터 스키마 통일 여부?]

### 책임 (3-tier)
- **AX Part:** 이상 탐지 모델 설계, baseline·정확도 측정 방법론, 3년차 본 적용 조건 정의 (CEO 보고용)
- **실행 TF:** ME·환경제어팀 중심 데이터 큐레이션, 알람 라벨 검증, 운영 시나리오 사례 발굴
- **전사 TF:** GPU·인프라, [SLOT: FDC 데이터 lake 시스템 owner?]

### 리스크
- **이상 ≠ 고장:** 알람이 자주 울려도 실제 고장 아닌 경우가 많아 precision 측정 자체가 까다로움
- *"완성 안 됨"* 을 정직하게 표현하지 못하면 CEO에게 *"왜 1년차에 결과 없냐"* 반박 — framing 매우 중요

### Framing 메시지 (CEO 보고용)
> *"#4는 1년차 *완성*이 아닌 *시그널 검증*. 진짜 예지보전은 3년차. 1년차 산출물은 *3년차에 본 적용 가능한 토대* 다."*

## QW #5 — 암묵지 순환고리 PoC (Enabler 횡단)

| 속성 | 값 |
|------|-----|
| **Stream** | 횡단 (Enabler 5.3b) |
| **주 능력** | Enabler (캡처·구조화·재투입) |
| **Layer 2 KPI** | E2 암묵지 캡처 사이클 횟수 — 1년 1회전 완주 |
| **본 적용 대상** | ✕ "사이클 1회전 검증" 수준 — 3년차에 정례화 |

### 12개월 PoC 졸업 기준 (본 적용 ✕)
1. **1개 영역 선정:** [SLOT: 어느 stream의 어느 의사결정 cluster를 첫 영역으로? — 추천: Track 2 검측 라벨 결정 또는 Track 3 실험 조건 결정]
2. **순환고리 4단계 1회전 완주:**
   - (a) **In-flow capture:** 일하는 흐름의 부산물로 의사결정·근거 캡처 (template 설계 + 사용 ≥10건)
   - (b) **구조화·인덱싱:** 캡처된 raw → RAG 입력 가능한 구조로 변환
   - (c) **RAG 재투입:** 다음 업무에서 검색·활용 (검색 빈도 측정)
   - (d) **활용 결과 피드백:** 캡처 양식·인덱싱 정제
3. **3년차 정례화 조건 정의:** 다수 영역으로 확대하기 위한 도구·인력·표준

### 데이터 의존성
- 의사결정 로그 (template 기반 입력)
- 사내 RAG 인프라 (Enabler 5.3a 명시지 RAG와 동일 스택)
- [SLOT: 캡처 도구 — 기존 노트 도구 / 채팅 / 별도 UI?]

### 책임 (3-tier)
- **AX Part:** 순환고리 4단계 설계, 캡처 template 표준, RAG 통합 설계, 3년차 정례화 조건 정의
- **실행 TF:** 선정 영역에서 캡처 실행, 사용성 피드백, 검색 빈도 측정
- **전사 TF:** RAG 인프라 (BGE-M3 + vector store)

### 리스크
- **"캡처를 위한 일" 함정:** 별도 인터뷰·문서화 시간을 만들면 지속 불가 → 일하는 흐름의 부산물 원칙 엄수
- 캡처된 데이터가 RAG에서 *유의미한 답을 만들지 못하면* 순환고리 자체가 무너짐 → (b) 구조화 단계 품질이 결정적
- E2 측정값 자체가 *"1회전 완주 했나"* 의 binary가 되어 정량 진척 표현 어려움 → 4단계별 mini-milestone 정의

### Framing 메시지 (CEO 보고용)
> *"#5는 RAG 품질의 상한을 결정하는 인프라적 PoC. 1년차 = 사이클 1회전. 3년차 = 다수 영역 정례. 5년차 = 전 stream 기본 동작."*

## 카드 간 의존성 매트릭스

| 의존성 | 영향 |
|--------|------|
| #2 → Enabler 5.3a 명시지 RAG | RAG 인프라가 #2 본 적용 전에 가동되어야 |
| #3 → A2 anchor 데이터 정리 | 검측 데이터 인벤토리가 baseline 측정의 선행 조건 |
| #4 → A3 anchor (FDC 사후분석률) | A3 측정값이 #4 baseline의 일부 |
| #5 → 다른 모든 QW | 암묵지가 잡히면 #1·#2·#3 RAG 품질 상한 ↑ — 그러나 1년차에는 영향 거의 없음 (사이클 1회전만) |
| #1·#2·#3 ↔ ITC AI 실행 TF | 본 적용 G1(2팀 이상)은 실행 TF 활성화 의존 |

## 열린 질문 (Round 4 진입점 후보)

1. **검측 시스템 ↔ 공정 변경 시스템 cross-reference 가능 여부** — #3 Hero KPI baseline 측정의 결정적 변수
2. **QW #4 선정 장비 종류** — ME·환경제어팀과 협의 필요
3. **QW #5 첫 영역 선정** — 검측 라벨 결정 vs 실험 조건 결정 중 어디?
4. **분석 보고서 작성시간 측정 방법** (#2) — 자기보고는 부정확, 시스템 로그는 데이터 없음
5. **AI 실행 TF 인력 capacity** — 5개 PoC 동시 진행 시 6팀에서 선출된 10~20명 인력으로 가능한가?
