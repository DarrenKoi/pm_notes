[캘리브레이션 샘플] LLM-only 답안

⚠️ 이 답안은 채점자 캘리브레이션용입니다. 참가자가 사내 RAG-LLM에 본 spec을 그대로 던졌을 때 나올 법한 답안을 재구성한 것. 의도적으로 모든 5축의 실패 패턴을 노출합니다. 실전 우수 답안의 모범이 아닙니다.

사용법: 채점자 풀이 본 답안을 먼저 채점해 "이 답안은 몇 점이어야 하는가"를 합의 → 실제 채점의 lower-bound 기준선으로 활용.

참가자: 홍길동 소속팀: DMI (Defect Metrology Inspection) 작성 일시: 2026-XX-XX

[영역 1] As-Is 진단 — DMI 팀 기준

1-1. 우리 팀 5 stream 중 가장 큰 비중을 차지하는 stream과 이유

DMI 팀은 stream 3 (데이터 생산) 의 비중이 가장 크다. 결함 계측·검측 데이터는 공정 개선의 핵심 기초 자료이기 때문에, 양산총괄·제조기술 등 여러 Biz에서 의뢰가 지속적으로 들어오고, 이 데이터를 안정적으로 생산하는 것이 DMI의 핵심 역할이다. stream 2 (분석)도 일정 부분 담당하며, stream 1 (의뢰 대응)을 통해 외부와 연결된다.

1-2. 사내 RAG/LLM 인프라가 현재 우리 팀에 어떻게 활용 가능한가

사내에는 Kimi-K2.5, Qwen3-VL 같은 LLM과 BGE-M3 임베딩 모델이 OpenAI 호환 API로 제공되고 있다. DMI 팀은 이를 다음과 같이 활용할 수 있다:

  • 문서 검색: 과거 의뢰 처리 이력, 기술 보고서를 RAG로 검색해 신규 의뢰에 대한 초기 답변 자동화
  • 이미지 분석: Qwen3-VL을 활용해 결함 이미지를 LLM이 해석하도록 함
  • 요약·보고서 자동화: 분석 결과를 LLM이 자동으로 정리해 보고서 초안 생성

이러한 인프라를 적극 활용하면 업무 생산성을 크게 높일 수 있다.

1-3. 우리 팀이 최근 처리한 비정형 의뢰 1건을 예로 들어 — AIX 대상이 되는 이유

DMI 팀은 다양한 비정형 의뢰를 받는다. 예를 들어 신규 결함 패턴이 발견되었을 때, 이를 분류하고 원인을 추정하는 의뢰가 들어온다. 이 경우 엔지니어가 과거 유사 사례를 찾아보고, 패턴을 비교하여 답변하는 과정이 필요하다. 이는 stream 1 (의뢰 대응) + stream 2 (분석)에 해당하며, RAG 기반 사례 검색과 VLM 기반 이미지 비교로 AI 자동화가 가능하다.

[영역 2] To-Be 로드맵 — ITC 전체

2-1. 1년 / 3년 / 5년 마일스톤

시기 마일스톤
1년차 RAG 기반 의뢰 응대 자동화 PoC 1건 / 암묵지 채굴 파이프라인 1개 영역 시범 / AI 챔피언 양성 6명 (팀별 1명)
3년차 5 stream 전반에 AI 적용 확대 / 암묵지 순환고리 다수 영역 정례화 / 정량 KPI 도입
5년차 전 stream 기본 동작 / 자율 운영 수준 도달 / 외부 Biz로 확산

2-2. 암묵지 순환고리를 ITC에 어떻게 도입할지 — 1년차 1회전

암묵지는 stock이 아니라 flow이므로, 일하는 흐름의 부산물로 자동 캡처되어야 한다. 1년차에는:

  1. In-flow capture: 엔지니어의 의사결정·근거를 업무 과정에서 자동으로 기록 (예: 분석 노트 자동 저장)
  2. 구조화·인덱싱: 캡처된 데이터를 정리해 검색 가능한 형태로 저장
  3. RAG 활용: 다음 업무에서 LLM이 참조
  4. 피드백 루프: 활용 결과·교정 사항이 캡처에 다시 반영

이 1회전을 1년 내에 1개 영역에서 검증한다.

2-3. 본인이 가장 자신 있는 1개 업무에 대한 채굴→구조화→RAG 적용 시나리오

DMI 팀의 결함 분류 업무를 예로 들면:

  • 채굴: 엔지니어의 분류 판단 근거를 분석 노트에 명시화하도록 표준 템플릿 도입
  • 구조화: 결함 유형·원인·대응 방안을 메타데이터로 정리
  • RAG 적용: 신규 결함 발견 시 RAG가 유사 사례를 검색해 1차 분류 제안

이 시나리오는 in-flow capture 원칙을 따르며, 별도의 인터뷰 시간을 만들지 않는다.

[영역 3] Governance — DMI 팀 기준

3-1. 성과 관리 지표 3개

  1. AI 대응 자동화율: 의뢰 중 AI가 1차 응답한 비율
  2. TAT 단축률: 의뢰 처리 시간의 단축 정도
  3. 암묵지 채굴 건수: 분기별 캡처된 의사결정 사례 수

이 중 DMI 팀에서만 측정 가능한 것은 3번 (암묵지 채굴 건수) — 우리 팀의 분석 노트 표준화 진행도와 직결되기 때문이다.

3-2. AIX 도입 시 본인 팀에서 예상되는 저항 1가지 + 대응

가장 큰 저항은 변화에 대한 거부감이다. 엔지니어들은 기존 방식에 익숙해져 있고, AI 도구 학습에 시간을 투자하는 것을 부담스러워할 수 있다.

대응:

  • 충분한 교육·온보딩 프로그램 제공
  • AI 챔피언을 통한 점진적 확산
  • 초기에는 빠른 성공 사례(Quick Win)를 만들어 동기 부여
  • 변화관리(Change Management) 원칙을 적용해 단계적 도입

[메타] 핵심 프롬프트 3개

#1. 우리 회사 기반기술센터의 역할과 AIX 로드맵을 정리해줘. 이유: 전체 구조를 빠르게 잡기 위해.

#2. 암묵지 순환고리를 1년차에 어떻게 도입할지 단계별로 설명해줘. 이유: 영역 2-2 답변 구조를 잡기 위해.

#3. AIX 도입 시 발생할 수 있는 저항과 대응 방안을 알려줘. 이유: 영역 3-2를 빠르게 채우기 위해.

채점자 합의가 필요한 회색지대

  • 축 2 점수: RAG가 가르쳐준 용어를 정확히 인용한 것이 "프레임 적용 능력"인가, 단순 "복사"인가? → 합의 필요. spec 권고는 "용어 사용은 OK지만, 본인 사례에 적용한 흔적이 없으면 70% cap".
  • "결함 분류 업무 예시"의 평가: 카테고리는 맞지만 구체 사례 없음. 0점인가, 부분 점수인가?
  • 영역 2-1의 마일스톤 표: ITC 전체 대상이라 본인 사례 D 요구가 약함. 그런데도 LLM이 흔히 내놓는 generic 마일스톤이라면 어느 축에서 감점?
