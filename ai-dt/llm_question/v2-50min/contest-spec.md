> 본 spec은 *문제 자체*가 아니라 *문제를 어떻게 설계할 것인가의 결정사항* 모음.

## 1. 평가의 1차 축

**LLM-인간 협업 능력**

- 순수 LLM prompt 능력도, 순수 도메인 사고력도 아님.
- 회사 관점에서 진짜 알고 싶은 것: "LLM이 만든 결과를 자기 도메인 지식으로 **검증·교정·확장**할 수 있는 사람인가."
- 모든 하위 설계는 이 1차 축에 종속.

**시험 목적 (v0.2 단순화)**: scope를 본인 업무 micro로 좁히면서, v0.1의 2차 목적이던 "CEO 보고용 ITC 통합 narrative 보조"는 더 이상 이 시험의 산출물이 아님. 목적은 **"개인의 AIX 협업 능력 평가"** 한 줄로 단순화.

## 2. 참가자

- **ITC 내부 인원만**. 본인 도메인 경험·암묵지가 답안 재료의 핵심.
- 답안 작성 시작 시 **본인 소속팀 선언** (6개 팀: MASK·소재·DMI·AT·ME·환경제어).
- 사전과제 없음 — 즉석에서 본인 암묵지를 끌어내는 능력 자체가 평가 대상.

## 3. 형태·시간·도구

| 항목 | 결정 |
|---|---|
| 형태 | 개인전 (1인 1LLM) |

### 시간 배분 (실질 45분 기준 — 옵션 Y: 메타 실시간 기록)

- 문제 읽기·접근 설계: 5분
- 영역1 (본인 Workflow + AIX 로드맵): ~18분
- 영역2 (고과 반영 찬반 + 도입 우려): ~12분
- 정리·교정: ~10분
- 핵심 프롬프트 3개 + 사유: **별도 단계 ✕ — 작성 중 답안 하단 고정 칸에 실시간 기록**

> 메타회고를 마지막 별도 5분으로 두면 시간 압박에 가장 먼저 희생됨 → 답안 MD 하단에 프롬프트 기록 칸을 미리 깔아두고 작성 중 채우게 함. 빈칸이면 축 4(20점) 0점 처리.

## 4. Deliverable 구조

### 4.1 본문 2개 영역 (v0.2 — 50분 / 본인 업무 micro scope)

| # | 영역 | 시간 | 흡수하는 bgk 항목 | 평가 표적 |
|---|---|---|---|---|
| 1 | **본인 Workflow + AIX 로드맵** | ~18분 | bgk ①(직무 세분화·workflow·AIX 적용·로드맵) + ②(AI 적용 후 업무 방향·집중점) + ⑤(진척 관리는 로드맵 안 한 단락으로 흡수) | 축 1 Domain grounding |
| 2 | **LLM 오답 검증** | ~12분 | (신규 — bgk 외) | 축 3 Critical refinement (1차 축 직격) |

**영역1 요구사항 (Workflow + AIX 로드맵):**
> 1. **본인 직무 workflow 표** (산문 ✕, 표 강제): `작업 유형 | 빈도(월 X건 등) | AI 적용 가능성(상/중/하) | 적용 방식` 형식. 3행 이상.
> 2. 위 표에서 **AIX 적용 1순위 1~2건 선정** + 이유.
> 3. **로드맵 — 단기/장기 2단계**: *단기*는 구체적으로(무엇을·어떻게·어떤 데이터로), *장기*는 방향만 1~2줄.
> 4. AI 적용 후 본인 업무가 어디로 가야 하는지 / 무엇에 더 집중할지 (bgk ②).
> 5. 진척을 어떻게 관리할지 1단락 (bgk ⑤).

**영역2 요구사항 (LLM 오답 검증):**
> 이번 세션에서 기내식(RAG-LLM)이 준 답 중 *도메인적으로 틀렸거나·불완전하거나·본인 맥락에 안 맞는* 것 **1개를 그대로 인용** → ① 무엇이 왜 잘못됐는지 → ② 본인이 어떻게 교정했는지 적으시오.

근거:
- Scope가 본인 업무 micro로 좁아짐 → breadth보다 **depth** 평가. 영역 2개로 충분.
- 영역2는 시험 1차 축("LLM 출력을 도메인 지식으로 검증·교정")을 정면 문항화. 실제 세션 산출물(LLM 답변 원문 + 교정)이 있어야 하므로 LLM-only 답안은 여기서 0점 수렴 → top 3 변별 결정타.
- 5개(bgk) → 2개 통합 이유: 실질 45분에 영역당 12분 이하면 LLM-카피 답안만 양산(v0.1 § 4.1 경고).
- v0.1의 "팀-고정/가변" 구분은 scope가 전부 본인 업무로 통일되면서 무의미 → 제거.

**bgk 항목 중 의도적 제외 (2026-05-20):**
- bgk ③(조직 변화·대응), ④(AI 활용 능력 고과 반영 찬/반) → 50분 scope 축소로 시험에서 제외. (④ 고과 찬반은 출제자 원안이었으나, 영역2를 더 변별력 높은 "LLM 오답 검증"으로 대체.)
- 영역2가 LLM 오답 인용+교정으로 위조 불가능한 trap이 되면서, v0.1 Q5/Q6에서 논의한 "본인 경험 강제 슬롯"은 자연 흡수됨.

### 4.2 메타 회고 — 실시간 기록 (옵션 Y)

핵심 프롬프트 3개 발췌 + **사유(영향 중심)**. 답안 MD 하단 고정 칸에 작성 중 채움:

```
## 핵심 프롬프트 기록 (작성 중 실시간으로 채워주세요)
[#1] 프롬프트 원문:
     이 프롬프트가 결과를 어떻게 바꿨나 / 왜 결정적이었나:
[#2] ...
[#3] ...
```

- 사유는 "왜 이렇게 물었나"(설계 의도)가 아니라 **"이 프롬프트가 LLM 출력을 어떻게 바꿨나 / 왜 결정적이었나"**(영향) 로 물음 → 프롬프트만 잘 쓴 사람과 *LLM 출력을 평가·교정한* 사람을 가름.
- **빈칸이면 축 4(20점) 0점 처리** 명시 → 시간 압박에도 빠뜨리지 않게 강제.
- 미니멀 양식 결과 → **Refinement(LLM 교정 흔적)는 본문 안에서도 직접 보여야 함**. 안내문에 "LLM이 제시한 generic 답을 어떻게 본인 맥락으로 교정·확장했는지 본문에 명시" 강제 문구 포함.

## 5. C-type (LLM-trap) 함정 질문 설계 원칙

**v0.2 결정 (A+B → 영역2 재설계 반영): 명시적 함정 문항 ✕ — 묵시적 trap + 영역2 자체가 trap.**
- micro scope에서는 "본인 업무의 구체 수치·사례를 쓰라"는 요구 자체가 RAG로 못 푸는 trap → 인위적 함정 추가는 시간 과부하라 제거.
- 영역2가 "LLM 오답 검증"(실제 세션 LLM 답변 원문 인용 + 도메인 교정)으로 확정되면서, **영역2 자체가 위조 불가능한 trap** 역할. Q5/Q6에서 논의한 별도 "경험 강제 슬롯"은 여기 흡수됨.
- 아래 § 5 표/예시는 *함정 재료 판별 기준*으로는 여전히 유효하나, 별도 함정 문항으로 출제하지는 않음 (참고용).

RAG가 *알 수 없는* 영역에서만 출제. 사전에 출제자가 RAG corpus를 spot-check해 확인.

| RAG 지식 | 함정 재료 여부 |
|---|---|
| 🟢 조직도, 5 stream 정의, AIX TF 일반 원칙 | ✕ (공통 지식, 함정 아님) |
| 🟡 팀별 의뢰 카테고리, FDC 일반 정의 | △ (부분적, 디테일에서 함정 가능) |
| 🔴 개별 팀 TAT 실측치, 암묵지 1건 구체 사례, AI Agent 우선순위 디테일, 팀 내 비공식 관행 | ✓ **(함정 재료)** |

### 함정 질문 예시 (초안 — 출제자 추가 검증 필요)

- **As-Is**: "당신 팀이 최근 1년간 가장 자주 받은 *비정형* 의뢰 유형 3가지를 적고, 각 유형의 평균 처리 TAT를 적으시오. 그 중 1개가 AI 자동화 대상이 되는 이유를 5 stream 중 어느 흐름에 해당하는지로 설명."
- **To-Be**: "ITC 6개 팀 중 *암묵지 순환고리 1년차 1회전*의 첫 대상으로 어느 팀을 추천하는가. 그 팀의 데이터 가용성·암묵지 양·확산 가능성 3축으로 근거를 제시."
- **Governance**: "AIX 도입 시 당신 팀에서 가장 강한 저항이 예상되는 1가지(직군·세대·업무 종류 등 구체화) + 본인이 직접 경험한 변화 거부 사례 1건 + 대응안."

## 6. 채점 rubric (5축 × 비대칭 weight, 총 100점)

> v0.2: 균등 20점 → 비대칭. micro scope에서 변별력은 축 1·3에 몰림. 점수 단봉 분포(60~80 몰림) 방지 위해 핵심 축에 weight + 체크리스트 sub-criteria 병용.

| 축 | 배점 | 평가 대상 | 채점 단서 |
|---|---|---|---|
| **1. Domain grounding** | **25점** | 본인의 *구체적·검증 가능한* 사례·수치가 박혀있나 | "최근 1년", "월 X건", 팀명·장비명 명시 |
| **3. Critical refinement** | **25점** | LLM의 generic 답을 본인 맥락으로 교정·확장한 흔적이 *본문에* 직접 드러남 | "LLM 제안은 X였으나 우리 팀은 Y" 류 명시 |
| **4. Meta-cognition** | **20점** | 핵심 프롬프트 3개 선택과 사유(영향 중심)가 의식적 협업 증명 | 표면적 프롬프트 vs 결정적 reframing 구분 |
| **2. Framework application** | **15점** | 5 stream / 암묵지 순환고리 / 3-tier 거버넌스 등 ITC-specific 도메인 언어 정확 적용 | CONTEXT.md 용어 정확도 (일반 컨설팅 용어 나열은 가산점 ✕) |
| **5. Narrative coherence** | **15점** | 영역1 → 영역2가 한 thesis로 묶임 | 영역 간 cross-reference, thesis 일관성 |

### 6.1 변별력 보강 — 축 1·3 체크리스트형 sub-criteria

**축 1 (Domain grounding 25점)** — 각 5점 × 5항목:
- [ ] 본인 업무의 구체적 의뢰/작업 유형 3건 이상 적시
- [ ] 정량 수치 명시 (월 X건 / TAT X시간 / 인력 X명)
- [ ] 5 stream 중 본인 업무가 어디 속하는지 mapping
- [ ] 장비명·도구명·시스템명 구체적 명시
- [ ] 본인 직접 경험한 사례 1건 이상 (전언 ✕)

**축 3 (Critical refinement 25점)** — 항목별 배분 (영역2 LLM 오답 검증 중심):
- [ ] 기내식 답변 원문을 그대로 인용했는가 (인용 없으면 위조 의심) (5점)
- [ ] 무엇이 왜 틀렸는지 *도메인 근거*로 설명 (LLM이 놓친 제약·예외 명시) (10점)
- [ ] 본인이 어떻게 교정했는지 — 구체적 대안 제시 (10점)

→ 채점자 2명이 체크 개수로 합의 → 체크 차이 = 점수 격차 → top 3 변별.

### 채점 운영 (v0.2)

- **채점 주체**: AX Part 전원 **공동 채점**. → 6팀 도메인 이질성은 채점 배정 문제 ✕.
- **핵심 원칙 (공동 채점 편차 방지)**: 채점은 **"도메인 사실의 진위"가 아니라 "구체적·검증 가능한 형식인가"**를 본다. 환경제어팀 수치가 *진짜 맞는지* 확인할 필요 없음 — "월 X건·장비명·LLM 교정 흔적"이 *있는지*만 평가. (도메인 비전문가도 일관 채점 가능)
- **채점 시간**: 50분·2영역 답안이라 1답안당 약 **8~12분** (90분안 기준 15~20분에서 단축). 참가자 수 × 12분으로 총 부하 산정.
- **사전 캘리브레이션**: [`sample-model-answer.md`](./sample-model-answer.md)(83~93점)·[`sample-llm-only-answer.md`](./sample-llm-only-answer.md)(22~35점)를 채점자 전원이 먼저 각자 채점 → 점수 갈리는 축의 기준 합의.
- **동점 tiebreaker** (1·2·3등 동점 시 우선순위 축): **축3(Critical refinement) > 축1(Domain grounding) > 축4(Meta-cognition)**. — 시험 1차 축(LLM 검증·교정) 순서.

## 7. 형평성 보정 (v0.2 — 채점 철학으로 흡수)

scope가 본인 업무 micro로 좁아지면서 형평성 위험이 *팀 단위*(큰 팀 유리) → *업무 단위*(AIX 적용 여지 큰 업무 유리)로 재정의됨.

**보정 방식 = 별도 슬롯 ✕, 채점 철학 명시 (옵션 B):**
- 시험이 측정하는 건 "좋은 AIX 기회를 가진 사람"이 아니라 **"LLM과 협업해 자기 업무를 정확히 분석·교정하는 능력"**.
- 따라서 **"AI 적용이 어려운 업무라면, *왜 어려운지를 도메인 근거로 정확히 분석*하는 것 자체가 축 1·3 만점 답안"** — rubric § 6 + 안내문에 명시.
- 역으로 **무리하게 AI를 갖다붙인 generic 로드맵은 감점** (도메인 검증 부재 = 축 3 미달).
- 결과: 단순·정형 업무 담당자도 불리하지 않음. v0.1의 "사례 빌리기 허용"은 scope 흐림 → 폐기.
