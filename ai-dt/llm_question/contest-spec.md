# ITC AIX 리포트 경진대회 — 출제 spec (v0.1)

> Grilling 세션(2026-05-15)으로 도출한 출제 설계 문서. 입력 메모는 [bgk.txt](./bgk.txt).
> 본 spec은 *문제 자체*가 아니라 *문제를 어떻게 설계할 것인가의 결정사항* 모음.

---

## 1. 평가의 1차 축

**LLM-인간 협업 능력**

- 순수 LLM prompt 능력도, 순수 도메인 사고력도 아님.
- 회사 관점에서 진짜 알고 싶은 것: "LLM이 만든 결과를 자기 도메인 지식으로 **검증·교정·확장**할 수 있는 사람인가."
- 모든 하위 설계는 이 1차 축에 종속.

## 2. 참가자

- **ITC 내부 인원만**. 본인 도메인 경험·암묵지가 답안 재료의 핵심.
- 답안 작성 시작 시 **본인 소속팀 선언** (6개 팀: MASK·소재·DMI·AT·ME·환경제어).
- 사전과제 없음 — 즉석에서 본인 암묵지를 끌어내는 능력 자체가 평가 대상.

## 3. 형태·시간·도구

| 항목 | 결정 |
|---|---|
| 형태 | 개인전 (1인 1LLM) |
| 시간 | 90분 |
| LLM | 사내 RAG-LLM만 (Kimi-K2.5 / Qwen3-VL 등 모델 선택 자유) |
| 외부 LLM | 사용 불가 (회사 방화벽 차단) |
| 인터넷 검색 | **허용** — 일반 웹 검색 OK (외부 LLM API와 구분) |
| Deliverable | **Markdown(.md) 파일** — 사내 블로그 업로드 형식 |
| 제출처 | 사내 블로그 |
| 수상 | 1·2·3등 |
| 사후 활용 | 우수 답안은 **센터 내 전체 이메일 공유** (내부 자산화) |

### 시간 배분 (예시)

- 문제 읽기·접근 설계: 10분
- 본문 작성 (3 영역 × ~17분): 50분
- 정리·교정: 15분
- 핵심 프롬프트 3개 + 1줄 이유: 5분
- 여유: 10분

## 4. Deliverable 구조

### 4.1 본문 3개 영역

| # | 영역 | 시간 | 팀-고정/가변 | 함정 슬롯(C) | 본인 사례 슬롯(D) |
|---|---|---|---|---|---|
| 1 | **As-Is 진단** (조직 역할·인프라/역량) | ~20분 | 팀-고정 | 1~2개 | 필수 1건 |
| 2 | **To-Be 로드맵** (1·3·5년 AIX 로드맵) | ~20분 | ITC 전체 | 1개 | 필수 1건 |
| 3 | **Governance** (성과 관리 + 도입 우려·대응) | ~10분 | 팀-고정 | 1개 | 필수 1건 |

근거:
- 1·3은 본인 팀 사례가 핵심 재료 → 팀-고정.
- 2는 본 프로젝트의 1차 목적(ITC 통합 로드맵)이라 팀-갇힘 회피 → ITC 전체.
- 5개(bgk.txt) → 3개 통합 이유: 90분에 5개는 영역당 12분, LLM-카피 답안만 양산됨.

### 4.2 메타 회고 (5분)

핵심 프롬프트 3개를 발췌:

```
[#1] 프롬프트 원문:
     이게 결정적이었던 이유 (1줄):
[#2] ...
[#3] ...
```

미니멀 양식 채택의 결과 → **Refinement(LLM 교정 흔적)는 본문 안에서 직접 보여야 함**.
출제 안내에 "LLM이 제시한 generic 답을 어떻게 본인 맥락으로 교정·확장했는지 본문에 명시" 강제 문구 포함.

## 5. C-type (LLM-trap) 함정 질문 설계 원칙

RAG가 *알 수 없는* 영역에서만 출제. 사전에 출제자가 RAG corpus를 spot-check해 확인.

| RAG 지식 | 함정 재료 여부 |
|---|---|
| 🟢 조직도, 5 stream 정의, AIX TF 일반 원칙 | ✕ (공통 지식, 함정 아님) |
| 🟡 팀별 의뢰 카테고리, FDC 일반 정의 | △ (부분적, 디테일에서 함정 가능) |
| 🔴 개별 팀 TAT 실측치, 암묵지 1건 구체 사례, AI Agent 우선순위 디테일, 팀 내 비공식 관행 | ✓ **(함정 재료)** |

### 함정 질문 예시 (초안 — 출제자 추가 검증 필요)

- **As-Is**: "당신 팀이 최근 1년간 가장 자주 받은 *비정형* 의뢰 유형 3가지를 적고, 각 유형의 평균 처리 TAT를 적으시오. 그 중 1개가 AI 자동화 대상이 되는 이유를 5 stream 중 어느 흐름에 해당하는지로 설명."
- **To-Be**: "ITC 6개 팀 중 *암묵지 순환고리 1년차 1회전*의 첫 대상으로 어느 팀을 추천하는가. 그 팀의 데이터 가용성·암묵지 양·확산 가능성 3축으로 근거를 제시."
- **Governance**: "AIX 도입 시 당신 팀에서 가장 강한 저항이 예상되는 1가지(직군·세대·업무 종류 등 구체화) + 본인이 직접 경험한 변화 거부 사례 1건 + 대응안."

## 6. 채점 rubric (5축 × 20점)

| 축 | 평가 대상 | 채점 단서 |
|---|---|---|
| **1. Domain grounding** (20점) | 본인 팀의 *구체적·검증 가능한* 사례·수치가 박혀있나 | "최근 1년", "월 X건", 팀명·장비명 명시 |
| **2. Framework application** (20점) | 5 stream / 암묵지 순환고리 / 3-tier 거버넌스 등 공통 프레임 정확 적용 | CONTEXT.md 용어 정확도 |
| **3. Critical refinement** (20점) | LLM의 generic 답을 본인 맥락으로 교정·확장한 흔적이 *본문에* 직접 드러남 | "LLM 제안은 X였으나 우리 팀은 Y" 류 명시 |
| **4. Meta-cognition** (20점) | 핵심 프롬프트 3개 선택과 1줄 이유가 의식적 협업 증명 | 표면적 프롬프트 vs 결정적 reframing 구분 |
| **5. Narrative coherence** (20점) | As-Is → To-Be → Governance가 한 thesis로 묶임 | 영역 간 cross-reference, thesis 일관성 |

### 채점 운영

- 1답안당 약 15~20분.
- 채점자 2명 cross-check 권장 (특히 축 1·3은 정성 평가 비중 큼).
- 사전 캘리브레이션: 출제팀이 모범 답안 1개·LLM-카피 답안 1개를 미리 작성해 채점 기준 합의.

## 7. 형평성 보정

ITC 6팀의 비대칭(MASK·소재·DMI vs ME·환경제어) 대응:

- **본인 사례 칸 = 복수 출처 허용**: 본인 팀 + 협업한 다른 팀 사례까지 OK.
- **함정 질문에 "1건이라도 좋다" 명시**: 양이 아니라 깊이 평가.
- **팀별 정답 ✕, 프레임 적용 ○**: 채점은 *내용의 정답*이 아니라 *공통 프레임 적용의 정확성*과 *사례의 구체성*.

### 결정사항이 spec에 미치는 영향

**(a) 인터넷 검색 허용 → 채점 축 2의 의미 변화**
- 일반 AIX 컨설팅 용어("변화관리", "Quick Win", "거버넌스 3-tier" 등)는 인터넷 + RAG 양쪽 다 알고 있음.
- 축 2 (Framework application) 평가는 **ITC-specific 도메인 언어 정확 적용**으로 좁혀야 함:
  - 5 stream의 *각 stream과 본인 팀 업무의 mapping 정확도*
  - 암묵지 순환고리 4단계(capture → 구조화 → RAG → 피드백)를 *본인 시나리오*에 적용했는지
  - 3-tier 거버넌스(전사 AIX TF / AX Part / AI 실행 TF)의 *역할 분담 정확도*
- 일반 컨설팅 용어 나열만으로는 축 2에서 더 이상 가산점 없음 → § 6 채점 가이드 보강 필요.

**(b) MD 제출 → 표현 제약**
- 이미지·도표 첨부는 사내 블로그가 허용하는 범위 내 (필요 시 별도 확인).
- 코드 블록·표·인용 등 MD 문법은 자유 활용 가능 → "구조적 글쓰기"가 약간의 메리트가 됨 (단, 축 5 Coherence와 중복 아니게 채점).

**(c) 전체 공유 → 참가자 동의 절차**
- 답안 표지에 체크박스 추가: `[ ] 본 답안이 우수 답안으로 선정될 경우 센터 내 공유에 동의함`
- 동의 ✕ 답안도 채점·수상 대상에는 포함 (공유만 제외).
