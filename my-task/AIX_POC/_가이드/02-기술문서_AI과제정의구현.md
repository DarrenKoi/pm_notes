# AIX POC 기술 문서 — AI 과제 정의 · Agent 설계 · To-Be 구현

> [기획 문서](./01-기획문서_AX서비스기획.md)에서 선정·연계된 과제를, **실제 구현 가능한 AI Agent + To-Be 프로세스**로 구체화하기 위한 설계 표준과 점검 기준을 정의한다. (Design Camp 12-Step 중 Track B: 8~12단계)

## 왜 필요한가? (Why)

기획에서 "무엇을 할지"가 정해졌다면, 기술 문서는 **"어떻게 동작하게 만들지"** 를 고정한다. 특히 두 가지를 강제한다.

- **역할 경계의 명시화** — AI가 어디까지 처리하고 사람이 어디서 최종 판단·승인하는지를 Task 단위로 못 박는다. (Human-AI Centric 원칙의 구현)
- **제약·성공조건의 사전 검증** — 기술뿐 아니라 조직·비용·정책·보안 제약과 KSF를 설계 시점에 확인해, 구현 단계의 좌초를 예방한다.

## 핵심 개념 (What) — To-Be 설계 5대 용어

To-Be 프로세스를 그릴 때 쓰는 표준 용어. (출처: [원문 10](../lectures/captures/10-check-and-review.md))

| 구분 | 정의 | 가이드 핵심 |
|------|------|-------------|
| **To-Be 프로세스** | AI를 반영해 목표 시점에 조직이 지향하는 이상적·실현 가능한 업무 흐름 | 이상적인 일하는 방식 |
| **Activity / Task** | To-Be 내에서 AI Agent·시스템·사람의 역할이 재정의된 실행 단위 작업 | 순차적 연결 고려 |
| **AI Agent 역할** | 특정 To-Be Task에서 사람을 보조/대체해 데이터 처리·분석·판단·생성을 수행하는 주체 | AI 기술 유형 참고 |
| **제약 사항** | 설계·추진 시 기술·시스템·조직·비용·정책 측면에서 넘을 수 없는 조건·한계 | 보안/외부데이터 등 |
| **KSF** | To-Be가 의도한 성과를 달성하기 위해 반드시 충족돼야 하는 핵심 성공 조건 | 선결 항목 위주 작성 |

## 핵심 개념 (What) — Process 체계: Decomposition · Map · SOP

To-Be를 설계하기 전에, 업무를 어떤 단위로 보느냐를 고정한다. 세 산출물은 각각 다른 질문에 답하며, **Agentic AI 설계에 서로 다른 것을 제공**한다. (출처: [강의 덱 V2.7](../lectures/README.md), 모듈 2 · Wrap Up)

| 구분 | 답하는 질문 | 형태 | Agent에게 주는 것 | 없이 만들면 |
|------|-------------|------|-------------------|-------------|
| **Process Decomposition** | "무엇이 있는가" | 계층 트리 (L1~L5) | Agent의 **범위·레벨**(어디에 배치) | 경계 불명확 → 중복·충돌·누락 |
| **Process Map** | "어떻게 흐르는가" | 흐름도(Flow) | Agent의 **트리거·분기·핸드오프**(어떻게 움직이나) | 언제 시작·종료, 누구에게 넘길지 모름 |
| **SOP** | "정확히 어떻게 하는가" | 단계별 지침서 | Agent의 **행동 규칙·판단 기준·안전망** | 스스로 규칙을 만들어 컴플라이언스 위반 |

**SOP → LLM 구현 매핑** (Agentic AI에서 SOP가 직접 코드가 된다):

- SOP의 절차 순서·판단 기준 → **LLM 행동 지침(System Prompt / Reasoning)**
- SOP의 "시스템 조작" → **LLM Tool & Function Call (API·MCP)**
- SOP의 예외 처리·제한 → **LLM Guardrail & Policy**

### 표준 프로세스 레벨 (L1~L5)

업무 분해의 공통 척도. 조직 규모에 따라 담당 조직은 L2, 팀 조직은 L3 내외로 편성하며, AI Agent는 기본적으로 **L4·L5 최소 실행 단위를 자동화**(상위 레벨은 Function call·API·MCP로 확대)한다. (출처: [강의 덱 V2.7](../lectures/README.md), 모듈 2)

| 레벨 | 명칭 | 정의 | 권장 개수 | 기재 단위 |
|------|------|------|-----------|-----------|
| L1 | 비즈니스 도메인 | 가치를 창출하는 최상위 업무 집합 | 1개 | 본부/부문 정체성 |
| L2 | 핵심 업무 영역 | 독립 운영되는 주요 직무 기능 | 3~5개 | 담당자별 R&R |
| L3 | 주요 활동 | 특정 산출물(Output)을 만드는 과업 묶음 | L2당 3~5개 | 주간 보고서 단위 |
| L4 | 상세 프로세스 | 최소 단위 실행 단계('동사' 중심) | L3당 3~5개 | "무엇을 어떻게" |
| L5 | Task | 실무자가 수행하는 최소 단위 작업 | — | 행동 단위 |

> 누락 점검: 작성한 업무 목록을 **PLAN(계획·기획) · DO(실행·운영) · SEE(점검·보고)** 로 훑어 빠진 영역이 없는지 본다. L4 하나를 읽었을 때 그 안에 할 일이 2개 이상 보이면 L3 수준이므로 더 쪼갠다.

## 어떻게 사용하는가? (How)

### Step 8. AI 과제 정의서 작성 (Agent Definition)

과제 1건당 1매. 아래 양식을 채운다. (출처: [원문 09](../lectures/captures/09-ai-task-definition-template.md))

```
┌──────────────────────────────────────────────────────────────┐
│ 과제명 : __________________________________________________   │
│ 구분   : ☐ Marketing  ☐ R&D  ■ Production  ☐ Support          │
│ Champion(담당 임원) : ________   담당 조직 : ________          │
├──────────────────────────────────────────────────────────────┤
│ [상세내역]                                                     │
│  • 업무 목표 (부문/팀)                                         │
│  • 부문/본부 핵심 업무 구조화                                  │
│  • 추진 배경 (Pain Point)                                      │
│  • 근본원인 (Big Why)                                          │
│  • AI Agent ─ 대상 업무 / 요구 사항 / 적용 범위 / 기대 효과    │
└──────────────────────────────────────────────────────────────┘
```

> 설계 의도: Pain Point·Big Why가 양식에 **필수 칸**으로 들어가 있어, 근본 원인 없이 과제를 정의할 수 없게 강제한다.

### Step 9~10. As-Is → To-Be 프로세스 (Swimlane)

업무 흐름을 가로 5열(Phase) Swimlane으로 그리고, 각 Task 박스를 **🤖 AI Agent 수행 / 🧑 Human 수행** 으로 구분한다. (출처: [원문 07](../lectures/captures/07-ai-agent-scope-template.md))

```
Phase1 ─→ Phase2 ─→ Phase3 ─→ Phase4 ─→ Phase5
  🤖        🧑        🤖        🤖        🧑
  🧑        🤖        🤖        🧑        🤖
  ...       ...       ...       ...       ...
범례:  🤖 AI Agent 수행 업무   🧑 Human 수행 업무
```

작성 원칙:
- **As-Is**를 먼저 그려 "가장 시간이 많이 걸리는 구간(병목)"을 찾는다.
- **To-Be**에서 그 병목 Task에 AI Agent를 배치해 공수를 줄인다.

**Process Modeling Rule (IDEF0형)** — 각 Task를 4요소로 정의하면 As-Is/To-Be가 흐름도뿐 아니라 표(ID·Activity·설명·기준유무·업무주기·TAT·System·Menu)로도 일관되게 떨어진다. (출처: [강의 덱 V2.7](../lectures/README.md), 모듈 2 Modeling Rule)

| 요소 | 정의 |
|------|------|
| **Input** | 선행 단계·외부에서 제공되는 데이터·문서·요청 |
| **Output** | 결과물·보고서·시스템 반영 값·의사결정 |
| **Constraint** | Task가 따르는 조건·규칙·가이드·법규. Input과 달리 변하지 않고 방향을 결정 |
| **Mechanism** | Task를 수행하는 주체(담당자)·인프라(시스템·AI Agent·설비) |

> BPMN Notation 요약: 시작=단선 원, 종료=굵은 원, Task=직사각형, 의사결정/분기=마름모(◇, o·+·x로 배타·병렬·포괄 표기), 순서=실선 화살표, 메시지·데이터=점선 화살표. 반려·재검토처럼 **되돌아오는 흐름**을 빠짐없이 표현한다.

**적용 AI 기술 분류** — To-Be Task마다 어떤 기술 유형으로 푸는지 명시한다. 한 Task에 여러 유형이 결합될 수 있다. (출처: [강의 덱 V2.7](../lectures/README.md), 모듈 8)

| 대분류 | 세부 기술 |
|--------|-----------|
| **생성형 (LLM)** | 검색 · 추론 · 최적화 · 요약 · 예측 |
| **분석형 (ML/DL)** | 분류 · 최적화 · 이상탐지 · 예측 · 시뮬레이션 |
| **데이터 변환** | 이상탐지 · 추론 · 데이터 변환(정형화·표준화) |

### Step 11. KSF & 제약 사항 정의

To-Be가 실제로 동작하려면 선결돼야 하는 조건(KSF)과, 넘을 수 없는 제약을 4개 측면으로 분리해 작성한다.

| 측면 | 예시 제약 |
|------|-----------|
| 기술 | 알고리즘 · GPU · 데이터 품질 |
| 조직 | 담당 인력 · 내부 역량 |
| 비용 | 구축 · 운영 비용 |
| 정책·보안 | 개인정보 · 내부 규정 |

### Step 12. 개발 일정 수립 (Action Planning)

KSF 선결 항목을 우선순위로 정렬하고, 현실적 일정과 액션 오너를 확정한다. Task를 4구분(**업무/프로세스 · 데이터 · AI 모델 · Agentic AI Prototype**)으로 분류해 분기(M+3/M+6/M+9/M+12) 그리드에 배치한다. (출처: [강의 덱 V2.7](../lectures/README.md), 모듈 9)

| 구분 | 작성 핵심 |
|------|-----------|
| 업무/프로세스 | AI 도입에 **선행되어야 하는 업무 기준·표준화 Task를 먼저** |
| 데이터 | 수집 → 적재 → 표준화 순서, **데이터 소스·규모를 구체적으로 명시** |
| AI 모델 | 알고리즘 설계 → 학습 → 실제 시스템 연동까지 순서대로 |
| Agentic AI Prototype | 단일 Agent 완성 후 타 Agent와의 연계·확장까지 고려 |

**모델 유형별 개발 Task 트랙** (AI 모델 라인에 참조):

| LLM 트랙 | ML/DL 트랙 |
|----------|------------|
| Use case 설계 → 기술 Stack·LLM 선정 | Use case 설계 → Data 수집 |
| Vector DB 구축 | EDA 및 Cleansing |
| 추론엔진 최적화(셋팅) | Feature Engineering → Algorithm 선정 |
| 프롬프트 엔지니어링 | 학습 및 튜닝 → 성능 지표 확인 |
| Tool 개발 및 API 연계 | 성능 평가 환경 구축 |
| → 운영 → 모니터링 | → 운영 → 모니터링 |

**일정 설계 원칙**
- 선행 Task(데이터 수집·가공, 보안 검토)를 로드맵 **앞단**에 배치.
- 의존관계 없는 Task는 **병행**해 일정을 늘리지 않는다(예: 보안 검토 ∥ Agent 개발).
- 당장 해결 못 하는 제약은 **현재 쓸 수 있는 데이터 1~2종으로 좁혀 올해 안 완성 가능한 수준**으로 설계하고, 나머지는 '제약 사항'에 명시("전사 데이터 플랫폼 구축 후 시작" 같은 무기한 전제 금지).

## 설계 검증 — Check & Review

To-Be 설계 완료 후 반드시 통과해야 하는 3대 점검. (출처: [원문 10](../lectures/captures/10-check-and-review.md))

### ✅ 1. AI·사람 역할의 명확한 구분
각 Task에서 AI 처리 영역과 사람의 최종 판단·승인 영역이 Task별로 구분돼 있는가?
> 예) AI: 매출 데이터 수집·분류·분석·이상값 탐지 → 사람: 원인 판단 및 보고서 최종 승인

### ✅ 2. 시간이 가장 많이 걸리는 구간 개선 여부
As-Is 최대 병목 구간에 AI가 배치되어 공수가 줄어드는 방향으로 To-Be가 설계됐는가?
> 예) As-Is: 매주 수율 데이터 수동 취합 4시간 → To-Be: AI 자동 취합 + 담당자 검토 30분

### ✅ 3. 제약 사항 다각도 검토 여부
기술 제약만이 아니라 조직·비용·정책·보안 제약을 각각 확인·작성했는가?

## 구현 매핑 예시 — 영업 프로세스 4개 Agent

L1~L5 분해 후 도출된 AI Agent 배치. (출처: [원문 05](../lectures/captures/05-process-level-definition.md), [원문 06](../lectures/captures/06-sales-ai-opportunity.md))

| Agent | 대상 L2 그룹 | 배치 Task |
|-------|--------------|-----------|
| AI Agent#1 | 시장 및 고객 개발 | CRM Data 추출·정제 / 외부 시장 Data 수집 / 매력도 분석 |
| AI Agent#2 | 영업 기회 관리 | 표준가격기반 견적초안 작성 |
| AI Agent#3 | 납품 및 고객 서비스 | ERP 출하지시서 생성 |
| AI Agent#4 | 수금 및 채권 관리 | 청구서 발행·대금 청구 |
