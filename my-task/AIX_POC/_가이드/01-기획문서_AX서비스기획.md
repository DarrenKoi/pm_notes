# AIX POC 기획 문서 — AX 서비스 기획 방법론

> SK Hynix "New AI Design Camp" 방법론을 기반으로, 우리 부문/팀의 업무를 AI Agent 과제로 전환(AX)하기 위한 기획 프레임워크와 진행 절차를 정의한다.

## 왜 필요한가? (Why)

AI 도입은 "어떤 기술을 쓸까"가 아니라 **"어떤 업무의 어떤 고통(Pain Point)을 없앨까"** 에서 출발해야 실패하지 않는다. 본 방법론은 다음 3가지 흔한 실패를 막기 위한 것이다.

- **기술 과시형 실패** — 화려한 모델을 도입했으나 실제 비즈니스 이익이 없음
- **추측 기반 실패** — 가용 데이터 검증 없이 설계해 구현 단계에서 좌초
- **전면 대체 환상** — AI로 사람을 통째로 대체하려다 현장 저항·품질 저하

이를 막기 위해 3대 원칙(Baseline)을 모든 기획 활동의 전제로 둔다. (출처: [원문 02](../lectures/captures/02-ax-consulting-methodology.md))

| 원칙 | 한 줄 정의 |
|------|-----------|
| **가치 중심 기획** (Value-Centric) | 기술적 화려함보다 실제 이익 등 비즈니스 임팩트 중심 |
| **데이터 중심 설계** (Data-Centric) | 추측이 아닌 사내외 가용 데이터 중심 |
| **인간 중심의 협업** (Human-AI Centric) | 사람을 대체가 아닌 **확장**하는 프로세스 설계 |

## 핵심 개념 (What)

### 1. 전체 기획 프로세스 — 4 Step

서비스 기획은 Discovery → Design → Validation → Execution 4단계로 흐른다. (출처: [원문 02](../lectures/captures/02-ax-consulting-methodology.md))

```
Step1. Discovery   →  Step2. Design   →  Step3. Validation  →  Step4. Execution
(기회 발굴)            (아이디어 설계)      (가치 검증)            (실행 전략 수립)
비즈니스 가치 기반      기술적 실현가능성     투자 효용성·재무효과    성공적 도입 로드맵
과제 식별              고려한 설계
```

| 단계 | 목표 | 세부 활동 |
|------|------|-----------|
| **Step1. Discovery** | 비즈니스 가치 기반 과제 식별 | 비즈니스 프로세스 세분화 · Pain Point 구조화/정량화 · AI 적합 영역 도출 |
| **Step2. Design** | 기술적 실현 가능성 고려 설계 | AI 기반 프로세스 재정의 · 데이터 가용성/구현가능성 진단 · AI Agent 설계 |
| **Step3. Validation** | 투자 효용성·재무 효과 산출 | 비즈니스 임팩트 정량화 · 재무 타당성 산출 · AI 도입 전략 구성 |
| **Step4. Execution** | 성공적 도입 로드맵 | AX 로드맵 수립 · AX 의사결정 지원 · AX 변화관리 전략 |

> Discovery·Design은 아래 **How**의 Phase 1~5에서, Validation·Execution은 **Phase 6~7**에서 실행 수준으로 다룬다. 이 문서 세트의 목표("내년 실행 가능")는 Discovery만으로 충족되지 않으며, Validation(가치 검증)·Execution(실행 로드맵)까지 채워져야 한다. (출처: [강의 덱 V2.7](../lectures/README.md), Step 3·4)

### 2. 실행 도구 — New AI Design Camp 12-Step

4 Step을 워크숍에서 실제로 돌리는 단위가 12-Step이다. 크게 두 Track으로 나뉜다. (출처: [원문 03](../lectures/captures/03-design-camp-12steps.md))

**Track A. AI Agent 적용 Idea 도출** (Step1 Discovery에 해당)
1. 부문/본부 핵심 업무 정의 (Task Identification)
2. AI Agent 대상 영역 분석 (Scope Analysis)
3. AI 구상도 맵핑 *(Optional, Target Scoping)*
4. Pain Point 도출 (Problem Structuring)
5. 근본 원인 분석 (Root-Cause Analysis, 3-Whys)
6. 문제 해결 Ideation (Disruptive Ideation)
7. 적정성 검토 (Impact Validation)

**Track B. AI Agent 실행 Process 구체화** (Step2 Design에 해당)
8. AI 과제 정의 (Agent Definition)
9. As-Is 업무 프로세스 정의 (Task Breakdown)
10. To-Be 업무 프로세스 정의 (Biz-Tech Alignment)
11. Key Success Factor 정의 (Task Prioritization)
12. 개발 일정 수립 (Action Planning)

## 어떻게 사용하는가? (How)

### Phase 1. SOP 정의 → Pain Point → 과제 선정 (3-STEP 요약)

현장에서 가장 먼저 적용하는 간이 3-STEP. (출처: [원문 01](../lectures/captures/01-kakao-3step-pain-point.md))

1. **STEP 1.** 우리 부문/팀/개인의 **SOP(표준업무절차)** 를 정의한다.
2. **STEP 2.** SOP 중 **자동화/효율화가 필요한 Pain Point** 를 식별한다.
3. **STEP 3.** 고통 정도(빈도·심각성·시급성·중요성) / 구현 난이도 / 비용 효용성을 고려해 **최종 AI 과제를 선정**한다.

> 핵심: 모든 업무에 AI를 넣지 않는다. **아프고(고통 큼) · 만들 수 있고(난이도 OK) · 남는 게 있는(효용 큼)** 교집합만 과제로 선정.

**후보 업무 선정 6기준** — "자동화의 효율성"과 "판단의 복잡성" 사이 최적점을 찾는 체크. 후보 업무가 아래에 부합할수록 1차 Agent 대상으로 적합하다. (출처: [강의 덱 V2.7](../lectures/README.md), M1)

| # | 기준 | 핵심 질문 |
|---|------|-----------|
| 0 | **데이터 디지털화** | 입력/출력 데이터가 디지털(텍스트·이미지·로그)로 존재하는가? |
| 1 | **병목 발생** | 이 업무 지연이 전체 프로세스 속도를 늦추고 있는가? |
| 2 | **반복성·빈도** | 일/주 단위로 규칙적으로 발생하는 고빈도 업무인가? |
| 3 | **비즈니스 가치** | 숙련 인력이 단순 반복에 과도한 시간을 쓰고 있는가? |
| 4 | **오류 허용 범위** | AI 실수가 치명적인가, 사람이 검토(Human-in-the-loop)로 보완 가능한가? |
| 5 | **규칙 기반** | 명확한 논리·매뉴얼이 있어 판단 근거를 설명할 수 있는가? |

> 선정 함정: "회의록 작성"(월 2회·30분, 임팩트 낮음)·"내 엑셀 자동화"(개인 효율)는 ✗. → "주 5회·매번 2시간 실적 데이터 취합", "부문 수율 분석 리드타임 단축"처럼 **반복 빈도·소요시간이 크고 팀·부문 핵심 목표에 직결**되는 업무가 ✓.

### Phase 2. 조직 목표와 과제 연계 (Top-Down)

과제는 개인의 흥미가 아니라 **조직 목표에서 연역**되어야 한다. (출처: [원문 04](../lectures/captures/04-org-goal-task-linkage.md))

```
조직 목표  →  업무 목표  →  관련(핵심) 업무  →  AI 대상 후보 업무(과제)
```

예시:
- 시장 지배력 강화 → M/S 확대 → 경쟁사/산업 동향 수집 → **글로벌 인텔리전스 자동 수집·분석**
- 데이터 기반 운영 효율화 → 수요예측 정확도 → 시장 동향 스크랩·보고서 작성 → **정기 리포트 초안 자동 생성**

### Phase 3. 업무 분해와 AI 배치 지점 식별 (L1~L5)

업무를 5단계로 분해해 **AI 개입 최소 단위(L5 Task)** 까지 내려간 뒤, 반복성·정형성·공수가 큰 Task에 Agent를 배치한다. (출처: [원문 05](../lectures/captures/05-process-level-definition.md), [원문 06](../lectures/captures/06-sales-ai-opportunity.md))

```
L1 Mega Process → L2 Process Group → L3 Process → L4 Activity → L5 Task ← AI Agent 배치 지점
```

### Phase 4. Pain Point → 근본 원인 → 해결 아이디어 (산출물)

Discovery의 핵심 산출물. Pain Point를 나열하는 데서 멈추지 않고 **근본 원인(3-Whys)** 을 파고든 뒤 해결 아이디어로 연결한다. (출처: [원문 08](../lectures/captures/08-final-output-example.md), [강의 덱 V2.7](../lectures/README.md) M3·M4)

| Pain Point | 근본 원인 | 해결 아이디어 |
|------------|-----------|----------------|
| 데이터 출처별 기준 불일치 | 단일 참조 기준 미정의 | Mapping Table + Agent 기반 자동 정합화 |
| 정보 수집 수작업 의존 | 자동 수집 체계 부재 | 자동 스크랩·이슈 트래킹 + Keyword Alert |

**Pain Point 정량화 (필수)** — "비효율적이다·번거롭다" 수준이면 다음 단계로 못 넘어간다. 6관점으로 탐색하고 **행동 + 수치**로 표현한다.

- 6관점: 속도·지연 / 품질·정확성 / 반복·비효율 / 협업·커뮤니케이션 / 데이터·시스템 / 규정·정책
- 표현 예: "자주 오류로 지연"(✗) → **"월 평균 3회 오류, 1회당 처리 4시간"**(✓, 누구나 같은 심각성 인식)
- Criticality 4기준(발생 빈도 / 영향 범위 / 시간 소모 / 품질 영향) 중 1개 이상으로 가장 critical한 Pain Point를 선별. 각 L4당 3개 이내.
- 판단 오류 3유형 주의: ① 표현상의 니즈를 문제로 오인(→비즈니스 '손실' 관점), ② Pain 단위를 너무 크게 잡음(→측정 가능한 행동 수준), ③ 증상만 보고 기술로 즉시 번역.

**근본 원인 6유형** — 3-Whys가 "담당자 실수" 같은 사람 탓에 머물지 않도록, 6가지 시각으로 확장해 분류한다.

| 유형 | 설명 | 예 |
|------|------|-----|
| 프로세스 | 절차·표준화·승인 흐름의 구조적 문제 | 승인 절차 길어 지연, 역할·책임 불명확 |
| 시스템 | 기능 부재·통합 미비·자동화 부족 | 시스템 간 연동 불가, 알림 부재로 누락 |
| 데이터 | 정확성·표준화·접근성 부족 | 부서별 수집 기준 불일치, 실시간 조회 불가 |
| 사람 | 역량·인식·책임 부재 | 숙련도 부족, 역할 혼선으로 의사결정 지연 |
| 정책·거버넌스 | 제도·규정·의사결정 체계 문제 | 규정-현실 상충 예외 다수, KPI 기준 불명확 |
| 외부요인 | 시장·법규·환경·공급망 변화 | 정책 변경으로 절차 추가, 협력사 변경 지연 |

> 해결 아이디어도 6유형(시스템 구축/프로세스 표준화/사람 교육/데이터 품질/정책 개선/해결불가→장기과제)에서 **2개 이상 관점을 결합**한다. 한 근본 원인을 단일 처방으로 끝내지 않는다.

### Phase 5. 적정성 검토 — 7항목 스코어카드 (Step 7 / Impact Validation)

여러 후보 과제 중 **무엇을 먼저 할지**를 정량 점수로 합의한다. 7개 항목을 후보별로 3점 척도(1·2·3)로 채점, 총 21점 만점. (출처: [강의 덱 V2.7](../lectures/README.md), M5)

| 평가 기준 | 항목 | 질문 | 척도 |
|-----------|------|------|------|
| **전략적 중요도** | ① 본원적 경쟁력 | 조직 핵심 사업과 연계성이 깊은가? | 1–2–3 |
| (Strategic Need) | ② 사업전략·목표·KPI | 당해 전략·목표·KPI와 연계성이 높은가? | 1–2–3 |
| **비즈니스 성과** | ③ 수익성-규모 | 비용 절감 또는 매출/효율 향상이 가능한가? | 1–2–3 |
| | ④ 수익성-수혜조직 | 수혜 조직의 규모/범위는 얼마나 되는가? | 1–2–3 |
| | ⑤ 확장성(Scalability) | 향후 유사 공정/상황으로 수평 전개·고도화 가능한가? | 1–2–3 |
| | ⑥ 시급성(Urgency) | 추진이 늦어지면 리스크(법규·손실)가 발생하는가? | 1–2–3 |
| **실행 용이성** | ⑦ 데이터 준비도 | 데이터가 충분하거나 준비 계획이 구체적·실현가능한가? | 1–2–3 |
| **총점** | | | **___ /21** |

채점 루브릭(예: ① 본원적 경쟁력) — 3점=핵심 경쟁우위를 직접 강화/차별화 가치 창출에 필수, 2점=운영 효율 개선·기존 사업 보조, 1점=핵심 활동과 관련성 낮음·단순 행정. ⑦ 데이터 준비도 — 3점=시스템 내 표준화·적재/정제 완료, 2점=산발적 존재(추가 정제 필요), 1점=데이터 없음/품질 낮음.

> 점수는 **정량값 자체보다 항목별 논리적 근거**가 핵심. 1점 항목이 있어도 전략적 중요도·시급성이 높으면 선정 가능하되, **그 제약을 어떻게 극복할지(예: "데이터 확보는 1Q 내 IT팀 협의")를 로드맵에 반영**해야 한다. 조원 간 점수 차가 큰 항목은 이유를 파악해 하나로 합의한다.

### Phase 6. 가치 검증 — Validation (Step 3)

선정 과제가 **투자할 가치가 있는지**를 재무·전략 관점으로 검증한다. "내년 실행 가능" 문서가 되려면 이 단계가 채워져야 한다. (출처: [강의 덱 V2.7](../lectures/README.md), Step 3)

1. **비즈니스 임팩트 정량화** — 수익성·확장성·시급성 관점으로 산출. 신뢰성 확보를 위해 보수적 **'최소 기대치'를 먼저, 'AI 도입 시 최대 기대치'를 다음**으로 산출(범위 제시).
2. **재무 타당성 산출**
   - **CapEx** = AI 인프라 구축비 + 추가 개발비 + 초기 데이터 정제/학습비
   - **OpEx** = 클라우드 사용료 + 유지보수 인건비 + 재학습비 + AI 모델 사용료
   - 투자 비용은 AI 사용량에 따라 상/중/하 가정으로 산출
   - **ROI = (투자 수익 − 투자 비용) / 투자 비용 × 100**
3. **AI 도입 전략 — 3대 질문(게이트)**
   - **전략 방향성 일치**: 이 과제가 전사 AX 전략·부서 KPI에 어떻게 기여하는가?
   - **재무적 타당성**: 투자 회수 기간은 언제이며, **순현재가치(NPV)는 양수**인가?
   - **도입 가능성**: 현업의 변화 수용도가 충분하며 리스크는 통제 가능한가?

> 사내 LLM 제약상 외부 API 비용은 제외되나, 사내 엔드포인트 사용에 따른 GPU·운영 공수를 OpEx로 산정한다. 실측 불가 항목은 엔지니어 확정 플레이스홀더(τ/s/r/N 등)로 둔다.

### Phase 7. 실행 전략 — Execution (Step 4)

검증된 과제를 **단계적으로 도입**하기 위한 로드맵·의사결정·변화관리. (출처: [강의 덱 V2.7](../lectures/README.md), Step 4)

**(1) AX 로드맵 — 3-Phase 단계적 이행** (단계별 성과지표 검증 후 다음 단계로 전환)

| Phase | 명칭 | 핵심 | 기간 |
|-------|------|------|------|
| Phase 1 | **PoC** | 핵심 가설 검증 | **2–3개월 내 단기 수행** |
| Phase 2 | **MVP** | 실무 적용 · 피드백 반영 · 핵심 사용자 확산 · 시스템 안정화 | — |
| Phase 3 | **전사 확산** | 표준 인프라 통합 · 타 조직 수평 전개 | — |

**(2) 포트폴리오 — Long List → Short List → Phased Plan**

여러 과제를 한 번에 추진하지 않는다. 아래 기준으로 우선순위화해 Short List를 뽑고 Phase에 배치한다.

- **비즈니스 가치**: 전략적 정렬(KPI) · 재무적 수익 · 고객/사용자 경험 · 시장 경쟁력
- **실행 가능성**: 데이터 준비도 · 기술적 난이도 · 리소스 가용성 · 확장성
- **시급도**: 비즈니스 긴급도 · 후행 과제의 선행 조건 여부
- **리스크·규제**: 윤리/보안(개인정보·편향·투명성) · 법적 규제 준수

**(3) AX 의사결정 지원** — 기술 용어가 아닌 **비즈니스 중심 언어**로 구조화. 이해관계자를 **영향력 × 관심도** 매트릭스로 분석해 핵심 관리 대상(집중관리·만족도 유지)을 식별.

**(4) 변화관리 — 저항요인 3종 대응**

| 저항요인 | 대응 |
|----------|------|
| **일자리 불안** | 단순 반복업무 대신 고부가가치 업무로 전환 안내 |
| **결과의 불신** | 설명가능한 AI + 사람 최종 검수 → 투명성·책임감 강조 |
| **사용 복잡성** | 쉬운 UI/UX 설계 + 밀착 교육 지원 |

> 단계적 확산을 위한 전문 조직 육성을 병행한다.
