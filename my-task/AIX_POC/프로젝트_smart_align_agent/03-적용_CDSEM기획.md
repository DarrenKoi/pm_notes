# AIX 적용 사례 — CD-SEM Auto Recipe Creation (기획/Discovery)

> [기획 방법론 틀(01)](../_가이드/01-기획문서_AX서비스기획.md)을 ITC AIX 실제 과제에 적용한 **첫 번째 사례**. New AI Design Camp Track A(Discovery, 1~7단계)를 따라 **CD-SEM Auto Recipe Creation** 과제를 발굴·구조화한다. 실행/To-Be 설계(Track B)는 후속 문서 `04-적용_CDSEM기술.md`에서 다룬다.

>
> 🔄 **변경 이력 (2026-06-29)**: R&D 연구소 컨텍스트 정정 — 구 L2 "측정 실행(양산 Run)"을 삭제하고, 양산 이관은 장비 개발 흐름의 종착점(장비 개발 & 양산 이관)으로 흡수했다. "양산"은 R&D 연구소의 운영 대상이 아니라 개발 완료품을 양산으로 이관하는 별도 업무다. End-to-End SOP의 [C] 단계를 "양산 Run"에서 "검증·등록 + 등록 Recipe 자동 계측"으로 재정의. (셋업 ①~⑦·병목 ③④⑥·근본원인 (a)(b)·PoC ③·⑥는 As-Is로 보존.)

## 과제 한 줄 정의

**Auto Recipe Creation — VLM 기반 GUI 제어(computer-use)로 CD-SEM Recipe를 자동 생성·등록한다.** VLM이 SEM 화면을 직접 보며 GUI를 클릭·입력해 Recipe를 생성하고, **실측 시작 시 발생하는 좌표 shift·align 실패·패턴 오인식을 실시간으로 재정합·수정**한다(1차 PoC = Align fail 대응 자동화 = **Smart Align Agent**). 엔지니어는 모니터링하다 **예외만 개입(Human-in-the-loop)** 한다. 완전자동화는 목표가 아니다.

## 왜 이 과제인가? (Why)

상위 방법론의 3대 원칙([원문 02](../lectures/captures/02-ax-consulting-methodology.md))에 비춰 이 과제가 적합한 이유:

| 원칙 | 이 과제에서의 충족 |
|------|--------------------|
| **가치 중심** | **R&D 디바이스 개발 지연 최소화**라는 조직 목표에 직결. 최대 병목(오인식 수정 반복)을 정조준. |
| **데이터 중심** | SEM 화면 이미지·reference recipe 등 **이미 존재하는 가용 데이터**로 동작. 추측 설계가 아님. |
| **인간 중심** | 엔지니어를 대체하지 않고 **확장** — 반복 셋업은 AI, 예외 판단·최종 검증은 사람. |

특히 이 과제는 일관 기준 등록으로 **측정 장비 간 Skew Zero**(숙련도·장비 편차 제거)를 확보하고, 사람의 작업시간 제약을 풀어 **24h 무중단 셋업 → MI 장비 가동률 극대화**(유휴시간 축소)를 가능케 한다는 점에서, 단순 효율화를 넘어 장비 운영/관리 최적화·검계측 효율화에 본원적으로 기여한다.

## 핵심 구조 (What)

### 1. 조직 목표 ↔ 과제 연계 (Top-Down)

과제는 개인 흥미가 아니라 조직 목표에서 연역한다([원문 04](../lectures/captures/04-org-goal-task-linkage.md) 양식 적용).

```
조직 목표            →  업무 목표              →  관련(핵심) 업무                →  AI 대상 후보 과제
R&D 디바이스           ① 검/계측 효율화          ① MI Recipe Solution 제공          Auto Recipe Creation
개발 지연 최소화        ② 장비 운영/관리 최적화    ② 측정 장비 간 Skew Zero           (1차 PoC = Align fail 대응 자동화)
(핵심)                 ③ 장비 개발·도입          ③ MI 장비 가동률 극대화
                                                ④ 장비 개발·도입 및 양산 이관
```

> 핵심업무 ①~③에 **한 과제(Auto Recipe Creation)** 가 동시에 기여한다 — ① 솔루션 본체, ② 일관 기준으로 숙련도·장비 편차 제거(구 "숙련도 편차 제거" 흡수), ③ 24h 무중단으로 유휴시간 축소(구 "처리량 확대 / 24h 무중단" 흡수). ④ 장비 개발·도입 및 양산 이관은 업무 목표 ③의 추적 도착지로, 양산 이관(개발 완료품을 양산으로 넘기는 별도 업무)을 포함하는 **ARC 범위 밖** 핵심 업무다. 추적성은 위→아래 한 방향.

### 2. 업무 분해 — L1~L5 (AI 배치 지점)

업무를 5단계로 분해해 AI 개입 최소 단위(L5 Task)까지 내려간다([원문 05](../lectures/captures/05-process-level-definition.md) 체계 적용). **L4를 셋업만이 아니라 의뢰 접수~모니터링 End-to-End로 펼친 뒤** AI 배치 후보를 식별한다.

```
L1 Mega Process : 계측(MI) 운영
  L2 Process Group : CD-SEM Recipe Management
    L3 Process : Recipe Creation
      L4 Activity : [A] 의뢰 접수·검토 → [B] Recipe 셋업(①~⑦) → [C] 검증·등록 → [D] 후속 계측 모니터링
        L5 Task (AI 배치 후보) : B-③ 패턴 등록 · B-④ 좌표·박스 · B-⑥ 오인식 수정 (핵심)
                                · D-① Recipe 준수 모니터링·이상탐지 (차기) · A-2 의뢰서 부족정보 플래깅 (보조)
```

> L3는 **Recipe Creation** — 의뢰 접수부터 셋업·검증·등록, 그리고 후속 계측 모니터링까지 Recipe 한 생애를 본다. (구 "Creation & Operation"의 *양산 Run* 구간은 R&D 컨텍스트 정정으로 제외 — 양산은 운영 대상이 아니라 이관 업무다. 세부 경계는 엔지니어 확인 항목.)

## 발굴 절차 (How)

> 깔때기 원칙 — **(1) End-to-End SOP를 펼치고 → (2) 단계별 Pain을 망라하고 → (3) 근본원인을 파고 → (4) 정량 기준으로 우선순위화해 과제를 선정**한다. 결론(⑥ 오인식)만 제시하지 않고 *선정에 이르는 과정*을 재현한다.

### Phase 1. End-to-End 표준 업무 절차(SOP) 정의 → 병목·공백 식별

Recipe Creation 셋업만이 아니라 **의뢰 접수부터 검증·등록·후속 계측 모니터링까지** Recipe 한 생애 전체 흐름을 펼친다(엔지니어 확인: [B] 셋업 ①~⑦은 확정, [A]·[C]·[D]의 세부 단계·경계는 확인 필요 항목).

```
[A] 의뢰 접수·검토
  A1 공정/소자 엔지니어가 MIDAS에 측정 의뢰서 작성·제출 (도면·측정위치·파라미터·측정방식, 현재 PPT 형태)
  A2 계측 엔지니어 의뢰서 접수·정보 충분성 검토 (부족 시 문의·반려)

[B] Recipe 셋업  (= 기존 As-Is ①~⑦, Recipe Creation)
  ① 측정 대상 레이어/패턴 확인 (도면·스펙 검토)
  ② 장비 GUI에서 신규 Recipe 생성 → 웨이퍼/샘플 로드
  ③ Alignment / Addressing 포인트 등록 (패턴 인식 템플릿 지정)   ❗ 병목
  ④ 측정 포인트(EP) 좌표·측정박스·측정 알고리즘 설정              ❗ 병목
  ⑤ 이미징 조건(배율·전압·FOV 등) 설정
  ⑥ 시험 측정(Try) → 오인식/실패 지점 수정 (반복)                ❗❗ 최대 병목
  ⑦ Recipe 저장·등록 → 검증

[C] 검증·등록  (⑦의 연장)
  C1 검증 완료 Recipe 등록 → 등록 Recipe로 개발 샘플 자동 계측 (수작업 거의 없음)
  C2 계측 결과 수집·1차 판정 (정상/이상)

[D] 후속 계측 모니터링·이상 대응
  D1 측정 안정성·Recipe 준수 모니터링 (재현성·드리프트·장비 상태)  ❗ 공백 (사람 주기점검·야간/주말 의존)
  D2 이상 원인 분석 (오인식 vs 장비 vs 공정 변동 구분)
  D3 Recipe 보정·재셋업 트리거 → [B]로 피드백 루프
```

- **셋업 병목 = B-③④⑥**(그중 ⑥ 오인식 수정 반복이 최대 고통), **운영 공백 = D-1 모니터링**(사람 주기점검 의존 → 이상 늦은 발견).
- **[A] 의뢰**: 의뢰서 정보 품질이 셋업 난이도를 좌우. MIDAS 전산화는 **별도 프로젝트(외부 변수)** 로, 본 과제는 부족정보 플래깅 선에서 연계.
- **[C] 검증·등록·자동 계측**: 등록 Recipe로 개발 샘플이 자동 측정되어 *수작업 비중이 낮음* → 자동화 후보 우선순위 낮음(단, 셋업 부실 시 오측 유발). ※ "양산 Run"이 아니라 R&D 개발 단계의 계측이며, 양산은 운영 대상이 아닌 별도 이관 업무다.
- **⑤ 이미징 조건**은 reference recipe 참고(동일) 또는 신규 공정은 엔지니어 협의.

### Phase 2. 망라적 Pain Point 리스트 (SOP 단계별)

선택된 가지 하나가 아니라 **A~D 전 단계의 Pain을 빠짐없이** 깐다. 심각도(🔴)는 브레인스토밍 예비 표기 — 정량 점수는 Phase 4에서.

| 단계 | Pain Point | 심각도(예비) | 비고 |
|------|------------|:---:|------|
| A 의뢰 | 의뢰서 정보 부족·표준 미흡 → 셋업 시 재문의·추측 | 🔴 | 숙련도 편차의 입력측 원인. MIDAS 전산화(외부 변수)와 연계 |
| B-③④ 셋업 | 패턴 등록·좌표/박스 설정이 수작업·숙련 의존 | 🔴🔴 | 셋업 병목 |
| **B-⑥ 셋업** | **시험측정 후 오인식/실패 수정 반복** | 🔴🔴🔴 | **최대 병목** (빈도·공수·숙련 의존 모두 높음) |
| B 전반 | 야간/주말 셋업 공백 (작업시간 제약) | 🔴 | 처리량 상한 제약 |
| C 검증·등록 | 셋업 부실 시 오측 유발 (등록 Recipe 자동 계측, 수작업 적음) | 🔵 낮음 | 자동화 여지 적음 |
| D-1 모니터링 | 이상 탐지가 사람 주기점검 의존 → 늦은 발견·야간 공백 | 🔴🔴 | **신규 자동화 후보**(실시간 이상탐지) |

### Phase 3. 근본 원인(3-Why) — 집중 Pain

상위 Pain을 근본 원인까지 파고들어 해결 아이디어로 연결한다([원문 08](../lectures/captures/08-final-output-example.md) 양식 적용).

| Pain Point | 근본 원인 (3-Why) | 해결 아이디어 |
|------------|-------------------|----------------|
| 오인식 수정 반복(B-⑥)이 오래 걸림 | 템플릿을 미리 만들어도 **실측 시작 시 전체 좌표 shift / align 실패** 발생 | **VLM이 SEM 이미지를 보고 실시간 좌표 재정합·재align, 오인식 자동 수정** (핵심) |
| 숙련도별 해석 편차 | 표준 판단기준 부재 + 의뢰자가 의뢰서를 명확히 작성하지 않는 경우 | VLM이 일관된 기준으로 수행 + 의뢰서 부족정보 플래깅 (MIDAS 연계) |
| 야간/주말 셋업 공백 | 사람 작업시간이라는 물리적 제약 | VLM 24h 무중단 수행, 사람은 예외만 개입 |
| 모니터링 이상 늦은 발견(D-1) | 실시간 자동 감시 부재, 사람 주기점검 의존 | (차기) VLM/룰 기반 실시간 측정 이상탐지 |

> 집중 근본 원인 = **(a) 패턴 인식 실패, (b) 측정박스/좌표 오류.** 셋업 해결 아이디어 전부가 이 둘로 수렴한다.

### Phase 4. 정량 우선순위화 & 과제 선정 (Impact Validation)

망라한 Pain을 **자동화 후보로 묶어 동일 잣대로 스코어링**한다. 기준 = 과제 선정 3-STEP([원문 01](../lectures/captures/01-kakao-3step-pain-point.md))의 교집합 — **고통(빈도·심각성·주요성) · 구현 난이도 · 비용 효율**.

> **점수 정의(척도)** — 빈도(발생 잦음), 심각성(불량·재측정 파급), 주요성(셋업 리드타임 기여). 표의 高/中/低는 **브레인스토밍 예비 평가(가설)** 이며, **1~5 점수·가중치·종합점수는 캠프에서 엔지니어가 정량 확정**한다(절대 수치 아님 — placeholder).

| 자동화 후보 | 대상 단계 | 빈도 | 심각성 | 주요성 | 구현 난이도 | 비용 효율 | 종합 우선순위 |
|-------------|-----------|:---:|:---:|:---:|:---:|:---:|:---:|
| **오인식 수정 자동화** (VLM 좌표 재정합) | B-⑥ | 高 | 高 | 高 | 中 (좌표정확도 리스크) | 高 | **1순위** |
| 패턴·좌표 설정 자동화 (VLM) | B-③④ | 高 | 中 | 高 | 中 | 高 | 2순위 *(⑥과 동일 Agent로 묶음)* |
| 실시간 측정 이상탐지 | D-1 | 中~高 | 高 | 高 | 中 | 中 | 3순위 *(차기 Phase)* |
| 의뢰서 부족정보 플래깅 | A-2 | 中 | 中 | 中 | 低 | 中 | 보조 *(MIDAS 외부 변수 연계)* |
| 등록 Recipe 자동 계측 | C-1 | — | 低 | 低 | (이미 자동) | — | 제외 *(자동화 여지 적음)* |

**선정 결과** — **1순위 = B-⑥ 오인식 수정 자동화**(+ B-③④를 같은 VLM Agent로 묶어 ②~⑥ 연속 수행). D-1 실시간 이상탐지는 **차기 Phase 후보**, A-2 플래깅은 **보조**(MIDAS 연계), C-1은 제외.

- **남는 리스크** = ⑥의 구현 난이도(정밀 좌표 재정합 정확도) → 회피하지 않고 **KSF#1 · Phase 0 PoC로 1순위 검증**([04](./04-적용_CDSEM기술.md) Step 11~12, [05](./05-적용_CDSEM_PoC실험설계.md)).

### Phase 5. To-Be 방향성 (Human-AI 경계 — 개념 수준)

상세 Swimlane은 Track B(`04`)에서 그리되, 발굴 단계에서 합의된 역할 경계는 다음과 같다.

- 🤖 **AI**: B-②~⑥ GUI 직접 조작 + SEM 이미지 판독 기반 좌표/align 재정합 + 오인식 수정.
- 🧑 **Human**: A 의뢰서 검토(필요시), 예외 핸드오프 처리, B-⑦ 최종 검증·승인.
- 🔭 **차기 범위**: C 검증·등록·자동 계측·D 모니터링은 1차 범위 밖 — D-1 Recipe 준수 모니터링·실시간 이상탐지를 차기 후보로 둔다.

## 기술/환경 제약 (기획 반영, 상세는 Track B)

- **외부 LLM API 전면 차단.** 사내 OpenAI-호환 내부 엔드포인트만 사용 — **Qwen3-VL-8B / Qwen3-VL-30B(비전)**, Kimi-K2.5(텍스트), BGE-M3(임베딩).
- DB(OpenSearch/ES)는 로컬 미가용 — 검증은 import/문법 수준.
- 99% 사내 문서가 DRM → 스크린샷+VLM 추출이 기본 파이프라인.
- **MIDAS 전산화는 외부 변수** — 본 과제는 의뢰서 정보 부족을 플래깅하는 선에서 연계.
