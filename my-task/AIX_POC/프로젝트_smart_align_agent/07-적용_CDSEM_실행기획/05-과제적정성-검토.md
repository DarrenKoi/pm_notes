# 05 · 과제 적정성 검토 (Step7 + Validation 흡수)

> 선정 후보 **2개** — **① CD-SEM Auto Recipe Creation (ARC, VLM GUI-제어 기반)** 와 **② 의뢰서 검증 AI Agent**(#04에서 (A)(B) 시스템 근본원인과 분리한 **상류 데이터 갈래 Pain①** 이 점수화 후보로 승격) — 를 7문항 × 1~3점 스코어카드로 **나란히 점수화**해, 종합 Σ/21과 착수 임계 τ_gate로 게이트 판정한다. 결과: **① 19/21 ≥ τ_gate → 1순위 즉시 착수**(→ [#06 과제 정의](./06-ai과제-정의.md)), **② 12/21 < τ_gate → 게이트 미달 → 보류/차기 과제**(MIDAS 전산화·의뢰서 표준화 선결 후 재평가). 이 종합점수가 **#10 착수(Execution) 게이트의 정량 근거**다.

## 왜 필요한가 (Why)

Discovery(#01~#04)는 "이 과제가 조직목표에서 연역됐고(#01), 대상 업무가 명확하며(#02), Pain이 정량화됐고(#03), 근본원인까지 파고들었다(#04)"를 보였다. 그러나 **발굴됐다는 사실이 곧 착수해도 좋다는 뜻은 아니다.** 자원이 한정된 이상, 후보를 객관적 잣대로 점수화해 "지금 이 과제를 실행할 가치가 있는가"를 게이트로 통과시켜야 한다.

- 과제 선정 3-STEP의 STEP 3은 **"고통의 정도(빈도·심각성·시급성/중요성) · 구현 난이도 · 비용 효용성을 고려해 최종 AI 과제를 선정한다"** 고 못박는다 (출처: [원문 01](../../lectures/captures/01-kakao-3step-pain-point.md)). 이 단계가 바로 그 STEP 3의 정량 게이트다.
- 방법론 4 Step 중 **Step 3 Validation(가치 검증)** = "투자 효용성 및 재무적 효과 산출 / 비즈니스 임팩트 정량화 / 재무적 타당성 산출" (출처: [원문 02](../../lectures/captures/02-ax-consulting-methodology.md)). 본 문서의 **'비즈니스 성과' 카테고리(수익성·확장성·시급성)** 가 이 Validation을 흡수한다 — 별도 Validation 문서를 두지 않고 적정성 스코어카드 안에 ROI 검증을 박는다.
- #03의 Pain 정량 컬럼(빈도·심각성·주요성)과 #04의 근본원인이 본 스코어카드의 **백데이터**다. 점수는 여기서 새로 지어내는 것이 아니라 앞 단계 산출물에서 **연역**된다(추적성).

> **이번 M5의 확장점 — 2-후보 비교**: ARC 단독이 아니라 후보를 **2개 나란히** 점수화해 상대 비교한다. #04에서 (A) 패턴 인식 실패·(B) 측정박스/좌표 오류 [시스템] 근본원인과 **분리한 상류 데이터 갈래 Pain①(의뢰서 정보 부족·표준 미흡)** 을 독립 후보②(의뢰서 검증 AI Agent)로 승격시켜, 같은 게이트 잣대로 "지금 착수(①) vs 선결 후 재평가(②)"를 정량 판정한다 (출처: [#04 근본원인 ideation](./04-근본원인-ideation.md) §3 — Pain① 상류 후순위 분리; 브레인스토밍 합의 2026-06-29).

## 핵심 산출 (What)

**3카테고리 × 7문항 × 2후보 비교 스코어카드** (각 문항 1~3점, 후보별 종합 Σ/21):

| # | 카테고리 | 문항 | 흡수 단계 |
|---|----------|------|-----------|
| 1 | 전략적 중요도 (Σ_A/6) | 본원적 경쟁력 (조직 본원적 활동 연계성) | Step7 |
| 2 | 전략적 중요도 | 사업전략·목표·KPI 연계성 | Step7 |
| 3 | **비즈니스 성과 (Σ_B/12)** | **수익성 — 규모 (ROI = 절감공수 ΔS × 처리량 N)** | **Validation** |
| 4 | **비즈니스 성과** | **수익성 — 수혜조직 규모/범위** | **Validation** |
| 5 | **비즈니스 성과** | **확장성** | **Validation** |
| 6 | **비즈니스 성과** | **시급성** | **Validation** |
| 7 | 실행 용이성 (Σ_C/3) | 실행 용이성 (데이터 준비 · 준비계획 구체성) | Step7 |

> **점수 척도**: 1점=미흡/불확실, 2점=보통, 3점=우수/확실. 본 표의 점수는 **대영(엔지니어) 확정 기준값**이다. 단, **①의 3번(수익성-규모)·7번(실행 용이성)** 은 ΔS·N 실측 및 데이터 라벨링·PoC 측정 전이라 **잠정/실측대기** 값이며, 가중치·최종 확정은 캠프/Champion이 정량 확정한다. (#03 5/5/5와 달리 여기 점수는 1~3점 척도 — 혼동 주의.)

## 작성 결과 (How: 2-후보 스코어카드 + Validation)

### 스코어카드 (① ARC vs ② 의뢰서 검증)

#### A. 전략적 중요도 (Σ_A / 6)

| 문항 | ① ARC | ② 의뢰서 검증 | 근거 |
|------|:---:|:---:|------|
| **1. 본원적 경쟁력** (본원적 활동 연계) | **3** | **2** | ① 계측(MI) Recipe Creation의 최대 병목(⑥)을 정조준하는 **핵심 가치사슬** 과제 (출처: 03 Phase 1·4). ② 의뢰서 검증은 셋업 병목을 *직접 해결*하지 않고 **입력 데이터 품질 보조** 성격이라 본원성 한 단계 낮음. |
| **2. 사업전략·목표·KPI 연계성** | **3** | **2** | ① #01 Top-Down 연계표에서 `조직목표 → 업무목표 → 핵심업무 → ARC`로 직접 연역, KPI(개발지연·가동률·Skew)에 1:1 매핑 (출처: [#01](./01-전략일관성-확인.md), 03 §1). ② 상류 데이터 품질 개선은 KPI에 **간접** 기여(셋업 효율의 전제 조건). |

#### B. 비즈니스 성과 (Σ_B / 12) — **Validation/가치검증 흡수 지점**

| 문항 | ① ARC | ② 의뢰서 검증 | 근거 |
|------|:---:|:---:|------|
| **3. 수익성 — 규모** (ROI = ΔS × N) | **3** *(잠정)* | **2** | ① ROI = (As-Is 셋업공수 s − To-Be s′) × 연간 처리량 N. ⑥이 최대 병목이라 ΔS 대부분이 ⑥ 절감에서 나옴 — 규모 큼 (출처: 03 Phase 4). **단 s·s′·N은 #07/#08 실측 전이라 점수 3은 잠정/실측대기.** ② 절감 규모는 셋업 직접 절감 대비 작음(재문의·추측 공수 절감 수준). |
| **4. 수익성 — 수혜조직 규모/범위** | **2** | **1** | ① 셋업 수행 계측팀 중심 — 수혜 폭 보통. ② 의뢰 전 부서(공정/소자)에 닿으나 **MIDAS 전산화 전이라 실제 적용 범위 협소** → 1점. |
| **5. 확장성** | **3** | **2** | ① ref recipe 보유 layer로 수평 확장, 동일 VLM Agent를 타 layer·장비군으로 일반화 → 확장 layer 수가 N을 키워 ROI 증폭 (출처: 03 Phase 4, #06 적용범위). ② 의뢰서 표준 정착 시 타 측정군으로 확장 가능하나 MIDAS 의존. |
| **6. 시급성** | **3** | **2** | ① 야간/주말 셋업 공백·가동률 제약이 현재 진행형, 개발 수요 증가 시 즉시 개발 지연으로 전이 (출처: 03 Phase 2). ② 입력 품질 보조 성격이라 시급성 한 단계 낮음(병목 직접 해소 아님). |

#### C. 실행 용이성 (Σ_C / 3)

| 문항 | ① ARC | ② 의뢰서 검증 | 근거 |
|------|:---:|:---:|------|
| **7. 실행 용이성** (데이터 준비 · 준비계획 구체성) | **2** *(잠정)* | **1** | ① SEM 이미지·ref recipe가 **이미 가용**(강점)하고 KSF#1·Phase0 PoC 실험설계가 존재 (출처: 03 Why "데이터 중심", [PoC 05](../05-적용_CDSEM_PoC실험설계.md)). **단 라벨링 공수·배율/노이즈 편차·DRM·PoC 측정 미완이라 2점은 잠정/실측대기**(#09 데이터 KSF에서 다룸). ② 구현 난이도 자체는 낮으나 **의뢰서 데이터가 비표준·미전산**이라 바로 학습/검증 불가 → 1점. |

### 종합 점수 (Σ / 21) — 후보 비교

```
                 ① ARC              ② 의뢰서 검증
Σ_A 전략 /6      3 + 3        = 6     2 + 2        = 4
Σ_B 비즈 /12     3 + 2 + 3 + 3 = 11   2 + 1 + 2 + 2 = 7
Σ_C 실행 /3      2            = 2     1            = 1
───────────────────────────────────────────────────
Σ /21            19 / 21             12 / 21  ⁽*⁾
```

> ⁽*⁾ **② 의뢰서 검증 = 항목 합 기준 12/21.** 원본 시트 총점은 **11**로 기재돼 있으나(1점 차 집계), 본 문서는 **각 문항 점수의 합(12)** 을 기준으로 적는다. (시트 표기 11 병기 — 캠프에서 합산 정정 확인 필요.)

- **① ARC = 19/21**: 전략 6/6 + 비즈니스 11/12 + 실행 2/3. (3번 수익성-규모·7번 실행 용이성은 잠정/실측대기 — 위 근거 참조.)
- **② 의뢰서 검증 = 12/21**: 입력 데이터 품질 '보조' 성격 + MIDAS 전산화 전 적용범위 협소 + 의뢰서 비표준·미전산이 점수를 끌어내림.

| 항목 | 값 | 판정 |
|------|:---:|------|
| **착수 임계 τ_gate** (제안) | **14~15 / 21** | 조별/Champion 합의 필요 (placeholder — 조직 자원·리스크 허용치에 맞춰 확정) |
| **① ARC** | **19 / 21** | **≥ τ_gate → 1순위 즉시 착수** → [#06 과제 정의](./06-ai과제-정의.md) 진행 |
| **② 의뢰서 검증** | **12 / 21** | **< τ_gate → 게이트 미달 → 보류/차기 과제** (아래 로드맵 참조) |
| 게이트 승인 주체 | `<담당 임원>`(Champion) | 확정 후 게이트 승인 주체 명기 |

- **판정 요약**: ① ARC는 임계를 명확히 상회(19 ≥ 14~15)하므로 **1순위로 즉시 착수**한다. 이는 #04에서 확정한 집중 근본원인 (A)(B) 해소 과제이자 03 Phase 4 "1순위 = B-⑥ 오인식 수정 자동화"와 일치한다(기존 결론 보존). ② 의뢰서 검증은 임계 미달이라 **이번 사이클 착수 대상이 아니다.**

### ② 의뢰서 검증 — 차기 로드맵 (선정 보류 시 극복방안 명시)

방법론 원칙상 **1점 항목이 있어도 선정한다면 그 항목의 극복방안을 명시**해야 한다. ②는 임계 미달로 *이번엔 보류*하되, 점수를 끌어내린 1점 항목(4번 수혜범위·7번 실행 용이성)의 **선결조건을 분명히 해 차기 재평가 경로**로 등재한다.

```
② 의뢰서 검증 AI Agent  =  보류 / 차기 과제
   선결조건 (1점 항목 극복):
     ├─ 4번(수혜범위 협소)  ← MIDAS 전산화 또는 의뢰서 표준화로 적용범위 확보
     └─ 7번(데이터 비표준)  ← 의뢰서 표준 필수항목 정의 → 전산 데이터화
   → 선결 충족 시 본 스코어카드로 재평가, τ_gate 재판정
```

- **#09 KSF 연계**: ②의 선결조건은 #09 KSF 문서의 **상류 데이터 의존성 플래그**로 넘긴다 — 의뢰서 표준화·MIDAS 전산화는 본 과제(ARC) 범위 밖 **외부 변수**이므로(출처: 03 Phase 1·5; SPEC §2), ② 착수는 그 외부 변수 해소에 종속된다. (참고: [#09 KSF·제약](./09-ksf-제약-정의.md))
- **#04 정합**: ②(의뢰서)는 #04에서 (A)(B)와 묶지 않고 **상류 데이터 후순위로 분리**한 갈래 그대로다 — 본 게이트가 그 분리를 정량(12 < 14~15)으로 재확인한다.

### 가치·재무 정량화 (Validation) — ① ARC 중심

본 단계가 4 Step의 **Step 3 Validation(가치 검증)** 을 흡수한다 (출처: [원문 02](../../lectures/captures/02-ax-consulting-methodology.md) Step3 "투자 효용성 및 재무적 효과 산출"). 게이트를 통과한 **① ARC** 의 **3·5번(수익성-규모·확장성) 칸이 곧 ROI 검증**이며, 별도 Validation 문서 대신 적정성 게이트 안에 재무 타당성을 박는다.

**ROI 산식 (placeholder 변수만 — 수치는 캠프 확정):**

```
연간 절감 가치  ΔValue = ΔS × N
  ΔS = s − s′      (셋업 1건당 공수 절감)
       s  = As-Is 셋업 소요시간   → #07 As-Is 프로세스에서 실측 (s placeholder)
       s′ = To-Be 셋업 소요시간   → #08 To-Be 프로세스에서 실측 (s′ placeholder)
       (ΔS의 대부분 = ⑥ 공정 variation 오측·인식 실패 반복 수정 절감, Recipe Management 최대 병목 — 출처 03 Phase 4)
  N  = 연간 처리량 (셋업 건수)     → 확장 layer 수에 비례 (N placeholder)
       + 24h 무중단 → MI 장비 가동률↑ (유휴시간 축소, 단순 효율화 초과 — 출처 03 Why)

ROI ≈ ΔValue / (구현·운영 투자)   → 투자측 placeholder는 #10 인력·일정에서 산정
```

- **수치 금지 원칙**: s·s′·r·N·τ_gate는 전부 placeholder다. 실측은 #07/#08, 검증 프로토콜은 [PoC 실험설계 05](../05-적용_CDSEM_PoC실험설계.md), 일정·인력 투자측은 #10에서 확정한다. 본 문서는 **산식과 게이트 구조 + 후보 비교 점수만 박제**한다.
- **확장성 = ROI 레버**: 확장 layer 수가 N을 키워 ΔValue를 증폭하므로, ① ARC의 5번(확장성)은 3번(수익성-규모)과 곱셈 관계로 묶인다.
- **② 의뢰서 검증의 가치**는 정량 ROI보다 **상류 데이터 품질(셋업 난이도 입력 변수) 개선**이라는 정성 가치 — ARC ROI의 *전제를 보강*하는 성격이라, 선결조건 해소 후 별도 재평가한다.
