# 07 · As-Is 업무 프로세스 정의 (Step9 / Design)

> CD-SEM Recipe Creation 셋업을 현재 어떻게 사람이 수행하는지 ①~⑦ 플로우차트와 단계별 테이블(소요시간·사용 시스템)로 고정해, To-Be 설계(#08)·수익성(#05)·기대효과(#10)의 백데이터로 삼는다.

## 왜 필요한가 (Why)

To-Be를 설계하려면 **무엇을 바꾸는지의 기준선(baseline)**이 먼저 있어야 한다. As-Is가 정량화되어 있지 않으면 "얼마나 좋아졌는가"를 말할 수 없고, 결국 #05 과제적정성(수익성 ΔS)도 #10 개발일정의 기대효과도 공중에 뜬다.

- **#08 To-Be의 출발선**: To-Be 플로우차트는 As-Is ①~⑦의 같은 축 위에 🤖/🧑 레인을 다시 칠하는 것이다. As-Is가 확정돼야 대비 테이블의 before 칸이 채워진다 (SPEC §3 `08-to-be-프로세스.md`).
- **#05 수익성의 분모·분자**: ROI = 절감공수 ΔS × 처리량 N. ΔS는 본 문서의 단계별 소요시간 `s₁..s₇`(특히 최대 병목 `s₆`)과 To-Be `s′`의 차에서 나온다 (SPEC §3 `05-과제적정성-검토.md`).
- **#10 기대효과의 근거**: ⑥ 공수↓·처리량 상한 돌파라는 기대효과는 As-Is에서 ⑥이 실제로 최대 병목임을 보여야 설득력이 생긴다 (SPEC §3 `10-개발일정-수립.md`).
- **#09 제약의 출처**: 사용 시스템 컬럼(특히 CD-SEM 장비 GUI)이 곧 "장비 SW 버전 의존"이라는 시스템·인프라 제약의 근거가 된다 (SPEC §3 `09-ksf-제약-정의.md`).

## 핵심 산출 (What)

두 가지를 산출한다.

1. **As-Is 플로우차트** — Recipe Creation 셋업 ①~⑦ 전부 🧑 사람 수행. ⑥(시험측정 → 오인식 수정)에 **반복 루프 화살표**를 그려 최대 병목을 시각화한다.
2. **단계별 테이블** — `단계 | 설명 | 소요시간 | 사용 시스템`. 소요시간은 실측 전이므로 placeholder `s₁..s₇`(`s₆`=최대), 합계 `ΣS` 행을 둔다. 사용 시스템은 MIDAS(의뢰서) · CD-SEM 장비 GUI · Recipe DB로 일반화한다.

> 7단계 정의·병목 위치(③④⑥, 최대=⑥)·AI/사람 분담은 엔지니어 확정 사실 베이스(SPEC §2)를 그대로 받는다. As-Is 단계별 분담은 본 단계에서 **전부 🧑 사람**이다 — 🤖 대체는 #08에서 시작된다.

## 작성 결과 (How)

### (1) As-Is 플로우차트 (①~⑦, 전부 🧑 사람 수행)

병목은 ③④⑥, 최대 병목은 ⑥ — 시험측정(Try) 후 오인식/실패 지점을 찾아 수정하고 다시 측정하는 루프를 통과 기준에 도달할 때까지 반복한다 (출처: [원문 07](../../lectures/captures/07-ai-agent-scope-template.md) 레인 도구 / SPEC §2 / 04 인용).

```
  [입력] 공정·소자 엔지니어가 MIDAS에 의뢰서 작성 (현재 PPT 형태)
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ① 측정 대상 레이어/패턴 확인 (도면·스펙 검토)               │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ② 신규 Recipe 생성 → 웨이퍼/샘플 로드                       │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ③ Alignment/Addressing 포인트 등록 (패턴 인식)   ◀ 병목    │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ④ 측정 포인트(EP) 좌표·측정박스·알고리즘 설정    ◀ 병목    │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ⑤ 이미징 조건(배율·전압·FOV 등) 설정                       │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
        ┌───────────────────────────────────────────────────┐
        │ 🧑 ⑥ 시험측정(Try) → 오인식/실패 지점 수정          │ ◀ 최대 병목
        │     (a) 패턴 인식 실패  (b) 측정박스·좌표 오류      │
        └───────────────────────────────────────────────────┘
              │                                        ▲
              │ 통과 기준 미달 → 수정 후 재측정          │
              └────────────────[ 반복 루프 ]────────────┘
              │
              │ 통과
              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ⑦ Recipe 저장·등록 → 검증                                  │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    [Recipe DB 등록 완료]

  범례:  🧑 Human 수행 업무   ◀ 병목 구간   [ 반복 루프 ] ⑥ 시험측정↔수정
```

⑥의 반복 루프가 As-Is 소요시간을 지배한다 — 좌표 shift/align 실패와 패턴 오인식이 누적될수록 Try-수정 사이클이 늘어나고, 이것이 `s₆`가 최대인 이유다 (04 인용: "시험측정 후 오인식/실패 지점 수정 반복(⑥)").

### (2) 단계별 테이블 (소요시간·사용 시스템)

소요시간은 실측 전이므로 `s₁..s₇` placeholder다 — 캠프에서 엔지니어 실측으로 확정한다(SPEC §1.5). `s₆`는 ⑥ 반복 루프 때문에 단계 중 최대이며, 합계 `ΣS = s₁+s₂+s₃+s₄+s₅+s₆+s₇`가 1건당 As-Is 셋업 리드타임이다.

| 단계 | 설명 | 소요시간 | 사용 시스템 |
|------|------|----------|-------------|
| 🧑 ① 측정 대상 확인 | 도면·스펙 검토로 측정 레이어/패턴 파악. 의뢰서 정보 부족 시 협의 | `s₁` | MIDAS(의뢰서) |
| 🧑 ② Recipe 생성·로드 | 장비 GUI에서 신규 Recipe 생성, 웨이퍼/샘플 로드 | `s₂` | CD-SEM 장비 GUI |
| 🧑 ③ 패턴 등록 *(병목)* | Alignment/Addressing 포인트(패턴 인식 템플릿) 수동 등록 | `s₃` | CD-SEM 장비 GUI |
| 🧑 ④ 좌표·박스 설정 *(병목)* | 측정 포인트(EP) 좌표·측정박스·측정 알고리즘 수동 설정 | `s₄` | CD-SEM 장비 GUI |
| 🧑 ⑤ 이미징 조건 설정 | 배율·전압·FOV 등 설정. reference 없으면 협의 | `s₅` | CD-SEM 장비 GUI / Recipe DB(ref 조회) |
| 🧑 ⑥ 시험측정·오인식 수정 *(최대 병목)* | 시험측정(Try) → (a)패턴 실패·(b)좌표 오류 지점 수정 → 재측정 **반복 루프** | `s₆` *(최대)* | CD-SEM 장비 GUI |
| 🧑 ⑦ 저장·검증 | Recipe 저장·등록 → 검증. 검증 누락 시 후공정 전파 위험 | `s₇` | CD-SEM 장비 GUI / Recipe DB(등록) |
| **합계** | **As-Is 셋업 리드타임 (1건당)** | **`ΣS`** | — |

> 시스템명은 일반화했다 — 구체 장비 SW 명칭·버전은 회사 기밀이라 제외하고 "CD-SEM 장비 GUI"로 통칭한다(SPEC §1.6). MIDAS는 기존 문서(03/04)에 등장하므로 의뢰서 입력 시스템으로 표기한다. MIDAS 전산화 자체는 외부 변수(별도 프로젝트)이며 본 과제는 부족정보 플래깅 선에서만 연계한다(SPEC §2).

**백데이터 연결**: 이 테이블의 소요시간 컬럼은 (1) #08 To-Be 대비 테이블의 before 값, (2) #05 수익성 ΔS = ΣS − ΣS′의 As-Is 항, (3) #10 기대효과(⑥ 공수↓·처리량 상한 돌파)의 정량 근거로 그대로 흐른다. 사용 시스템 컬럼은 #09 시스템·인프라 제약(장비 SW 버전 의존)의 출처가 된다 (SPEC §3 `07-as-is-프로세스.md`).
