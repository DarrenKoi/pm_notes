# 09 · 핵심 성공 요소(KSF)·제약 정의 (Step11 / Design)

> To-Be 프로세스(#08)가 의도한 성과를 내려면 **반드시 선결되어야 할 조건**을, 4관점(업무 프로세스·데이터·시스템/인프라·조직 문화)에서 「핵심 성공요소 × 장애 요인 × 선결 조건 × 적용 기술」 매트릭스로 박제한다.

## 왜 필요한가(Why)

To-Be 흐름(②~⑥ VLM 연속수행 + ⑦ 사람 최종승인)은 **그 자체로 성립하지 않는다.** 좌표 재정합이 부정확하면 ⑥ 자동화가 무너지고, GUI 자동제어가 불안정하면 ②~⑤가 멈추며, 엔지니어가 신뢰하지 않으면 현장에 안 깔린다. KSF 정의는 "무엇이 충족돼야 이 설계가 작동하는가"를 **착수 전에** 못박아, #10 개발일정의 게이트(🚦)와 리스크 관리의 출처가 되게 한다.

KSF는 **반드시 선결해야 할 항목 위주로** 작성한다 — To-Be 프로세스가 의도한 성과를 달성하기 위해 반드시 충족돼야 하는 핵심 성공 조건이다. 제약 사항은 설계·추진 시 기술·시스템·조직·비용·정책 측면에서 넘을 수 없는 조건과 한계다. (출처: [원문 10](../../lectures/captures/10-check-and-review.md))

> 본 단계는 **장표(slide) 한 장**으로 읽혀야 한다. 4×4 매트릭스 + 선결 우선순위가 핵심이다.

## 핵심 산출(What)

- **KSF 매트릭스**: 4관점(rows) × 4컬럼(핵심 성공요소 · 장애 요인 · 선결 조건 · 적용 기술).
- **관점 축**: 업무 프로세스 / 데이터 / 시스템·인프라 / 조직 문화. (브레인스토밍 합의(2026-06-24) — 04의 기술/조직/비용/정책 축을 실행 관점으로 재배열)
- **선결 우선순위 리스트**: #1 → #4 순서로, 무엇을 먼저 검증·확보할지 명시.
- **연계 보장**:
  - 조직 문화의 선결(⑦ 사람 최종승인 유지)은 #08의 🧑 레인과 일치한다.
  - 장애 요인 컬럼은 #10 개발일정의 리스크 게이트(🚦) 입력이 된다.
  - 선결 #1(좌표 재정합 정확도)은 Phase0 PoC(04 Step12 / 05 실험설계)와 동일 대상이다.

## 작성 결과(How)

### KSF 매트릭스 (4관점 × 4컬럼)

| 관점 | 핵심 성공요소(KSF) | 장애 요인 | 선결 조건 | 적용 기술 |
|------|-------------------|-----------|-----------|-----------|
| **업무 프로세스** | ②~⑥를 **연속 수행**하고, 예외 발생 시 **핸드오프 기준**대로 사람에게 넘긴다 | ⑥ 좌표 재정합 잔존 오류 · "어디서부터 사람에게 넘길지" 예외 기준 모호 | 신뢰도 임계값 τ·재시도 횟수 r 정의 + ⑦ 최종승인 게이트 확정 | 신뢰도 룰엔진 · 핸드오프 오케스트레이션 |
| **데이터** | SEM 이미지·참조(ref) recipe·라벨이 **충분히 확보**됨 | 배율·노이즈별 이미지 편차 · 라벨링 공수 · 사내 문서 99% **DRM** | 숙련 엔지니어 라벨 확보 + 스크린샷→VLM 추출 파이프라인 구축 | Qwen3-VL 추출 · BGE-M3 검색 · 라벨링 툴 |
| **시스템·인프라** | GUI **자동제어가 안정**적이고 VLM 추론 자원이 확보됨 | 장비 SW 버전 의존성 · GPU 자원 한계 · **외부 LLM API 전면 차단** | 사내 Qwen3-VL 엔드포인트 · GPU 서버 · 섀도우(오프라인) 환경 · 안전 인터록 | Qwen3-VL-8B/30B · computer-use · GPU 서버 |
| **조직 문화** | 엔지니어가 **신뢰**하고 현장이 **수용**한다 | AI의 장비 직접 제어에 대한 거부감 · 검증 권한 귀속 이슈 | ⑦ **사람 최종승인 유지** · 운영팀 협업 · 단계적 도입 | 설명가능(explainable) 리포트 · HITL 승인 UI |

(출처: 04 Step11 인용 + 브레인스토밍 합의(2026-06-24) — 4관점으로 재배열. 사내 LLM 제약·DRM 파이프라인은 [원문 10](../../lectures/captures/10-check-and-review.md)의 "제약 다각도 검토" 적용)

> **정합성 박제**
> - 조직 문화 선결 「⑦ 사람 최종승인 유지」 = #08 To-Be의 🧑 레인(⑦ 사람 최종승인 + 예외 핸드오프)과 **동일**. (불일치 시 설계 오류)
> - 장애 요인 컬럼(⑥ 잔존 오류 / 버전 의존 / DRM / 거부감) = #10 개발일정의 **리스크 게이트(🚦)** 입력.
> - 적용 기술의 LLM은 전부 **사내 OpenAI-호환 엔드포인트**(Qwen3-VL-8B/30B·BGE-M3). 외부 API 사용 없음.

> ⚠️ **안전·보안·감사 보강 (장비 직접 제어 과제이므로 필수)**
> 본 과제는 VLM이 GUI로 **장비를 직접 제어**하므로, 일반 KSF에 더해 다음을 시스템·인프라/정책·보안 관점의 선결로 못 박는다.
> - **안전 인터록·롤백**: 실 웨이퍼·라인에 영향을 주는 동작은 인터록으로 사전 차단하고, 잘못된 제어는 즉시 롤백한다(섀도우/오프라인 우선 검증).
> - **사고 책임소재**: AI 오작동 시 책임자·대응 절차를 사전 지정한다(= `<담당 임원>`·운영팀, #10 RACI와 연결).
> - **보안 승인·감사 추적**: DRM 문서가 스크린샷+VLM 파이프라인을 거치는 경로의 데이터 유출 점검 + 사내 엔드포인트 격리·접근로그를 **보안팀이 승인**한다. AI 판정은 모델 버전과 함께 **감사 로그**로 남겨 재현 가능해야 한다.

### 선결 우선순위 (먼저 검증·확보할 순서)

```
#1  좌표 재정합 정확도              ← 최대 리스크. ⑥ 자동화의 성립 여부 자체.
    = Phase0 PoC 검증 대상           (04 Step12 / 05 실험설계)
    └ 미달 시 과제 재설계 게이트
        ▼
#2  GUI 자동제어(computer-use) 안정 ← ②~⑤가 멈추지 않게. 장비 SW 버전 의존성 관리.
        ▼
#3  핸드오프 기준(예외 임계값)      ← "어느 정도 오인식이면 사람에게 넘기는가" τ/r 정의.
        ▼
#4  신뢰·수용(현장 도입)            ← 엔지니어 신뢰 + 운영팀 협업 + 단계적 도입.
```

- **#1**은 다른 모든 선결의 전제다 — 재정합이 안 되면 #2~#4를 채워도 ⑥ 자동화가 성립하지 않는다. 그래서 Phase0에서 **실 장비 영향 0**의 섀도우 환경으로 단독 검증한다. (출처: 04 Step12 / [05 PoC 실험설계](../05-적용_CDSEM_PoC실험설계.md))
- 임계값 τ(재정합 정확도)·r(재시도 횟수)·s/s′(소요시간)·N(처리량)은 **캠프에서 엔지니어가 확정**할 placeholder다. 본 문서는 변수와 검증 순서만 고정한다.
