# AIX 적용 사례 — CD-SEM Recipe Creation 자동화 (기술/To-Be 설계)

> [기술 방법론 틀(02)](./02-기술문서_AI과제정의구현.md)을 ITC AIX 실제 과제에 적용한 **두 번째 사례**. [03 기획(Discovery)](./03-적용_CDSEM기획.md)에서 발굴·구조화한 **CD-SEM Recipe Creation 자동화** 과제를, New AI Design Camp Track B(실행 구체화, 8~12단계) — AI 과제 정의서 · To-Be Swimlane · KSF/제약 — 로 구체화한다.

## 왜 필요한가? (Why)

03에서 "⑥ 오인식 수정 반복을 VLM으로 자동화한다"는 방향이 정해졌다. 본 문서는 그 방향을 **실제 구현 가능한 형태**로 고정하며, 02가 강제하는 두 가지를 CD-SEM 맥락에서 못 박는다.

- **역할 경계의 명시화** — VLM이 GUI를 직접 조작·판독하는 구간(②~⑥)과, 엔지니어가 최종 검증·승인·예외 처리하는 구간(①·⑦·예외)을 Task 단위로 분리한다. **완전자동화가 아니라 Human-in-the-loop**임을 설계로 박제한다.
- **제약·성공조건의 사전 검증** — 03에서 ⚠️로 남긴 "정밀 좌표 재정합 정확도" 리스크를 포함해, 기술·조직·비용·정책·보안 제약과 KSF를 구현 착수 전에 확정한다.

## 핵심 개념 (What) — To-Be 5대 용어의 CD-SEM 매핑

02의 표준 용어([원문 10](./source/10-check-and-review.md))를 본 과제에 1:1로 대입하면:

| 구분 | CD-SEM 과제에서의 의미 |
|------|------------------------|
| **To-Be 프로세스** | VLM이 GUI를 직접 제어해 Recipe를 생성하고, 실측 좌표/align을 실시간 재정합하는 셋업 흐름 (As-Is ①~⑦의 재정의) |
| **Activity / Task** | As-Is의 ①~⑦ 각 단계. 그중 ②~⑥이 AI 수행으로 재정의되는 실행 단위 |
| **AI Agent 역할** | 특정 Task에서 SEM 화면을 판독해 GUI를 클릭·입력하고, 좌표 재정합·오인식 수정을 수행하는 주체 (사내 Qwen3-VL 기반) |
| **제약 사항** | 외부 LLM 차단, 좌표 재정합 정확도, 장비 직접 제어 안전성, MIDAS 외부 변수 등 (How §Step 11) |
| **KSF** | VLM 좌표 재정합 정확도가 엔지니어 수용 기준 이상 + 예외 핸드오프 기준 명확화 (How §Step 11) |

## 어떻게 사용하는가? (How)

```
┌──────────────────────────────────────────────────────────────────────┐
│ 과제명 : VLM GUI-제어 기반 CD-SEM Recipe Creation 자동화               │
│ 구분   : ☐ Marketing  ☐ R&D  ■ Production  ☐ Support                   │
│ Champion(담당 임원) : <담당 임원>   담당 조직 : ITC AIX / MI            │
├──────────────────────────────────────────────────────────────────────┤

│  • 업무 목표(부문/팀) : 신규 공정 셋업 리드타임 단축 · 24h 무중단 셋업  │
│                         · 숙련도 편차 제거                              │

│                         shift / align 실패 발생                        │

│     ─ 적용 범위 : 신규 Recipe 생성. reference recipe가 있는 layer 우선, │
│                  점진 확장 (신규 공정은 엔지니어 협의 유지)             │

│                  · 숙련도 무관 일관 기준 수행                           │
```

### Step 9~10. As-Is → To-Be 프로세스 (Swimlane)

각 단계(Task) 박스를 **🤖 AI Agent 수행 / 🧑 Human 수행** 으로 구분([원문 07](./source/07-ai-agent-scope-template.md)). 가로축 = As-Is ①~⑦ 흐름.

```
 ①        ②         ③         ④         ⑤         ⑥          ⑦
의뢰서   Recipe     패턴      좌표·박스   이미징     오인식      저장·
검토     생성·로드   등록      ·알고리즘   조건       수정(반복)   검증·승인
🧑       🤖         🤖        🤖         🤖         🤖          🧑
(필요시)            ▲병목     ▲병목      (ref참고)   ▲최대병목   (최종)
                                                       │
                              ── 예외(임계값 초과) ──▶ 🧑 엔지니어 핸드오프
범례:  🤖 AI Agent 수행 업무   🧑 Human 수행 업무
```

| 단계 | As-Is (사람) | To-Be (재정의) | 수행 |
|------|--------------|----------------|------|
| ① 의뢰서 검토 | 엔지니어가 도면·스펙 검토 | 의뢰서 부족정보 자동 플래깅, 필요 시 사람 검토 | 🧑 (필요시) |
| ② Recipe 생성·로드 | GUI 수동 생성, 샘플 로드 | VLM이 GUI 조작해 신규 Recipe 생성·로드 | 🤖 |
| ③ 패턴 등록 *(병목)* | Align/Addressing 템플릿 수동 지정 | VLM이 SEM 화면 판독해 패턴 등록 | 🤖 |
| ④ 좌표·박스 *(병목)* | EP 좌표·측정박스·알고리즘 수동 설정 | VLM이 좌표·박스 자동 설정 | 🤖 |
| ⑤ 이미징 조건 | reference 참고 또는 협의 | reference 동일 시 VLM 자동 적용 | 🤖 |
| ⑥ 오인식 수정 *(최대 병목)* | 시험측정 → 실패 지점 수동 반복 수정 | **VLM이 실측 이미지 기반 좌표 재정합·재align·오인식 자동 수정** | 🤖 |
| ⑦ 저장·검증 | 저장·등록·검증 | 엔지니어 최종 검증·승인 | 🧑 |

작성 원칙대로, **최대 병목 ⑥에 AI를 배치**해 공수를 줄이고, 임계값을 넘는 예외는 사람에게 핸드오프한다.

### Step 11. KSF & 제약 사항 정의

#### KSF (선결 성공 조건 — 우선순위 순)

1. **VLM 좌표 재정합 정확도가 엔지니어 수용 기준 이상** — 03에서 ⚠️로 지목한 최대 리스크. 이게 안 되면 ⑥ 자동화 자체가 성립 안 함. **PoC 1순위 검증 대상.**
2. **GUI 제어(computer-use)의 안정적 장비 SW 연동** — 클릭·입력 신뢰성, 장비 SW 버전 의존성 관리.
3. **예외 핸드오프 기준의 명확화** — "어느 정도 오인식이면 사람에게 넘기는가"의 임계값 정의(신뢰도/재시도 횟수).
4. **의뢰서 입력의 최소 표준화** — 부족정보 플래깅이 동작하려면 입력 포맷 합의 필요 (MIDAS 전산화와 연계, 단 외부 변수).

#### 제약 사항 (4측면 — 02의 다각도 검토 적용)

| 측면 | CD-SEM 과제의 실제 제약 |
|------|--------------------------|
| **기술** | 외부 LLM API 전면 차단 → 사내 **Qwen3-VL-8B/30B**(비전)만 사용 / 정밀 좌표 재정합 정확도 / SEM 이미지 배율·노이즈별 일반화 / GUI 자동제어 안정성(장비 SW 버전 의존) / DB(OpenSearch·ES) 로컬 미가용 → 검증은 import·문법 수준 |
| **조직** | 장비 운영팀 협업 필수 / 엔지니어 신뢰 확보(⑦ 검증은 사람 유지) / 숙련 엔지니어의 라벨·피드백 확보 |
| **비용** | 사내 VLM 추론용 GPU 자원 / 장비 다운타임 없이 검증할 오프라인·섀도우 환경 구축 |
| **정책·보안** | 장비 직접 제어 권한 및 안전 인터록(실 웨이퍼 영향 방지) / 사내 문서 99% DRM → 스크린샷+VLM 추출 파이프라인 / 개인정보·내부 규정 준수 |

### Step 12. 개발 일정 수립 (Action Planning)

KSF 우선순위를 단계로 환산(구체 일자는 별도 계획에서 확정):

```
Phase 0  좌표 재정합 정확도 PoC (KSF#1)  — 섀도우 환경, 실 장비 영향 0  → 실험 설계: 05
   ▼     판단: 수용 기준 미달 시 과제 재설계 / 충족 시 진행
Phase 1  단일 layer Recipe 자동 생성 (②~⑤) + ⑥ 부분 자동화
   ▼
Phase 2  ⑥ 오인식 수정 본격 자동화 + 예외 핸드오프(KSF#3) 도입
   ▼
Phase 3  layer 확장 · 운영 이관 · 의뢰서 표준화 연계(KSF#4)
```

## 설계 검증 — Check & Review (CD-SEM 적용)

02의 3대 점검([원문 10](./source/10-check-and-review.md))을 본 과제에 적용:

### ✅ 1. AI·사람 역할의 명확한 구분
🤖 AI: ②~⑥ GUI 직접 조작 + SEM 이미지 판독 기반 좌표/align 재정합 + 오인식 수정.
🧑 Human: ① 의뢰서 검토(필요시) · 임계값 초과 예외 핸드오프 · ⑦ 최종 검증·승인.
> Task별 경계가 Swimlane 표에 1:1로 명시됨 → 통과.

### ✅ 2. 시간이 가장 많이 걸리는 구간 개선 여부
As-Is 최대 병목 = ⑥ 오인식 수정 반복. To-Be에서 ⑥을 VLM 실시간 재정합으로 대체하고 사람은 예외만 처리.
> 최대 병목에 AI를 정조준 배치 → 통과.

### ✅ 3. 제약 사항 다각도 검토 여부
기술뿐 아니라 조직(운영팀 협업·신뢰)·비용(GPU·섀도우 환경)·정책·보안(장비 제어 권한·DRM) 모두 작성.
> 4측면 작성 완료 → 통과.

## 구현 매핑 — AI Agent 배치 지점

03의 L1~L5 분해에서 도출된 AI 배치 Task:

| 구분 | 내용 |
|------|------|
| L1 Mega Process | 계측(MI) 운영 |
| L2 Process Group | CD-SEM Recipe Management |
| L3 Process | Recipe Creation |
| L4 Activity | As-Is ①~⑦ |
| **L5 Task (AI 배치)** | **③ 패턴 등록 · ④ 좌표·박스 설정 · ⑥ 오인식 수정** (핵심: ⑥) |

> 영업 예시가 프로세스 그룹별 4개 Agent로 나뉜 것([원문 06](./source/06-sales-ai-opportunity.md))과 달리, 본 과제는 **단일 Recipe Creation 흐름 내 연속 Task(②~⑥)를 하나의 VLM Agent가 GUI 제어로 수행**하는 구조다.
