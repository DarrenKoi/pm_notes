# 11 · Review — 산출물 검산 · 패키징 · 발표 (템플릿)

> 한 줄: 앞선 10단계 산출물을 **새로 쓰지 않고 검산**하여, 6개 점검 질문에 답이 떨어지는지(✅/△/❌) 판정하고 묶음을 패키징·발표·피드백 루프로 마무리한다.

## 이 단계가 무엇인가 (Why)

Review는 **정합성 검산** 단계다. 새 내용을 생성하지 않는다 — 이미 만든 산출물(01~10)이 서로 모순 없이 한 줄(전략→Pain→근본원인→과제 정의→To-Be→KSF→일정)로 꿰어지는지 확인한다. (출처: New AI Design Camp Check & Review, [원문 10](../source/10-check-and-review.md); [02 기술문서 설계 검증](../02-기술문서_AI과제정의구현.md))

핵심 원칙 — **이 문서는 결함 탐지기다.** ✅가 떨어지지 않으면 그 질문이 가리키는 출처 산출물이 부실하다는 신호이고, △/❌ 칸은 해당 단계 파일로 돌아가 보강하라는 **작업지시서**가 된다. Review에서 무언가를 "고치는" 것이 아니라 **어디를 고쳐야 하는지 가리킨다.**

## 작성할 산출물 (What)

두 부분으로 구성된다: **A. 설계 검증 체크리스트**(6질문) + **B. 마무리**(패키징·발표·피드백).

### A. 설계 검증 체크리스트 — 6질문 (질문 고정, 판정·근거 빈칸)

> 원문 10의 3대 점검(역할 구분 / 병목 개선 / 제약 다각도)에 본 키트 고유 점검 3개(KSF 명확 / 전략 추적성 / 가치 정량)를 더한 6개. **질문은 바꾸지 말고, 판정과 근거만 채운다.**

| # | 점검 질문 (고정) | 판정 ✅/△/❌ | 근거 — 어느 산출물이 답하나 |
|---|------------------|:-----------:|------------------------------|
| 1 | AI·사람 역할이 Task별로 명확히 구분되는가? | `___` | `<08 To-Be 🤖/🧑 레인 + 02 AI 대체 판정>` |
| 2 | 시간이 가장 많이 걸리는 병목 구간을 개선하는가? | `___` | `<03 Pain 정량화(최대 병목) → 07 As-Is 소요시간 → 08 To-Be 대체>` |
| 3 | 제약을 다각도로 검토하는가(조직 인력·역량·비용·보안 정책)? | `___` | `<09 KSF·제약 4관점>` |
| 4 | KSF가 명확히(선결 조건 위주로) 작성되었는가? | `___` | `<09 선결 우선순위 #1~ + 05 준비계획 구체성>` |
| 5 | 과제가 전략에서 연역됐는가(추적성)? | `___` | `<01 Top-Down 연계표(출발) ↔ 06 과제 정의서 각 필드(도착)>` |
| 6 | 가치가 정량화됐는가? | `___` | `<05 수익성·ROI(ΔS×N)>` |

> **판정 가이드**: ✅ = 그 출처 파일이 질문에 답을 낸다 / △ = 구조는 잡혔으나 값 미확정(placeholder) / ❌ = 출처 파일이 질문에 답하지 못함(→ 해당 단계로 회귀해 보강). 실측 수치가 아직 placeholder(τ=임계·점수, s=소요시간, r=재시도, N=처리량)인 항목은 △가 정상이며, "캠프/워크숍 후 확정"이라는 후속 작업으로 넘긴다(아래 B-3).

#### 교차 정합성 스팟체크 (모순 탐지 · 선택) — 파일 간 값이 어긋나지 않는지(불일치=설계 오류)

| 검산 항목 | 대조 대상 | 기대 |
|-----------|-----------|------|
| AI 수행 범위 | `<06 적용범위 ↔ 08 🤖 레인>` | 동일 |
| 사람 최종 승인 단계 | `<08 🧑 레인 ↔ 09 조직문화 선결>` | 동일 |
| 운영 기준 임계값(θ/τ/r) | `<08 운영기준 ↔ 09 선결 ↔ 10 Test 프로토콜>` | 같은 변수명·정의 |
| KSF 4관점 ↔ 일정 테마 레인 | `<09 4관점 ↔ 10 칸반 행>` | 1:1 정렬 |
| placeholder 표기 통일 | `<전 파일>` | 지어낸 수치 없음 |

### B. 마무리 체크 (빈칸)

#### B-1. 산출물 패키징 — 00-README가 10개를 한 묶음으로 인덱싱

- [ ] 00-README 인덱스 표가 실제 산출물 파일·단계와 **빠짐없이 일치**하는가? (누락=패키징 결함)
- [ ] 4-Step(Discovery/Design/Validation/Execution)↔단계 매핑이 빠짐없는가?
- [ ] 읽는 순서·핵심 흐름 다이어그램이 들어 있는가?

#### B-2. 발표 — 산출물을 그대로 장표 본문으로

- 발표 대상(audience): `___` ← *누가 듣나(스폰서 임원/현업/심사위원)?*
- 핵심 메시지(closing): `___` ← *A 체크리스트 6질문 통과 = "설계가 점검을 통과했다"를 클로징으로*
- [ ] 각 산출물(표/플로우/칸반)을 1산출물=1~2장으로 변환했는가?
- [ ] 표·플로우차트를 마크다운/ASCII로 작성해 **변환 호환**(PPTX 등)을 확보했는가?

#### B-3. 피드백 반영 — placeholder 확정 → 문서 업데이트 루프

> 워크숍 후 피드백의 핵심은 **placeholder 확정**이다. 이것이 △를 ✅로 올리는 작업이며 Review가 남기는 후속 작업 목록이다.

| 확정 대상 | placeholder | 확정 주체 | 반영 위치 |
|-----------|-------------|-----------|-----------|
| 스폰서/담당 임원 | `<담당 임원>` | 조직 | `<06 담당임원 필드>` |
| 신뢰도·정확도·임계 | τ, θ | 엔지니어(캠프) | `<08 운영기준 · 09 선결 · 10 Test>` |
| As-Is/To-Be 소요시간 | s, s′ | 실측(Phase 0~1) | `<07 · 08 · 05 ROI>` |
| 재시도 임계·처리량 | r, N | 엔지니어·파일럿 | `<08 · 05 수익성>` |

업데이트 루프: **피드백 → placeholder 값 입력 → 해당 단계 파일 수정 → 6번 판정 재검산 → 00-README last_updated 갱신.** 값이 채워지면 05 ROI·10 기대효과가 가설에서 실측 근거로 전환되고, A의 6번이 △→✅로 바뀐다.

## 퍼실리테이션 가이드 (How to run)

**준비** → **검산(발산)** → **모순 점검(수렴)** → **마무리 정리**. 타임박스 권장 60~90분.

1. **준비**(10분): 01~10 산출물을 한 화면에 띄운다. Review는 새로 쓰는 자리가 아니라 **검산하는 자리**임을 먼저 공유한다.
2. **검산**(30분): 6질문을 하나씩 읽고, 각 질문에 답하는 산출물을 가리키며 ✅/△/❌를 합의한다.
3. **모순 점검**(15분): 교차 정합성 스팟체크로 파일 간 값 충돌을 잡는다.
4. **마무리 정리**(15~30분): 발표 대상·핵심 메시지 확정, placeholder 후속 작업 목록 작성.

참가자에게 던질 질문(이 단계 고유):
- "각 점검 질문에 우리 산출물이 **실제로** 답하나? 어느 파일의 어느 칸이?"
- "✅가 안 떨어지는 항목의 **출처 단계**는 어디인가? 거기로 돌아가 무엇을 보강해야 하나?"
- "파일 사이에 값이 서로 어긋나는 곳은 없나(같은 변수가 다른 정의)?"
- "발표 **대상**은 누구이고, 그들에게 **핵심 메시지** 한 문장은?"
- "아직 placeholder인 값은 누가, 언제 확정하나? 어떻게 **반영 루프**로 돌리나?"

**완료 정의**: 6질문 모두 판정·근거가 채워졌고(△는 후속 작업으로 명시), 교차 정합성에서 모순이 없으며, 00-README 인덱스가 실제 파일과 일치하고, 발표 대상·메시지와 placeholder 확정 목록이 적혔다.

## 흔한 실수 · 체크포인트

- **✅ 도장만 찍고 근거 출처가 없다** — 판정 옆 "근거" 칸을 **반드시 산출물 링크/위치**로 채운다. 출처 없는 ✅는 검산이 아니다.
- **Review에서 내용을 새로 쓴다** — Review는 검산이지 생성이 아니다. 부족하면 **해당 단계 파일로 돌아가** 고치고, Review는 "어디를 고칠지"만 가리킨다.
- **△를 결함으로 오해한다** — 실측 전 placeholder(τ/s/r/N)는 △가 정상. 단, **누가·언제 확정할지** B-3에 후속 작업으로 반드시 남긴다.
- **피드백 루프가 없다** — 발표하고 끝내지 않는다. placeholder 확정 → 파일 수정 → 재검산 → 인덱스 갱신의 루프가 닫혔는지 다음 라운드에서 다시 본다.
- **00-README 인덱스 누락** — 인덱스가 실제 파일/단계와 1:1로 맞는지 최종 확인(누락=패키징 결함).

**통과 기준**: 6질문이 모두 근거 링크와 함께 판정됐고, △ 항목은 후속 작업으로 추적되며, 패키징·발표·피드백 루프 3종이 모두 채워졌다.
