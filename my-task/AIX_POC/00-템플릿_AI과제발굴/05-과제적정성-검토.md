# 05 · 과제 적정성 검토 — 템플릿

> 한 줄: 발굴한 후보 과제를 **7문항 × 1~3점 스코어카드**로 점수화하고, 종합 Σ/21과 착수 임계(threshold)로 "지금 실행할 가치가 있는가"를 게이트 통과시킨다.

## 이 단계가 무엇인가 (Why)

앞 단계(01~04)는 후보 과제가 **조직목표에서 연역됐고(01) · 대상 업무가 명확하며(02) · Pain이 정량화됐고(03) · 근본원인까지 파고들었음(04)** 을 보였다. 그러나 **발굴됐다 ≠ 착수해도 좋다.** 자원이 한정된 이상, 후보를 객관적 잣대로 점수화해 게이트를 통과시켜야 한다.

- 과제 선정 3-STEP의 STEP 3은 **"고통의 정도(빈도·심각성·시급성/중요성) · 구현 난이도 · 비용 효용성을 고려해 최종 AI 과제를 선정한다"** 고 못박는다 (출처: [원문 01](../source/01-kakao-3step-pain-point.md)). 이 단계가 바로 그 정량 게이트다. 핵심: **아프고(고통 큼) · 만들 수 있고(난이도 OK) · 남는 게 있는(효용 큼)** 교집합만 과제로 선정한다 (출처: [01 기획문서](../01-기획문서_AX서비스기획.md) Phase 1).
- 방법론 4 Step 중 **Step 3 Validation(가치 검증)** = "투자 효용성 및 재무적 효과 산출 / 비즈니스 임팩트 정량화 / 재무적 타당성 산출" (출처: [원문 02](../source/02-ax-consulting-methodology.md)). 본 단계의 **'비즈니스 성과' 카테고리(수익성·확장성·시급성)** 가 이 Validation을 흡수한다 — 별도 Validation 문서를 두지 않고 적정성 스코어카드 안에 ROI 검증을 박는다.
- 점수는 여기서 새로 지어내지 않는다. 03의 Pain 정량(빈도·심각성·주요성)과 04의 근본원인이 본 스코어카드의 **백데이터**다 → 앞 단계 산출물에서 **연역**한다(추적성).

## 작성할 산출물 (What)

### A. 스코어카드 — 3카테고리 × 7문항 (각 1~3점, 종합 Σ/21)

> 점수 척도 가이드: **1점 = 미흡/불확실**, **2점 = 보통**, **3점 = 우수/확실**. 각 칸은 빈칸으로 두고, 워크숍에서 참가자 합의로 채운다. 근거 칸에는 **앞 단계(01~04) 산출물의 어느 부분에서 연역됐는지**를 반드시 적는다(점수만 적고 근거를 비우지 말 것).

#### A-1. 전략적 중요도 (Σ_A / 6)

| 문항 | 점수(1~3) | 근거 (← 어느 앞 단계에서 연역?) |
|------|:---:|------|
| **1. 본원적 경쟁력** — 조직의 본원적 활동(핵심 가치사슬)과 연계되나? 주변부 효율화가 아닌가? | ___ | ___ (← 02 대상업무영역 / 03 painpoint 주요성) |
| **2. 사업전략·목표·KPI 연계성** — 조직목표→KPI에 1:1로 매핑되나? | ___ | ___ (← [01 전략일관성](./01-전략일관성-확인.md) 연계표) |

#### A-2. 비즈니스 성과 (Σ_B / 9) — **Validation/가치검증 흡수 지점**

| 문항 | 점수(1~3) | 근거 (← 어느 앞 단계 / ROI 산식) |
|------|:---:|------|
| **3. 수익성** — ROI = 절감공수 ΔS × 처리량 N 이 충분한가? | ___ | ___ (← 03 Pain 정량 + 아래 'ROI 정량화' 블록) |
| **4. 확장성** — 다른 영역·조직·라인으로 수평 확장 가능한가? (확장 = N을 키우는 ROI 레버) | ___ | ___ (← 02 대상업무영역 범위) |
| **5. 시급성** — 지금 안 하면 비용/리스크가 커지나? (시급성 = 03 Pain 척도와 연동) | ___ | ___ (← 03 painpoint 시급성·심각성) |

#### A-3. 실행 용이성 (Σ_C / 6)

| 문항 | 점수(1~3) | 근거 (← 데이터/계획 준비도) |
|------|:---:|------|
| **6. 데이터 준비** — 필요한 데이터가 이미 가용한가? (추측 설계가 아닌가?) | ___ | ___ (← 데이터 존재/접근성 확인) |
| **7. 준비계획 구체성** — 착수 계획·검증 프로토콜(PoC 실험설계 등)이 구체적인가? | ___ | ___ (← PoC/실험설계 존재 여부) |

### B. 종합 점수 (Σ / 21) + 착수 임계 설정란

```
Σ = Σ_A + Σ_B + Σ_C
  = ( ___ + ___ )  +  ( ___ + ___ + ___ )  +  ( ___ + ___ )
    └ 전략 /6 ┘       └ 비즈니스 성과 /9 ┘     └ 실행 /6 ┘
  = ___ / 21
```

**착수 임계(threshold) 설정란** — 워크숍에서 합의해 채운다:

| 항목 | 값 | 결정 근거 |
|------|:---:|------|
| 착수 임계 **τ_gate** (예: 14~15 / 21) | ___ / 21 | ___ (조직의 자원·리스크 허용치에 맞춰 합의) |
| 종합 Σ | ___ / 21 | (위 표 합산) |
| 판정 | **착수 / 보류 / 재설계** | Σ ≥ τ_gate → **착수**(다음 단계 06으로). 미달 → 보류 또는 재설계 |
| 게이트 승인 주체 | `<담당 임원/Champion>` | (착수 승인 책임자) |

> 이 종합점수가 마지막 단계(개발일정 수립)의 **착수 게이트 정량 근거**가 된다. 점수표는 "왜 이 과제를 지금 한다"의 객관 증빙으로 보관한다.

### C. 가치·ROI 정량화 블록 (Validation 흡수)

본 단계가 4 Step의 **Step 3 Validation(가치 검증)** 을 흡수한다 (출처: [원문 02](../source/02-ax-consulting-methodology.md) Step3 "투자 효용성 및 재무적 효과 산출"). 위 스코어카드의 **3.수익성 · 4.확장성 칸이 곧 ROI 검증**이다 — 별도 Validation 문서 없이 적정성 게이트 안에 재무 타당성을 박는다.

**ROI 산식 (변수만 정의 — 실제 수치는 앞으로 채움):**

```
연간 절감 가치  ΔValue = ΔS × N
  ΔS = s − s′          (작업 1건당 공수 절감)
       s  = As-Is 1건 소요시간   → 07 As-Is 프로세스에서 실측 (s = ___)
       s′ = To-Be 1건 소요시간   → 08 To-Be 프로세스에서 실측 (s′ = ___)
  N  = 연간 처리량 (작업 건수)    → 확장 범위에 비례 (N = ___)

ROI ≈ ΔValue / (구현·운영 투자)   → 투자측은 10 개발일정의 인력·기간에서 산정
```

**변수 의미(placeholder — 빈칸으로 둔다):**

| 기호 | 의미 | 어디서 확정? |
|------|------|------|
| **s** | As-Is(현행) 1건 소요시간 | 07 As-Is 프로세스 실측 |
| **s′** | To-Be(개선) 1건 소요시간 | 08 To-Be 프로세스 추정 |
| **ΔS** | 1건당 절감 공수 (s − s′) | 위 둘에서 계산 |
| **N** | 연간 처리량(건수) | 대상 범위·확장 범위 |
| **τ** | 스코어카드 점수/임계 (1~3점, τ_gate) | 워크숍 합의 |

> **수치 금지 원칙**: s·s′·N·τ는 전부 빈칸이다. 실측은 07/08, 검증 프로토콜은 PoC 실험설계, 투자측은 10에서 확정한다. 본 단계는 **산식과 게이트 구조만 박제**하고 빈칸은 워크숍/엔지니어 확정 몫으로 남긴다. **확장성 = ROI 레버**: 확장 범위가 N을 키워 ΔValue를 증폭하므로, 4번 확장성은 수익성(3번)과 곱셈 관계로 묶인다.

## 퍼실리테이션 가이드 (How to run)

**타임박스 권장: 60~90분** (준비 10 / 발산·채점 40 / 수렴·임계합의 20 / 정리 10).

1. **준비** — 01~04 산출물(연계표·L분해·Pain 정량·근본원인)을 벽에 붙여 둔다. 스코어카드 7행 빈 표를 띄운다.
2. **발산(문항별 채점)** — 문항을 하나씩 던지고, 참가자가 1~3점을 **각자 먼저** 적게 한 뒤 공개한다(앵커링 방지). 점수가 갈리면 근거를 말하게 해 합의한다. 각 문항에 대해 다음을 묻는다:
   - **1. 본원적 경쟁력** — "이 과제가 조직의 핵심 가치사슬과 연결되나? 아니면 곁가지 효율화인가?"
   - **2. 전략·KPI 연계** — "01에서 만든 연계표의 어느 조직목표·KPI에 1:1로 붙나?"
   - **3. 수익성** — "돈이 되나? ΔS(절감공수)와 N(처리량)을 곱하면 가치가 충분한가?"
   - **4. 확장성** — "다른 팀/라인/제품으로 수평 확장 가능한가? 확장하면 N이 얼마나 커지나?"
   - **5. 시급성** — "지금 안 하면 무슨 손해가 커지나? 03의 시급성 점수와 일치하나?"
   - **6. 데이터 준비** — "필요한 데이터가 이미 있나? 추측이 아니라 실제로 접근 가능한가?"
   - **7. 준비계획 구체성** — "착수 계획·검증 프로토콜(PoC)이 구체적인가, 아니면 막연한 의지인가?"
3. **수렴(임계 합의)** — Σ를 합산하고, **τ_gate(착수 임계)** 를 조직의 자원·리스크 허용치에 맞춰 합의해 채운다(예: 14~15/21). Σ ≥ τ_gate면 착수, 미달이면 보류/재설계로 판정한다.
4. **정리** — 게이트 승인 주체(`<담당 임원>`)와 판정 결과를 기록하고, ROI 산식 블록의 변수(s·s′·N)를 누가 언제 실측할지 다음 단계(07/08) 책임자로 넘긴다.

**완료(Done) 정의**: 7문항 점수+근거가 모두 채워졌고 · Σ와 τ_gate가 확정됐으며 · 착수/보류/재설계 판정과 승인 주체가 적혔다.

## 흔한 실수 · 체크포인트

- **점수 인플레(전부 3점)** — 모든 칸이 만점이면 게이트가 무력화된다. 1·2점이 하나도 없으면 "정말 우수한가, 아니면 후하게 준 건가"를 되묻는다. 통과 기준: 적어도 한 문항은 정직하게 낮은 점수가 나오는 채점.
- **데이터 준비(6번) 과대평가** — "데이터 있다"는 *존재*와 *바로 쓸 수 있음*은 다르다. 라벨링 공수·접근권한·DRM·품질 편차를 빼고 3점 주지 말 것. 존재만 확인됐으면 후속 KSF에서 다룰 리스크를 근거 칸에 명시한다.
- **근거 없는 점수** — 점수만 적고 근거 칸을 비우면 추적성이 끊긴다. 모든 점수는 01~04 산출물의 특정 항목에서 연역돼야 한다.
- **수익성(3번)을 막연한 인상으로 채점** — ΔS·N을 정의하지 않고 "효과 클 듯"으로 3점 주지 말 것. ROI 산식 블록의 변수를 누가 실측할지 정하지 않으면 점수는 미확정으로 둔다.
- **임계(τ_gate) 없이 점수만** — Σ만 구하고 임계를 합의하지 않으면 게이트가 아니다. 반드시 τ_gate를 먼저 정하고 Σ와 비교한다.
