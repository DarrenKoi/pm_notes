# 02 · 대상 업무영역 정의 — 템플릿

> 한 줄: 대상 업무영역(L1)을 L2 묶음 × L3 일 순서로 펼치고, 각 L3에 **AI 대체 판정**을 얹어 "어디를 AI로/사람으로" 가리고 **1차 PoC 대상**을 좁힌다.

## 이 단계가 무엇인가 (Why)

[01-전략일관성-확인](./01-전략일관성-확인.md)에서 "조직목표 → 핵심업무 → AI 후보 과제"의 추적성을 세웠다. 하지만 그 핵심업무가 **더 큰 업무 지도의 어디에 위치하는지**, 그리고 그 주변 업무 중 **무엇을 AI로 대체하고 무엇을 사람이 쥐고 갈지**를 가리지 않으면 PoC 범위가 흐려진다. 이 단계는 그 경계를 그린다.

- **"모든 칸에 AI를 넣지 않는다"**: AI는 **반복성·정형성·공수·데이터 가용성**이 큰 지점에만 선별 배치한다 (출처: [원문 06](../source/06-sales-ai-opportunity.md) "반복성·정형성·공수가 큰 Task 지점 선별", [01 기획문서 STEP2/STEP3](../01-기획문서_AX서비스기획.md)).
- **개입 최소 단위는 L5(Task)**: 업무를 L1~L5로 분해해야 AI를 배치할 자리를 정밀하게 식별할 수 있다 (출처: [원문 05](../source/05-process-level-definition.md) 해석 포인트). **이 단계는 L1~L3까지만** 가로로 펼쳐 1차 대상 L3를 고르고, L4~L5 분해와 정량화는 다음 단계 [03-painpoint-도출-정량화](./03-painpoint-도출-정량화.md)로 넘긴다.
- **다음 단계가 이걸 왜 필요로 하나**: 이후 모든 산출물(과제 정의·As-Is/To-Be·일정)이 **같은 범위 위에서** 작성되려면, 여기서 1차/차기/범위 밖을 명문화해 합의해 두어야 한다.

## 작성할 산출물 (What)

세 가지를 만든다: **① L1~L3 가로 레이아웃**, **② AI 대체 판정 표**, **③ 범위 결정**. 아래 빈 양식을 그대로 채운다.

### A. L1~L3 가로 레이아웃 (Process Level Map)

L1은 **하나**(이번 워크숍이 다룰 업무영역 전체). 그 아래 L2를 **N개 컬럼**으로 좌→우에 두고, 각 컬럼 안에서 L3가 위→아래 흐름을 갖는다. (포맷 출처: [원문 05](../source/05-process-level-definition.md)의 L1~L5 가로 위계 표)

```
L1 (Mega Process) :  〈대상 업무영역 전체 = ___________〉
================================================================================================================
                                                                                      │ 1차 PoC 대상 ★
   ┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────────┐
L2 │ 〈L2-A〉            │   │ 〈L2-B〉            │   │ 〈L2-C〉            │   │ 〈L2-D〉  …(N개)    │
   │   ___________   ★  │   │   ___________       │   │   ___________       │   │   ___________       │
   └─────────┬───────────┘   └─────────┬───────────┘   └─────────┬───────────┘   └─────────┬───────────┘
             │ (L3 flow)               │ (L3 flow)               │ (L3 flow)               │ (L3 flow)
             ▼                         ▼                         ▼                         ▼
L3   〈L3 ①〉 ___________     〈L3 ①〉 ___________     〈L3 ①〉 ___________     〈L3 ①〉 ___________
        │                       │                         │                        │
        ▼                       ▼                         ▼                        ▼
     〈L3 ②〉 ___________ ★★  〈L3 ②〉 ___________     〈L3 ②〉 ___________ ◇   〈L3 ②〉 ___________
        │                       │                         │                        │
        ▼                       ▼                         ▼                        ▼
     〈L3 ③〉 ___________     〈L3 ③〉 ___________     〈L3 ③〉 ___________     〈L3 ③〉 ___________
        │
        ▼
     〈L3 ④〉 ___________

  ★  = 1차 PoC 대상 L2
  ★★ = 1차 PoC 대상 L3
  ◇  = 차기(2차) 대상 L3
```

> **작성 가이드**
> - L1 박스 = 워크숍이 다루는 업무영역 한 덩어리(예: "○○ 운영" 전체).
> - L2 컬럼 = L1을 "큰 묶음"으로 가른 것. 보통 3~5개. 서로 겹치지 않고 합치면 L1 전체가 되도록(MECE) 자른다.
> - 각 L2 아래 L3 = 그 묶음 안의 **실제 일 순서**(접수→처리→검증→마무리 식). 화살표는 시간 흐름.
> - ★/★★/◇ 마킹은 판정 표(B)를 채운 뒤 되돌아와 붙인다. 처음부터 정답을 정하지 말 것.
> - 도메인 위계가 확실치 않으면 해당 칸 옆에 **`(확인 필요 — 도메인)`** 을 달고, 현장 전문가 검증 전까지 가설로 취급한다.

### B. AI 대체 판정 표

각 L3가 AI에 맞는지/사람이 쥐어야 하는지를 **4기준**으로 판정해 한 행씩 채운다.

**판정 범례**

| 아이콘 | 의미 |
|--------|------|
| 🤖 | **AI 대체** — 반복·정형·고공수, AI가 주(主) 수행 |
| 🤝 | **부분 대체** — AI 보조 + 사람 판단 병행(Human-in-the-loop) |
| 🧑 | **사람** — 판단·승인·물리작업, AI 부적합 |
| ⚙️ | **기존 자동** — 이미 시스템/설비가 자동화, 신규 AI 불필요 |

**판정 기준 (4가지 — 클수록 🤖 쪽으로)**

| 기준 | 묻는 것 | 높을 때 의미 |
|------|---------|--------------|
| **반복성** | 같은 작업이 얼마나 자주·규칙적으로 반복되나? | 자동화 ROI ↑ → AI 후보 |
| **정형성** | 입력·절차·판단 규칙이 명확히 정의되나?(예외가 적나) | 모델·룰로 다루기 쉬움 → AI 후보 |
| **공수** | 사람 손이 얼마나 많이 드나?(시간·인원) | 대체 시 절감 효과 ↑ |
| **데이터 가용성** | 학습/판단에 쓸 데이터가 이미 쌓여 있나? | 없으면 아무리 정형이어도 당장 불가 |

> 4기준이 **모두 높으면 🤖**, 일부만 높거나 최종 판단이 사람 몫이면 **🤝**, 비정형·맥락·물리작업이면 **🧑**, 이미 자동이면 **⚙️**. 특히 **데이터 가용성이 낮으면** 다른 셋이 높아도 1차 대상에서 내린다(선결 과제로 [09-ksf-제약-정의](./09-ksf-제약-정의.md)에 기록).

각 L2별로 표를 하나씩 만든다(빈 양식):

#### 〈L2-A〉 ___________

| L3 | 판정 | 근거 (반복성·정형성·공수·데이터가용성) |
|----|:----:|------|
| 〈L3 ①〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |
| 〈L3 ②〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |
| 〈L3 ③〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |
| 〈L3 ④〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |

#### 〈L2-B〉 ___________

| L3 | 판정 | 근거 |
|----|:----:|------|
| 〈L3 ①〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |
| 〈L3 ②〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |
| 〈L3 ③〉 ___________ | 🤖/🤝/🧑/⚙️ | ___________ |

> 〈L2-C〉, 〈L2-D〉 … 도 같은 형식으로 추가. L3가 많으면 행을 늘린다.

### C. 범위 결정 (Scope Decision)

판정 표에서 🤖/🤝가 모인 곳 중 4기준이 가장 강한 한두 개를 1차로 고른다(빈 양식):

```
1차 PoC 대상  =  〈L2-?〉 › 〈L3 ?〉 ___________
                 └ 근거: ___________ (반복성·정형성·공수·데이터 가용성 중 무엇이 최댓값인가)
                 └ 집중 지점(내부 최대 병목): ___________

차기(2차) 대상 =  〈L2-?〉 › 〈L3 ?〉 ___________
                 └ 근거: ___________
                 └ 착수 시점: 1차 안정화 후 ___________ 단계

범위 밖(이번)  =  〈___〉(이미 자동) · 〈___〉(물리/판단 작업) · 〈___〉(사람 승인 유지)
```

> **작성 가이드**: 1차는 **하나(많아야 둘)** 로 좁힌다. 욕심내면 PoC가 흐려진다. 차기·범위 밖을 적어두는 이유는 "안 하기로 했다"를 합의로 남겨, 이후 단계에서 범위가 슬금슬금 늘어나는 것을 막기 위해서다.

## 퍼실리테이션 가이드 (How to run)

**타임박스 권장: 60~90분** (준비 10 → 발산 25 → 수렴 20 → 정리 15)

1. **준비** — 01에서 확정한 "핵심업무 / AI 후보 과제"를 화이트보드 상단에 붙인다. 그것이 속한 업무영역 전체(L1)를 한 문장으로 합의.
2. **발산 (L2→L3 펼치기)** — 도메인 전문가가 업무를 큰 묶음(L2)으로 가르고, 묶음마다 일 순서(L3)를 포스트잇으로 깐다. 빠짐·겹침을 점검(MECE).
3. **수렴 (판정 얹기)** — L3마다 4기준을 묻고 🤖/🤝/🧑/⚙️ 스티커를 붙인다. 근거를 한 줄로 받아 적는다.
4. **정리 (범위 확정)** — 🤖/🤝가 모인 곳에서 1차/차기/범위 밖을 도출하고, A 다이어그램에 ★/★★/◇를 마킹.

**참가자에게 던질 질문** (이 단계 고유)
- "이 업무영역(L1)을 **큰 묶음(L2)**으로 나누면 몇 덩어리인가요? 합치면 전체가 되나요?"
- "각 묶음의 **실제 일 순서(L3)**는 어떻게 흘러가나요? 접수에서 마무리까지 짚어 주세요."
- "각 L3는 얼마나 **반복적·정형적**인가요? 예외가 자주 튀나요?"
- "그 일에 **사람 손이 얼마나** 드나요? 가장 손이 많이 가는 지점은?"
- "거기 쓸 **데이터가 이미 있나요?** 없다면 무엇부터 모아야 하나요?"
- "그래서 **어디를 AI로, 어디를 사람이** 쥐어야 할까요? 사람이 끝까지 잡아야 할 승인·판단은?"

**완료 정의 (산출 기준)**
- L1 1개 + L2 N개 + 각 L2의 L3 플로우가 다이어그램으로 그려졌다.
- 모든 L3에 🤖/🤝/🧑/⚙️ 판정과 한 줄 근거가 붙었다.
- 1차/차기/범위 밖이 명문화되고 참가자 합의를 받았다.
- 확정 못한 위계는 `(확인 필요 — 도메인)`으로 표시되어 후속 검증 대상으로 남았다.

## 흔한 실수 · 체크포인트

- **L2를 너무 잘게 쪼갬** → 컬럼이 8개 넘으면 일 묶음이 아니라 작업 목록이다. 3~5개로 묶어라. *통과 기준: L2 ≤ 5.*
- **L2를 너무 크게 묶음** → "전부"가 한 칸이면 판정이 안 된다. 안에서 일 순서(L3)가 3개 이상 나오도록 가른다.
- **대체 판정을 근거 없이** → "🤖 같아서 🤖"는 금지. **반복성·정형성·공수·데이터가용성** 중 어느 게 높아서 그 판정인지 한 줄로 적힌 행만 인정.
- **데이터 가용성 무시** → 정형·반복이어도 데이터가 없으면 1차 불가. 빠뜨리면 PoC가 모델 학습 데이터부터 막힌다.
- **1차 대상을 욕심내 여러 개** → 1차는 하나(많아야 둘). 나머지는 차기로 미뤄 합의.
- **사람 몫을 안 남김** → 최종 승인·판단·물리작업까지 🤖로 칠하면 현장 수용 저항·리스크가 커진다. **HITL(🤝)·사람(🧑) 경계**를 분명히.
