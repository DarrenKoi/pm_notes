# 07 · As-Is 업무 프로세스 정의 — 템플릿 (Step9 / Design)

> 한 줄: 대상 업무를 **지금 사람이 어떻게 하는지** 플로우차트 + 단계 테이블(소요시간·사용 시스템)로 고정해, To-Be 설계(#08)·수익성(#05)·기대효과(#10)의 **기준선(baseline)**을 만든다.

## 이 단계가 무엇인가 (Why)

To-Be를 그리려면 **무엇을 바꾸는지의 기준선**이 먼저 있어야 한다. As-Is가 정량화되어 있지 않으면 "얼마나 좋아졌는가"를 말할 수 없고, 결국 #05 과제적정성(수익성 ΔS)도 #10 개발일정의 기대효과도 공중에 뜬다 (출처: [02 기술문서 Step9](../02-기술문서_AI과제정의구현.md)).

- **#08 To-Be의 출발선**: To-Be 플로우차트는 As-Is와 같은 축 위에 🤖/🧑 레인을 다시 칠하는 것이다. As-Is가 확정돼야 대비 테이블의 *before* 칸이 채워진다.
- **#05 수익성의 분모·분자**: ROI ≈ 절감공수 ΔS × 처리량 N. ΔS는 본 단계의 단계별 소요시간(특히 최대 병목)과 To-Be 소요시간의 차에서 나온다.
- **#10 기대효과의 근거**: "공수↓·처리량 상한 돌파"라는 기대효과는 As-Is에서 **어느 단계가 실제로 최대 병목인지**를 보여야 설득력이 생긴다.
- **#09 제약의 출처**: 사용 시스템 컬럼이 곧 "특정 시스템·버전 의존"이라는 시스템·인프라 제약의 근거가 된다.

> 산출 도구는 원문 07의 Swimlane Template(🤖 AI Agent 수행 / 🧑 Human 수행 구분)을 따른다 (출처: [원문 07](../source/07-ai-agent-scope-template.md)). 단, **As-Is에서는 거의 모든 칸이 🧑 사람**이다 — 🤖 대체는 #08 To-Be에서 시작된다.

## 작성할 산출물 (What)

두 가지를 산출한다: **(1) As-Is 플로우차트**, **(2) 단계별 테이블**. 모든 칸은 빈칸/꺾쇠 placeholder다.

### (1) As-Is 플로우차트 (빈 틀)

`〈①〉→〈②〉→…` 순서로 단계 박스를 세로로 잇고, 모든 박스 앞에 🧑를 붙인다(As-Is는 전부 사람 수행). 병목 단계는 박스 우측에 `◀ 병목`, 가장 심한 단계는 `◀ 최대 병목`으로 표시한다. **재시도/재작업이 도는 단계**는 아래로 빠지는 화살표가 다시 위로 돌아오는 `[ 반복 루프 ]`로 그린다.

> 작성 가이드: 단계 수는 고정이 아니다 — 실제 업무 흐름대로 박스를 늘리거나 줄인다(아래 틀은 ①~④ + 반복 루프 예시일 뿐). 박스 안에는 *동사로 끝나는 한 줄 작업명*만 넣는다.

```
  [입력] 〈이 업무를 트리거하는 입력·의뢰는 무엇인가〉
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ① 〈첫 단계 작업명 ___〉                                     │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ② 〈다음 단계 작업명 ___〉                        ◀ 병목    │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
        ┌───────────────────────────────────────────────────┐
        │ 🧑 ③ 〈시도→확인 후 잘못된 곳을 고치는 단계 ___〉   │ ◀ 최대 병목
        └───────────────────────────────────────────────────┘
              │                                        ▲
              │ 〈통과 기준 미달 → 수정 후 재시도〉      │
              └────────────────[ 반복 루프 ]────────────┘
              │
              │ 〈통과〉
              ▼
  ┌──────────────────────────────────────────────────────────────┐
  │ 🧑 ④ 〈마지막 단계 작업명(저장·검증·전달 등) ___〉             │
  └──────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    [산출] 〈이 업무의 최종 결과물은 무엇인가〉

  범례:  🧑 Human 수행 업무   ◀ 병목 구간   [ 반복 루프 ] 〈어느 단계가 반복되나〉
```

### (2) 단계별 테이블 (빈 양식)

| 단계 | 설명 | 소요시간 | 사용 시스템 |
|------|------|----------|-------------|
| 🧑 ① 〈작업명〉 | 〈무엇을·왜 하는지 한 줄〉 | `s₁` | 〈___〉 |
| 🧑 ② 〈작업명〉 *(병목)* | 〈___〉 | `s₂` | 〈___〉 |
| 🧑 ③ 〈작업명〉 *(최대 병목)* | 〈… → 재시도 **반복 루프**〉 | `s₃` *(최대)* | 〈___〉 |
| 🧑 ④ 〈작업명〉 | 〈___〉 | `s₄` | 〈___〉 |
| **합계** | **As-Is 리드타임 (1건당)** | **`ΣS`** | — |

> **소요시간(`s`)을 어떻게 채우나**: 실측 전에는 빈칸 대신 단계별 placeholder `s₁..sₙ`을 두고, 가장 오래 걸리는 단계에 `(최대)`를 표기한다. 합계 행 `ΣS = s₁+…+sₙ`이 **1건당 As-Is 리드타임**이다. 캠프 중 엔지니어 실측(또는 로그/타임스탬프)으로 `s₁..sₙ`·`ΣS`에 실제 분/시간을 채워 확정한다. **추정 없이 빈칸으로 두지 말 것** — 최소한 "대·중·소" 상대 크기라도 표기해야 병목이 드러난다.
>
> **사용 시스템을 어떻게 채우나**: 각 단계에서 실제로 띄워 쓰는 화면/툴(의뢰 입력 시스템, 작업 GUI, 참조 DB 등)을 적는다. 같은 시스템이 여러 단계에 반복 등장하면 그 시스템이 #09 제약(버전·접근 의존)의 후보다. **구체 제품·버전 명칭이 기밀이면 일반 명칭으로 통칭**한다(예: "작업 GUI", "참조 DB").

**백데이터 연결**: 소요시간 컬럼은 (1) #08 To-Be 대비 테이블의 *before* 값, (2) #05 수익성 `ΔS = ΣS − ΣS′`의 As-Is 항, (3) #10 기대효과(병목 공수↓)의 정량 근거로 그대로 흐른다. 사용 시스템 컬럼은 #09 시스템·인프라 제약의 출처가 된다.

## 퍼실리테이션 가이드 (How to run)

진행 순서: **준비 → 발산(흐름 받아쓰기) → 수렴(병목·소요시간 합의) → 정리(테이블 확정)**. 타임박스 권장 **45~60분**.

1. **준비 (5분)**: 화이트보드에 가로축을 비워두고 "입력 → … → 산출"만 그려둔다. #06 과제 정의서의 "AI Agent 대상 업무 범위"를 옆에 띄워 As-Is 그릴 구간을 한정한다.
2. **발산 (15~20분)**: 현장 담당자에게 일을 **있는 그대로** 말하게 하고 박스를 채운다. *지금 어떻게 하는지*만 — 개선안은 받아 적되 별도 칸(parking lot)에 빼둔다.
3. **수렴 (15분)**: 병목·반복 루프·소요시간을 합의한다.
4. **정리 (10분)**: 단계별 테이블에 `s₁..sₙ`·`ΣS`·사용 시스템을 옮겨 적고, 미확정 수치는 placeholder로 두되 실측 책임자·기한을 함께 적는다.

**참가자에게 던질 질문 (이 단계 고유, SPEC §3 기준)**:
- "지금 이 일을 **처음부터 끝까지 순서대로** 말해주세요 — 첫 트리거(입력)부터 최종 결과물까지."
- "각 단계는 보통 **몇 분/몇 시간** 걸리나요? 가장 오래 걸리는 단계는?"
- "각 단계에서 **어떤 시스템/화면**을 쓰나요?"
- "**어디서 막히나요(병목)?** 다시 앞으로 되돌아가 **재작업하는 구간**이 있나요? 보통 몇 번 반복?"

**완료 정의(이 단계 통과 기준)**:
- [ ] 입력→산출까지 단계 박스가 빠짐없이 이어졌다 (전부 🧑 사람 수행).
- [ ] 병목(`◀`)·최대 병목·반복 루프가 시각적으로 표시됐다.
- [ ] 테이블에 `s₁..sₙ`·`ΣS`·사용 시스템이 모두 있다 (실측은 미정이라도 placeholder/상대크기 존재).

## 흔한 실수 · 체크포인트

- **To-Be를 As-Is에 섞음** — "AI가 하면 빨라질 텐데"를 As-Is 박스에 적는 순간 기준선이 오염된다. 개선안은 전부 parking lot으로 빼고, As-Is는 **전부 🧑 사람**으로만 그린다. (🤖는 #08에서 등장)
- **소요시간 추정 없이 빈칸** — `s`를 비워두면 #05 ΔS·#10 기대효과가 계산 불가다. 실측 전이라도 placeholder + 상대 크기(대/중/소)로 **최대 병목만큼은** 반드시 표기한다.
- **병목을 표시 안 함** — 모든 단계가 평평하면 To-Be가 어디를 칠지 알 수 없다. 최소 1개는 `◀ 최대 병목`이어야 한다.
- **반복 루프 누락** — 재시도/재작업이 As-Is 시간을 지배하는 경우가 많다. "통과 기준 미달 → 되돌아감"을 화살표로 그렸는가.
- **시스템 명칭에 기밀 노출** — 구체 제품·버전 대신 일반 명칭으로 통칭했는가.

**통과 기준**: 위 5개 함정을 모두 피했고, "이 As-Is 한 장으로 #08 before 칸·#05 ΔS 분자가 채워지는가?"에 Yes면 통과.
