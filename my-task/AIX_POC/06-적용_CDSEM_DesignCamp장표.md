## 장표 0 — 표지

# **CD-SEM Auto Recipe Creation** (VLM GUI-제어 기반)

> 실측 좌표 재정합·오인식 수정(1차 PoC = Align fail 대응 자동화 = **Smart Align Agent**)을 AI가 수행하고, 엔지니어는 예외만 개입한다 (Human-in-the-loop)

| | |
|---|---|
| **구분** | ☐ Marketing ☐ R&D ■ **Production** ☐ Support |
| **담당 조직** | ITC AIX / MI |
| **Champion** | `<담당 임원>` |
| **발표일** | New AI Design Camp (2026) |

**한 줄 정의** — VLM이 SEM 화면을 직접 보며 GUI를 클릭·입력해 Recipe를 생성하고, **실측 시작 시 발생하는 좌표 shift·align 실패·패턴 오인식을 실시간으로 재정합·수정**한다. 완전자동화는 목표가 아니다.

> 🎤 **발표 스크립트**
> "저희 과제는 CD-SEM **Auto Recipe Creation**, 그중에서도 **실측이 시작될 때 좌표가 틀어지고 패턴을 잘못 인식해 엔지니어가 반복 수정하는 구간(1차 PoC = Align fail 대응 자동화)**을 VLM으로 자동화하는 것입니다. 핵심 메시지 하나만 기억해 주십시오 — **완전자동화가 아니라, 반복은 AI가 하고 예외 판단·최종 검증은 사람이 하는 Human-in-the-loop**입니다. 구분은 Production, 담당은 ITC AIX/MI입니다."

## 장표 1 — 왜 이 과제인가 (Baseline 3대 원칙 충족)

> 과제는 개인 흥미가 아니라 **3대 원칙(가치·데이터·인간 중심)** 으로 검증한다. (출처: [원문 02](./source/02-ax-consulting-methodology.md))

| 원칙 | 이 과제에서의 충족 |
|------|--------------------|
| **가치 중심** | **R&D 디바이스 개발 지연 최소화**(조직 목표)에 직결. 최대 병목(오인식 수정 반복 ⑥)을 정조준. |
| **데이터 중심** | SEM 화면 이미지·reference recipe 등 **이미 존재하는 가용 데이터**로 동작. 추측 설계가 아님. |
| **인간 중심** | 엔지니어를 대체하지 않고 **확장** — 반복 셋업은 AI, 예외 판단·최종 검증은 사람. |

➕ **단순 효율화를 넘는 지점** — 일관 기준으로 **측정 장비 간 Skew Zero**(숙련도·장비 편차 제거)를 확보하고, 사람 작업시간 제약을 풀어 **24h 무중단 셋업 → MI 장비 가동률 극대화**(유휴시간 축소)를 가능케 한다.

> 🎤 **발표 스크립트**
> "왜 이 과제냐? 캠프 3대 원칙으로 걸러봤습니다. **가치** — 조직 목표인 R&D 디바이스 개발 지연 최소화에 바로 꽂힙니다. **데이터** — 새로 수집할 게 아니라 이미 쌓여 있는 SEM 이미지와 reference recipe로 됩니다. **인간** — 엔지니어를 빼는 게 아니라, 야간·주말 반복 셋업을 AI가 메워 사람을 더 가치 있는 판단에 쓰게 합니다. 특히 일관 기준으로 장비 간 Skew를 없애고, 24시간 무중단으로 **장비 가동률을 끌어올립니다.**"

## 장표 2 — [Step 1] 조직목표 ↔ 과제 연계 (Top-Down)

> 캠프 템플릿([원문 04](./source/04-org-goal-task-linkage.md))을 그대로 미러링 — 조직 목표 → 업무 목표 → 핵심 업무 → **AI 대상 후보 과제**.

| 조직 목표 | 업무 목표 | 관련(핵심) 업무 | AI 대상 후보 과제 |
|-----------|-----------|-----------------|--------------------|
| **R&D 디바이스 개발 지연 최소화** (핵심) | • 검/계측 효율화<br>• 장비 운영/관리 최적화<br>• 장비 개발·도입 | • **MI Recipe Solution 제공**<br>• **측정 장비 간 Skew Zero**<br>• **MI 장비 가동률 극대화**<br>• 장비 개발·도입 및 양산 이관 | **Auto Recipe Creation**<br>(1차 PoC = Align fail 대응 자동화) |

```
조직 목표            →  업무 목표              →  관련(핵심) 업무                →  AI 대상 후보 과제
R&D 디바이스           ① 검/계측 효율화          ① MI Recipe Solution 제공          Auto Recipe Creation
개발 지연 최소화        ② 장비 운영/관리 최적화    ② 측정 장비 간 Skew Zero           (1차 PoC = Align fail 대응 자동화)
(핵심)                 ③ 장비 개발·도입          ③ MI 장비 가동률 극대화
                                                ④ 장비 개발·도입 및 양산 이관
```

> 한 과제(Auto Recipe Creation)가 핵심업무 ①~③에 동시 기여 — ① 솔루션 본체, ② 일관 기준으로 숙련도·장비 편차 제거(구 "숙련도 편차 제거" 흡수), ③ 24h 무중단으로 유휴 축소(구 "처리량 확대 / 24h 무중단" 흡수). ④ 장비 개발·도입 및 양산 이관은 업무 목표 ③의 추적 도착지(양산 이관 = 개발 완료품을 양산으로 넘기는 별도 업무, **ARC 범위 밖**).

> 🎤 **발표 스크립트**
> "이 과제가 위에서 내려온 것임을 보여드립니다. **조직 목표**는 R&D 디바이스 개발 지연 최소화입니다. 거기서 **업무 목표** — 장비 운영/관리 최적화, 검/계측 효율화 — 가 나오고, 이걸 좌우하는 **관련(핵심) 업무**가 MI Recipe Solution 제공·측정 장비 간 Skew Zero·MI 장비 가동률 극대화입니다. 이 셋에 동시에 기여하는 AI 후보가 **Auto Recipe Creation**이고, 1차 PoC로 **Align fail 대응 자동화**을 칩니다. 바텀업 아이디어가 아니라 Top-Down 연계라는 점을 강조하고 싶습니다."

## 장표 3 — [Step 2~3] L1~L5 프로세스 분해 (End-to-End) · AI 배치 후보

> 캠프 템플릿([원문 05](./source/05-process-level-definition.md))을 미러링 — AI 개입 **최소 단위는 L5(Task)**. L4를 셋업만이 아니라 **의뢰 접수~모니터링 End-to-End로 펼친 뒤** 배치 후보를 식별.

| L1 (Mega) | L2 (Group) | L3 (Process) | L4 (Activity) | L5 (Task) — **AI 배치 후보** |
|-----------|------------|--------------|---------------|------------------------------|
| 계측(MI) 운영 | CD-SEM Recipe Management | Recipe Creation | **[A] 의뢰 접수 → [B] 셋업①~⑦ → [C] 검증·등록 → [D] 후속 모니터링** | **B-③ 패턴 등록 · B-④ 좌표·박스 · B-⑥ 오인식 수정** (핵심)<br>C-① Recipe 준수 모니터링(차기) · A-2 의뢰서 플래깅(보조) |

```
L1 계측(MI) 운영
 └ L2 CD-SEM Recipe Management
    └ L3 Recipe Creation
       └ L4 : [A] 의뢰 접수·검토 → [B] Recipe 셋업(①~⑦) → [C] 검증·등록 → [D] 후속 계측 모니터링
          └ L5(AI 후보) : B-③ 패턴 · B-④ 좌표·박스 · B-⑥ 오인식 수정 (핵심) / C-① Recipe 준수 모니터링(차기)
```

> 🎤 **발표 스크립트**
> "업무를 캠프 방식대로 5레벨로 쪼갰습니다. 중요한 건 L4를 **셋업만이 아니라 의뢰 접수부터 검증·등록, 후속 모니터링까지 Recipe 전 생애**로 펼쳤다는 점입니다. 그래야 어디에 Pain이 있는지 빠짐없이 봅니다. 그 위에서 AI를 꽂을 **L5 후보**를 찍으면 — 핵심은 셋업의 ③ 패턴, ④ 좌표, ⑥ 오인식이고, 계측 데이터·모니터링은 차기 후보입니다. (양산은 우리 운영 대상이 아니라 개발 완료품을 양산으로 넘기는 별도 이관 업무라 펼침에서 제외했습니다.) 다음 장에서 전체 SOP를 펼쳐 보이겠습니다."

## 장표 4 — End-to-End 표준 업무 절차(SOP) · 병목·공백

> Recipe Creation 셋업만이 아니라 **의뢰 접수→셋업→검증·등록→후속 모니터링** Recipe 전 생애를 펼침. ❗ = 병목/공백. ([B] 셋업 ①~⑦은 엔지니어 확정 / [A]·[C]·[D] 세부는 확인 필요)

```
[A] 의뢰 접수·검토
  A1 공정/소자 엔지니어가 MIDAS에 측정 의뢰서 제출 (도면·측정위치·파라미터·측정방식, 현재 PPT)
  A2 계측 엔지니어 의뢰서 접수·정보 충분성 검토 (부족 시 문의·반려)
[B] Recipe 셋업  (= As-Is ①~⑦)
  ① 레이어/패턴 확인  ② Recipe 생성·웨이퍼 로드
  ③ Alignment/Addressing 등록 ❗   ④ 좌표·박스·알고리즘 ❗   ⑤ 이미징 조건
  ⑥ 시험측정(Try)→오인식/실패 수정 (반복) ❗❗ 최대병목     ⑦ 저장·등록·검증
[C] 검증·등록  (⑦의 연장)
  C1 검증 완료 Recipe 등록 → 등록 Recipe로 개발 샘플 자동 계측   C2 결과 수집·1차 판정
[D] 후속 계측 모니터링·이상 대응
  D1 측정 안정성·Recipe 준수 모니터링(재현성·드리프트) ❗ 공백   D2 이상 원인 분석   D3 보정·재셋업 → [B] 피드백
```

| 구간 | 성격 | 자동화 시사점 |
|------|------|----------------|
| A 의뢰 | 입력 품질이 셋업 난이도 좌우 | 부족정보 플래깅(보조). MIDAS 전산화는 **외부 변수** |
| B 셋업 | **병목 집중**(③④⑥) | **1순위 자동화 대상** |
| C 검증·등록 | 등록 Recipe로 개발 샘플 자동 계측, 수작업 적음 | 자동화 여지 낮음 (※양산 아님 — 양산은 별도 이관 업무) |
| D 모니터링 | 사람 주기점검 의존 → 늦은 발견 | **차기 후보**(Recipe 준수 모니터링·이상탐지) |

> 🎤 **발표 스크립트**
> "표준 업무 절차 전체입니다. 앞서 지적받은 대로 셋업 한 토막이 아니라 **A 의뢰 접수 → B 셋업 → C 검증·등록 → D 후속 모니터링**까지 펼쳤습니다. 보시면 빨간 표시, 즉 병목은 **B 셋업의 ③④⑥에 몰려** 있고, D 모니터링은 사람이 주기적으로 들여다봐서 이상을 늦게 잡는 **공백**입니다. C 검증·등록은 등록된 Recipe로 개발 샘플이 자동 측정돼 손이 거의 안 갑니다 — 참고로 양산은 우리 운영 대상이 아니라 개발 완료품을 양산으로 넘기는 별도 이관 업무입니다. 그래서 **자동화 1순위는 B 셋업, 차기는 D 모니터링**으로 가닥이 잡힙니다. A 의뢰서는 MIDAS 전산화가 별도 프로젝트라 저희는 부족정보 플래깅으로만 연계합니다."

## 장표 5 — [Step 4] 망라적 Pain Point 리스트 (SOP 단계별)

> 선택된 가지 하나가 아니라 **A~D 전 단계 Pain을 빠짐없이** 깐다(캠프 [원문 08](./source/08-final-output-example.md) 양식). 🔴 = 심각도 예비 표기 — 정량 점수는 다음 장.

| 단계 | Pain Point | 심각도(예비) | 비고 |
|------|------------|:---:|------|
| A 의뢰 | 의뢰서 정보 부족·표준 미흡 → 셋업 시 재문의·추측 | 🔴 | 숙련도 편차의 입력측 원인 (MIDAS 연계) |
| B-③④ 셋업 | 패턴 등록·좌표/박스 설정이 수작업·숙련 의존 | 🔴🔴 | 셋업 병목 |
| **B-⑥ 셋업** | **시험측정 후 오인식/실패 수정 반복** | 🔴🔴🔴 | **최대 병목** (빈도·공수·숙련 모두 높음) |
| B 전반 | 야간/주말 셋업 공백 (작업시간 제약) | 🔴 | 처리량 상한 제약 |
| C 검증·등록 | 셋업 부실 시 오측 유발 (등록 Recipe 자동 계측, 수작업 적음) | 🔵 낮음 | 자동화 여지 적음 |
| D-1 모니터링 | 이상 탐지가 사람 주기점검 의존 → 늦은 발견·야간 공백 | 🔴🔴 | **신규 자동화 후보** |

> 🎤 **발표 스크립트**
> "Pain Point도 셋업 한 군데만 본 게 아니라 **A부터 D까지 전 단계를 훑어** 표로 깔았습니다. 빨간 점이 많을수록 아픈 곳인데, 압도적인 1위가 **B-⑥ 오인식 수정 반복**입니다 — 빈도·공수·숙련 의존이 다 높습니다. 그다음이 셋업의 ③④, 그리고 모니터링 D-1입니다. 반대로 C 검증·등록(등록 Recipe 자동 계측)은 파란색, 별로 안 아픕니다. 이렇게 **넓게 펼쳐 놓고** 다음 장에서 점수로 우선순위를 매깁니다."

> 🎤 **발표 스크립트**

## 장표 7 — [Step 7] 정량 우선순위화 & 과제 선정

| 의뢰서 부족정보 플래깅 | A-2 | 中 | 中 | 中 | 低 | 中 | 보조 *(MIDAS 연계)* |

## 장표 10 — [Step 11] KSF · 제약 사항 (4측면)

> 캠프 용어 정의([원문 10](./source/10-check-and-review.md)) 적용 — KSF = 반드시 선결할 핵심 성공 조건 / 제약 = 기술·조직·비용·정책·보안 다각도.

### KSF (선결 성공 조건 — 우선순위 순)

1. **VLM 좌표 재정합 정확도가 엔지니어 수용 기준 이상** — 최대 리스크. 안 되면 ⑥ 자동화 자체가 성립 안 함. **PoC 1순위.**
2. **GUI 제어(computer-use)의 안정적 장비 SW 연동** — 클릭·입력 신뢰성, 장비 SW 버전 의존성 관리.
3. **예외 핸드오프 기준의 명확화** — "어느 정도 오인식이면 사람에게 넘기는가" 임계값(신뢰도/재시도) 정의.
4. **의뢰서 입력의 최소 표준화** — 부족정보 플래깅 동작을 위한 포맷 합의 (MIDAS 연계, 외부 변수).

### 제약 사항 (4측면)

| 측면 | CD-SEM 과제의 실제 제약 |
|------|--------------------------|
| **기술** | 외부 LLM API 전면 차단 → 사내 **Qwen3-VL-8B/30B**만 / 정밀 좌표 재정합 정확도 / 배율·노이즈별 일반화 / GUI 자동제어 안정성(장비 SW 버전 의존) / DB 로컬 미가용 → 검증은 import·문법 수준 |
| **조직** | 장비 운영팀 협업 필수 / 엔지니어 신뢰 확보(⑦ 검증은 사람 유지) / 숙련 엔지니어 라벨·피드백 확보 |
| **비용** | 사내 VLM 추론용 GPU 자원 / 장비 다운타임 없이 검증할 오프라인·섀도우 환경 |
| **정책·보안** | 장비 직접 제어 권한 및 안전 인터록(실 웨이퍼 영향 방지) / 사내 문서 99% DRM → 스크린샷+VLM 추출 / 개인정보·내부 규정 준수 |

> 🎤 **발표 스크립트**
> "성공하려면 반드시 충족할 조건을 우선순위로 정리했습니다. **1순위가 좌표 재정합 정확도** — 앞서 ⚠️로 짚은 그 리스크입니다. 2순위 GUI 제어 안정성, 3순위 예외 핸드오프 기준, 4순위 의뢰서 표준화 순입니다. 제약은 캠프 가이드대로 기술만 적지 않았습니다. **조직**(운영팀 협업·엔지니어 신뢰), **비용**(GPU·섀도우 환경), **정책·보안**(장비 제어 권한·DRM)까지 4측면 다 봤습니다. 특히 외부 LLM이 막혀 있어 **사내 Qwen3-VL만 쓴다**는 게 큰 전제입니다."

## 장표 11 — [Step 12] 개발 일정 (Phase 0~3)

> KSF 우선순위를 단계로 환산. 구체 일자는 별도 계획에서 확정.

```
Phase 0  좌표 재정합 정확도 PoC (KSF#1)  — 섀도우 환경, 실 장비 영향 0   → 실험 설계: 장표 13 / 05
   ▼     판단 게이트: 수용 기준 미달 시 과제 재설계 / 충족 시 진행
Phase 1  단일 layer Recipe 자동 생성 (②~⑤) + ⑥ 부분 자동화
   ▼
Phase 2  ⑥ 오인식 수정 본격 자동화 + 예외 핸드오프(KSF#3) 도입
   ▼
Phase 3  layer 확장 · 운영 이관 · 의뢰서 표준화 연계(KSF#4) · D-1 모니터링 이상탐지(차기 후보)
```

> ✋ **No-Regret 원칙** — 가장 비싸고 위험한 가정(좌표 정확도)을 **Phase 0에서 가장 싸게(오프라인 이미지만)** 먼저 깬다. 미달이면 GUI 제어 개발에 자원을 쓰기 전에 멈춘다.

> 🎤 **발표 스크립트**
> "일정은 KSF 순서를 그대로 단계로 바꿨습니다. **Phase 0가 게이트**입니다 — 실 장비 안 건드리고 오프라인 이미지만으로 좌표 정확도를 검증하고, 여기서 기준 미달이면 GUI 제어 개발에 돈을 쓰기 전에 과제를 재설계합니다. 통과하면 Phase 1에서 단일 layer부터 자동 생성, Phase 2에서 ⑥ 본격 자동화와 예외 핸드오프, Phase 3에서 layer를 넓히고 운영에 넘기며 **모니터링 이상탐지를 차기 후보로** 붙입니다. 싸고 위험한 것부터 먼저 깬다 — 이게 일정의 철학입니다."

## 장표 13 — [Appendix] KSF#1 좌표 재정합 PoC 실험 설계 (Phase 0)

> KSF#1을 검증하는 실험 *설계*(실행 아님). 상세: [05 PoC 실험설계](./05-적용_CDSEM_PoC실험설계.md).

**가설** — H1: 사내 VLM(Qwen3-VL)이 shift/align 실패 상황에서도 좌표를 **엔지니어 수용 기준(τ) 이내로 재정합** 가능 / H0: 오차가 기준 초과로 ⑥ 자동화 불가.

**격리 설계** — GUI 제어(가정 A) 제외, **"이미지→좌표"(가정 B)만** 검증 → 실 장비 영향 0, 실패 원인 분리.

| 측정 대상 | 성공 기준 (형식만 고정, 수치는 엔지니어 확정) |
|-----------|----------------------------------------------|
| (a) 패턴 인식 | 좌표 오차 **P95 ≤ τ** (τ = 측정박스/정렬 마진 기반) |
| (b) 측정박스·좌표 | 재정합 성공률 **≥ s%** (shift/align 실패 케이스 복구율) |
| (c) 오인식 검출 | **recall ≥ r** + false-alarm **≤ f** |

**판단 게이트** — PASS(P95≤τ & recall≥r)→Phase 1 / CONDITIONAL(일부 layer)→범위 좁혀 진행 / FAIL→과제 재설계(fine-tune·전용 모델·hybrid).

**엔지니어 확정 필요** — `τ` / `s` / `r` / `f` / 데이터 규모 `N` · layer 구성 / 정답 라벨 출처 · hard set 기준.

> 🎤 **발표 스크립트**
> "부록이지만 가장 중요한 장입니다. 1순위 리스크를 어떻게 검증할지 실험을 설계했습니다. 핵심 두 가지 — **첫째, GUI 제어를 빼고 '이미지에서 좌표를 맞히는 능력'만 격리**해 봅니다. 그래야 실패해도 원인이 분명하고 실 장비도 안 건드립니다. **둘째, 합격선 τ·s·r 같은 숫자는 제가 정하지 않습니다** — 측정박스 마진과 공정 스펙에 달려 있어 엔지니어가 확정할 자리로 비워뒀습니다. 결과가 미달이면 fine-tune이나 전용 모델로 방향을 틉니다. **오늘 보고드리는 건 이 실험의 설계까지**이고, 실행과 수치는 다음 단계입니다."

## 장표 14 — 마무리 — Ask & 다음 단계

### 한 장 요약
- **무엇** — CD-SEM **Auto Recipe Creation**: VLM GUI 제어로 Recipe를 자동 생성하고, 1차 PoC로 좌표 재정합·오인식 수정(⑥ = Align fail 대응 자동화)을 친다.
- **어떻게** — ②~⑥은 AI, ①·⑦·예외는 사람 (Human-in-the-loop).
- **검증** — KSF#1(좌표 정확도)을 Phase 0 PoC로 가장 먼저 친다. 미달 시 멈춘다.

### Ask (의사결정 요청)
1. **Champion(담당 임원) 지정** — `<담당 임원>`.
2. **PoC 수용 기준 확정** — τ / s / r / f / N · layer 구성 (엔지니어 협의).
3. **선정 정량 점수 확정** — Pain별 빈도·심각성·주요성 + 난이도·비용 (1~5 점수·가중치).
4. **데이터·환경 승인** — 정답 라벨용 기존 Recipe·SEM 이미지 접근, 오프라인 섀도우 환경.
5. **장비 운영팀 협업 채널** 개설 — ⑦ 검증·라벨·피드백 루프.

### 다음 단계
```
오늘(설계 합의) → 엔지니어 수치·점수 확정 → Phase 0 PoC 실행 → 게이트 판정 → Phase 1 착수
```

> 🎤 **발표 스크립트**
> "정리하겠습니다. **반복은 AI, 판단은 사람**으로 CD-SEM 셋업의 최대 병목을 자동화하는 과제이고, 무리한 완전자동화 대신 가장 위험한 가정부터 싸게 검증하는 계획입니다. 오늘 이 자리에서 결정해 주셨으면 하는 건 다섯 가지 — **Champion 지정, PoC 합격 기준 확정, 선정 점수 확정, 데이터·섀도우 환경 승인, 운영팀 협업 채널**입니다. 이게 모이면 바로 Phase 0 PoC에 들어갈 수 있습니다. 감사합니다."
