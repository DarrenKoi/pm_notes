# 원문 10 — Check & Review (To-Be 설계 용어 정의 & 점검 포인트)

## Definition (용어 정의)

| 구분 | 정의 (Definition) | 가이드 핵심 |
|------|-------------------|-------------|
| To-Be 프로세스 | AI를 반영하여 향후 목표 시점에서 조직이 지향하는 이상적·실현 가능한 업무 흐름을 구조화한 프로세스 | 이상적인 일하는 방식 |
| Activity / Task | To-Be 프로세스 내에서 AI Agent, 시스템, 사람의 역할이 재정의된 실행 단위 작업 | 순차적 연결을 고려 |
| AI Agent 역할 | 특정 To-Be Task에서 사람을 보조하거나 대체하여 데이터 처리·분석·판단·생성 작업을 수행하는 주체 | AI 기술 유형 참고 |
| 제약 사항 | To-Be 프로세스 설계·추진 시 기술·시스템·조직·비용·정책 측면에서 반드시 고려하거나 넘을 수 없는 조건과 한계 | 예시: 외부 데이터 수집 상 어려움, 보안 관련 이슈 등 |
| KSF (Key Success Factor) | To-Be 프로세스가 의도한 성과를 달성하기 위해 반드시 충족되어야 하는 핵심 성공 조건 | 반드시 선결해야 할 항목 위주로 작성 |

## Points (점검 포인트)

### ✅ AI·사람 역할의 명확한 구분
각 Task에서 AI가 처리하는 부분과 사람이 최종 판단·승인하는 부분이 Task별로 구분하여 작성되어 있는가?
> 예) AI Agent: 매출 데이터 수집·분류·분석·이상값 탐지 → 사람: 탐지된 원인 판단 및 보고서 최종 승인

### ✅ 시간이 가장 많이 걸리는 구간 개선 여부
As-Is에서 가장 시간이 많이 걸리던 구간에 AI가 배치되어 있고, 해당 구간의 공수가 줄어드는 방향으로 To-Be가 설계되어 있는가?
> 예) As-Is: 매주 수율 데이터 수동 취합 및 정리에 4시간 소요 → To-Be: AI 자동 취합으로 담당자 검토 시간 30분

### ✅ 제약 사항 다각도 검토 여부
기술 측면 제약만 적지 않고, 조직(인력·역량)·비용·정책·보안 측면에서도 제약이 있는지 각각 확인하고 작성했는가?
> 예) 기술: "알고리즘·GPU·데이터 품질" / 조직: "담당 인력·내부 역량" / 비용: "구축·운영 비용" / 정책·보안: "개인정보·내부 규정"
