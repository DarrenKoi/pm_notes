## 2. 과제 설계 방법론 (AX 서비스 기획 컨설팅 방법론)

### 2.1 Baseline 3원칙
| 원칙 | 의미 |
|------|------|
| 가치 중심 기획 (Value-Centric) | 기술적 화려함보다 실제 이익 같은 비즈니스 임팩트 중심 기획 |
| 데이터 중심 설계 (Data-Centric) | 추측이 아닌 사내외 가용 데이터 중심 설계 수행 |
| 인간 중심의 협업 (Human-AI Centric) | AI가 전체 업무를 대체하는 것이 아니라 사람의 능력을 확장하는 프로세스 설계 |

### 2.2 서비스 기획 Process — 4 Step

| Step | 명칭 | 한 줄 정의 | 3대 활동 |
|------|------|-----------|----------|
| **Step1. Discovery** | 기회 발굴 | 비즈니스 가치 기반의 과제 식별 | ① 비즈니스 프로세스 세분화 ② Pain Point 구조화 및 정량화 ③ AI 적합 영역 도출 |
| **Step2. Design** | 아이디어 설계 | 기술적 실현 가능성을 고려한 설계 | ① AI 기반 프로세스 재정의 ② 데이터 가용성 및 구현가능성 진단 ③ AI Agent 설계 |
| **Step3. Validation** | 가치 검증 | 투자 효용성 및 재무적 효과 산출 | ① 비즈니스 임팩트 정량화 ② 재무적 타당성 산출 ③ AI 도입 전략 구성 |
| **Step4. Execution** | 실행 전략 수립 | 성공적인 도입을 위한 로드맵 | ① AX 로드맵 수립 ② AX 의사결정 지원 ③ AX 변화관리 전략 수립 |

#### Step1. Discovery 상세
1. **비즈니스 프로세스 세분화**: MECE¹ 관점으로 상위 Level 업무를 하위 Level로 상세화. 시간 흐름/기능 순서에 따라 업무를 순차 나열, 공백·중복 제거. 예) 구매관리: 업체 조사 → 견적 비교 → 단가 승인 → 계약 체결.
   - ¹ MECE(Mutually Exclusive, Collectively Exhaustive): 중복 없이·누락 없이 분석 설계하는 프레임워크.
2. **Pain Point 구조화 및 정량화**: Issue Tree 분석으로 문제 원인을 계층적 분석(문제 현상 → 근본 원인 1·2·3). 원인 연관 데이터를 분석해 수치적으로 정량화(예: 에러로 인한 재작업률 20%, 주당 10시간 투입 등).
3. **AI 적합 영역 도출**: 비즈니스 가치 × 구현가능성 2축으로 업무 과제 분석, 각 영역별 특징 기준으로 과제 우선순위화.

#### Step2. Design 상세
1. **AI 기반 업무 프로세스 재정의**: 사용자와 AI 역할 중심으로 재설계(예: 업체 조사 → 견적 비교 → 단가 승인 → 계약 체결을 AI Agent + 사용자 + 최종승인 + 외부도구 흐름으로). **AI 기반 설계 3원칙** — ① UI 설계(요청 방식·출력 형태 정의) ② Agent Reasoning(재설계된 프로세스 기반 추론 로직) ③ Tool 활용(DB조회·웹 검색 등 기능 정의). AI 오류 차단을 위해 **AI 판단 기준 + 사람의 최종 승인**을 반드시 명시해 신뢰성 확보.
2. **데이터 가용성 및 구현 가능성 진단**: ① Format 정의(정형/비정형/반정형 + 디지털화 수준) ② 접근성 검토(저장 위치 클라우드/온프레미스, 보안 연동) ③ 데이터 거버넌스(DLM으로 최신성·업데이트 주기). "데이터 있음" 같은 모호한 답이 아니라 상세 정보 확인 — 예: "최근 3년치 구매 계약서 500건, OCR 스캔 완료, 개인정보 처리 필요".
3. **AI Agent 설계**: ① 개발범위 확인(인프라·추가 개발 요소·활용 AI 모델·사용자 접근 채널) ② **AI Prompting 설계** — Role / Context / Instruction / Constraint / Format ③ 예산 산정(개발환경 비용·AI 모델 사용료·개발 공수).

#### Step3. Validation 상세
1. **비즈니스 임팩트 정량화**: O/I(Output/Input) 관점으로 산출 — 수익성·확장성·시급성. 보수적 관점에서 '최소 기대치'를 먼저 산출하고 AI 도입에 따른 '최대 기대치'를 산출(신뢰성 확보 목적).
2. **재무적 타당성 산출**:
   - **CapEx** = AI 인프라 구축 비용 + 추가 개발 비용 + 초기 데이터 정제/학습 비용
   - **OpEx** = 클라우드 사용료 + 유지보수 인건비 + 재학습 비용 + AI 모델 사용료
   - AI Agent 설계의 CapEx·OpEx로 투자 비용 산출(사용량에 따라 상/중/하 가정).
   - **ROI = (투자 수익 − 투자 비용) / 투자 비용 × 100**
3. **AI 도입 전략 구성 — 3대 질문**:
   - 전략 방향성 일치: "이 과제가 전사 AX 전략 및 부서 KPI 달성에 어떻게 기여하는가?"
   - 재무적 타당성: "투자비 회수 기간은 언제이며, 순현재가치(NPV)는 양수인가?"
   - 도입 가능성: "현업의 변화 수용도가 충분하며, 리스크는 통제 가능한가?"

#### Step4. Execution 상세
1. **AX 로드맵 수립 — PoC→MVP→전사확산 3-Phase** (단계별 성과지표 정의·검증 후 다음 단계로 전환):
   - Phase 1 **PoC**: 핵심 가설 검증 · 2~3개월 이내 단기 수행 → 검증
   - Phase 2 **MVP**(Minimum Viable Product): 실무 적용 · 피드백 반영 · 핵심 사용자 확산 · 시스템 안정화 → 검증
   - Phase 3 **전사 확산**: 표준 인프라 통합 · 타 조직 수평 전개
2. **AX 도입 의사결정 지원**: 이해관계자를 **영향력 × 관심도** 2축으로 분석 → 핵심 관리 대상 식별(집중 관리, 만족도 유지 영역). 기술 용어가 아닌 비즈니스 중심 언어로 구조화해 의사결정 지원.
   - 영향력 高·관심도 高 = **집중 관리** / 영향력 高·관심도 低 = **만족도 유지** / 영향력 低·관심도 高 = **지속적 정보 제공** / 영향력 低·관심도 低 = **관찰**
3. **AX 변화관리 전략 수립 — 저항 요인 분석 및 대응 3종**:
   | 저항 요인 | 대응 |
   |----------|------|
   | 일자리 불안 | 단순 반복업무 대신 고부가가치 업무 전환 안내 |
   | 결과의 불신 | 설명가능한 AI 도입, 사람이 최종 검수 → 투명성/책임감 있는 AI 강조 |
   | 사용 복잡성 | 쉽게 사용 가능한 UI/UX 설계 및 밀착 교육 지원 |
   - 단계적 확산을 위한 전문 조직 육성.

### 2.3 New AI Design Camp 12-Step 단계별 구성

| # | 영문 단계 | 산출/행위 | Camp 모듈 매핑 |
|---|-----------|-----------|----------------|
| 1 | Task Identification | 업무 영역 식별 및 Level 체계화 / 부문·본부 핵심 업무 정의 | M1·M2 |
| 2 | Scope Analysis | AI 개입 최소 단위까지 분해 / AI Agent 대상 영역 분석 | M2 |
| 3 | Target Scoping | 핵심 KPI 연계 대상 영역 선정 / AI 구상도 맵핑(Optional) | M2 |
| 4 | Problem Structuring | Pain Point 구조화·도출 | M3 |
| 5 | Root-Cause Analysis | 3-Whys 기반 근본 원인 파악 | M4 |
| 6 | Disruptive Ideation | 혁신적 아이디어 도출 / 문제 해결 Ideation | M4 |
| 7 | Impact Validation | 비즈니스 임팩트 기반 적정성 검증 | M5 |
| 8 | Agent Definition | 업무 단위 Agent 역할 정의 / AI 과제 정의 | M6 |
| 9 | Task Breakdown | As-Is 일하는 방식 작성 | M7 |
| 10 | Biz-Tech Alignment | To-Be 일하는 방식 작성 | M8 |
| 11 | Task Prioritization | 실행 위한 선결 처리 항목 도출 / KSF 정의 | M8 |
| 12 | Action Planning | 현실적 일정 수립 및 액션 확정 | M9 |

> 1~7 = **AI Agent 적용 Idea 도출**, 8~12 = **AI Agent 실행 Process 구체화**.

### 2.4 단계별 Focusing Point (각 단계에서 끌어낼 것 / 보정 기준)

- **Task Identification(1)**: 목표 달성 위한 부문/본부 주요 업무 영역 가시화(Lv2 업무 영역 정의). 조직 목표와 Align해 AI 적용 대상 업무 선정 후 Process Map에 정의.
- **Scope Analysis(2)**: 업무 영역이 적정 수준(L2~3)인지 판단하고 범위 이탈 시 즉각 보정. 조별 세분화 수준 차이 나지 않게 균형 조정.
- **Target Scoping(3)**: 핵심 KPI 직결·실질 임팩트 영역 판별(전략적 중요도·투입 자원 현황·개선 임팩트). Process Map 상 해당 영역 확인 → Agent화 대상 영역의 Lv4까지 세분화. AI 개입 가능한 최소 행위 단위(L4)까지 분해하고 사람 판단 영역과 AI 대체 가능 지점 구분.
- **Problem Structuring(4)**: 선정된 AI 대상영역 Pain Point 도출·정량화. 모호한 문제의식을 '정량적 Pain Point'로 전환, 다양한 관점에서 확장·검증.
- **Root-Cause Analysis(5)**: 모든 Pain Point별 3-Whys로 '근본적 원인'까지 도달. 근본 원인을 6가지 유형에 맞춰 구조화.
- **Disruptive Ideation(6)**: 기존 프로세스를 벗어나 AI 활용 혁신적 해결 아이디어. 단순 보조를 넘어 AI Agent 활용 미래 업무상으로 전환. 시간·비용 제약에서 벗어나 도출.
- **Impact Validation(7)**: 정량 점수보다 평가 지표별 논리적 근거·비즈니스 임팩트 우선 검증. 정의 결과 공유 시 팀 간 피드백으로 과제 보완.
- **Agent Definition(8)**: AI Agent를 '기술'이 아닌 '업무 단위'로 정의. 하나의 Agent에 과도한 역할이 몰리면 분리.
- **Task Breakdown(9)**: 기존 업무 프로세스를 Task 레벨로 분해. 최소 업무 단위로 Breakdown하고 In&Out Data·연계 시스템 구체화.
- **Biz-Tech Alignment(10)**: 사람과 AI의 역할·책임 경계 명확히 설정. 실행 Task를 업무 프로세스·In&Out 데이터·AI모델로 구조화.
- **Task Prioritization(11)**: '막연한 걱정'이 아닌 실제 실행을 가로막는 구체적 Factor 도출. 제약 고려해 실행 가능성 판단, 실행 우선순위 합의.
- **Action Planning(12)**: '형식적 계획'이 아닌 현실적·구체적 일정 수립. Camp 종료 후 실제 실행으로 이어질 Action 명확히 계획.

### 2.5 Camp Reference — 모듈 진행 패턴 (Facilitation)
각 모듈은 동일 흐름으로 진행: **Module 수행(개별 작성) → 수행 결과 공유 → Peer Feedback → Facilitator Feedback**.
- Module 수행: 모듈 내용·Facilitation Guide 숙지 후 개별 작성 ("지금부터 ○분간 개인 작업을 진행해 주세요.")
- 수행 결과 공유: 그룹 내 1인당 1~2분씩 공유, 청중은 경청하며 피드백 포인트 메모 ("작성하신 내용을 한 분씩 돌아가며 자유롭게 공유해 주세요.")
- Peer Feedback: 발표자 내용에 대해 잘된 점/보완 포인트 구체적으로 공유 (1~2분) ("인상적이었던 점이나 궁금한 점을 자유롭게 나눠 주세요.")
- Facilitator Feedback: 전체 내용 종합, 핵심 인사이트·보완점 공유 ("본 모듈 내용을 짧게 정리하고 다음으로 넘어가겠습니다.")
- ★ 중간 작업 결과를 Random하게 Quick Presentation 실시.

## 4. M2. 업무 영역 정의 (업무 정의 및 Process Map 맵핑)

**수행 방법 3단계**: ① 부문/본부 업무 레벨 정의(L2~3 가시화) → ② Process Map 맵핑(주요 업무 영역 중 대상 영역 확인) → ③ AI Agent 대상영역 분석(L4까지 세분화).

**진행 TIP**: Team Building에서 합의한 대상 업무 영역에 대한 Process Breakdown, AI 적용 가능성 높은 L2·L3 정의부터. AI 구상도 준비 시 Agent Mapping, 아니면 대상업무 평가 기준에 따라 적용 가능성 높은 영역 선정.

### 4.1 [참고] 프로세스 용어와 의미
- **Process Decomposition** (= Process Hierarchy 정의): L1~L5 계층으로 Value Chain → Process Group → Process → Activity → Task로 내려가는 작업. BPM에서는 Process Landscape 설계/Process Map 작성이라 칭함.
- **Business Process Model**: Decomposition 작업을 통한 실제 프로젝트 산출물 명칭.
- **Process Map**: 업무 프로세스의 흐름·순서·관계·역할을 시각적으로 표현한 다이어그램. Decomposition이 "무엇이 있는가"라면 Process Map은 "어떻게 흐르는가". (Decomposition = 목차 / Process Map = 업무 흐름도)

| 구분 | Process Decomposition | Process Map |
|------|----------------------|-------------|
| 질문 | "무엇이 있는가?" | "어떻게 흐르는가?" |
| 형태 | 계층 트리 (L1~L5) | 흐름도 (Flow Diagram) |
| 정보 | 업무 목록·구조·범위 | 순서·분기·핸드오프·시스템·산출물 |
| 작성 시점 | 프로젝트 초기 (범위 정의) | 중기 (As-Is 분석 / To-Be 설계) |
| 주요 사용처 | ERP Scope 정의, 조직 설계 | BPR 설계, SOP 작성, RPA 자동화 |

작업 상관관계: ① Decomposition으로 전체 범위 계층화 → ② "어떤 프로세스를 다룰지" 합의·우선순위 선정해 As-Is Process Map 작성 → ③ 현황 문제점(병목·중복·낭비) 파악, 개선 반영한 To-Be Process Map 설계 → ④ 새로운 역할·시스템·흐름 정의, To-Be Map 기반 ERP Configuration·SOP·RPA 시나리오 구체화.

- **SOP** (Standard Operating Procedure): Decomposition="목차", Process Map="챕터 내용", SOP="각 페이지 실행 지침". ERP·DX 프로젝트는 이 셋을 모두 만들어야 완전한 업무 체계. 흔한 실수: Process Map 없이 SOP 바로 작성, 또는 SOP 없이 Process Map만 만들고 현장 배포.

| 구분 | ① Process Decomposition | ② Process Map | ③ SOP |
|------|------------------------|---------------|-------|
| 핵심 질문 | 무엇이 있는가? | 어떻게 흐르는가? | 정확히 어떻게 하는가? |
| 형태 | 계층 트리 | 흐름도 | 문서(텍스트+표+이미지) 단계별 지침서 |
| 수준(Granularity) | L1~L3 Process | L3 Process~L4 Activity | L4 Activity~L5 Task (실무자 행동 단위) |
| 포함 정보 | 업무명·계층·범위·정의·담당 조직 | 흐름·순서·분기·핸드오프·시스템·산출물·KPI | 세부 절차·판단 기준·사용 양식·스크린샷·예외 처리·주의사항·개정 이력 |
| 주요 사용자 | 경영진·프로젝트 팀·EA 담당자 | 프로세스 오너·IT 설계자·컨설턴트 | 실무 담당자·신입·감사인·QA |
| 작성 시점 | 프로젝트 착수(범위·구조 합의) | As-Is 분석→To-Be 설계 | To-Be 확정 후(Go-Live 직전) |
| 예시 | 공정 개발 > 단위공정 개발 > 식각공정 개발 | 식각공정 DoE 설계→실험 실행→CD-SEM 계측→결과 분석 흐름도 | "식각 DoE 실험 절차서: 장비 셋업→가스 유량 설정→데이터 저장 위치→비정상 시 조치" |

### 4.2 표준 프로세스 레벨 체계
| 레벨 | 명칭 | 설명 |
|------|------|------|
| L1 | Mega Process / Value Chain | 기업 경영의 최상위 분류 |
| L2 | Process Group / Chain | L1을 기능별 세분화한 프로세스 집합, 조직 팀/업무 영역과 대응 |
| L3 | Process | 구체적 업무 목적을 가진 실제 업무 흐름 단위, 인풋·아웃풋 명확 |
| L4 | Activity | 프로세스 구성 개별 활동, 담당자·시스템 트랜잭션 특정 |
| L5 | Task | 실무자가 실제 수행하는 최소 단위 작업 |

> 조직규모에 따라 다르나 담당 조직은 L2, 팀 조직은 L3 내외 편성. 팀 조직 내 업무는 L4~L5 수준으로 Break down.

### 4.3 도메인별 Process Level 정의 예시 (L1→L5) 및 AI 적용 기회

원본 덱은 7개 도메인 각각에 대해 ① L1~L5 계층표와 ② AI Agent 적용 기회 맵을 제공. 핵심 L3~L5 골격:

- **구매** (L1 구매): 구매 계획(수요 예측→MRP 실행: 생산계획 Data 연결·재고 리드타임 파라미터 검증·순소요량 계산 및 결과 검증) · 공급업체 관리(공급업체 전략 수립, 신규등록→사전심사: RFI 양식작성·발송, 응답서 수집·1차 스크리닝, 현장 실사 / 성과 평가) · 구매 계약(계약 조건 협상→가격 협상: 원가구조분석, 가격 제안서 검토·역제안, 최종 단가 합의·문서화 / 계약서 작성·체결→초안 작성: 표준계약 Template 선택, 협상 조건·특약 삽입, 내부 기안·결재 요청) · 구매 실행(구매 발주→PO 생성·승인: 구매요청서 확인·PO 전환, 결재선 상신·승인, 공급업체 PO 발송·수신 확인). → AI Agent#1~#4 후보.
- **영업** (L1 영업): 시장 및 고객 개발(시장 분석·목표 고객 설정→고객 세그먼트 분석: 내부 CRM Data 추출·정제, 외부 시장 Data 수집, 매력도·접근성 분석 / 리드 발굴·육성) · 영업 기회 관리(고객 니즈 파악·기회 검증→미팅 전 사전 조사, 니즈 인터뷰, 결과 CRM 입력 / 제안서·견적 작성→견적 산출·가격 승인: 표준가격 기반 견적 초안, 특별할인 승인 요청, 최종 견적서 발송, SO 생성·확인) · 납품 및 고객 서비스(납품·납기 관리→출하 지시·물류 조율: ERP 출하지시서 생성, 3PL 운송 예약·확인, 조건 대응·최종합의 문서화, 클레임·VOC 처리) · 수금 및 채권 관리(청구서 발행·대금 청구→세금 계산서 발행: 납품 완료·청구 기준 검토, ERP 청구 문서 생성, 전자세금계산서 발행·발송, 채권 회수·연체 관리). → AI Agent#1~#4.
- **연구개발** (L1 연구개발): 기술 기획·로드맵 수립(기술 트렌드·경쟁사 분석→특허·논문 조사: 검색 키워드/IPC 코드 정의, 특허 DB 검색·수집, 특허 지형도·핵심특허 분석 / R&D 과제 기획·예산 확보) · 소자 회로 설계(소자 구조 설계·T-CAD 시뮬레이션: 기준 소자 구조 모델링, 파라메터 스윕 시뮬레이션, DoE 설계·시뮬레이션 배치 실행, Id-Vg·DIBL·SS 결과 분석·최적값 선정, 레이아웃 설계·DRC/LVS 검증) · 공정 개발(단위공정 개발·최적화→식각공정 개발: 초기 레서피 설계·스크리닝 실험, DoE 실험 설계·본 실험 실행, 최적 레시피 선정·재현성 확인, 공정 통합·모듈 검증) · 수율 향상·양산 이관(수율 분석·불량 메커니즘 분석→Wafer Bin Map 분석: YMS 데이터 추출·Bin 정의 확인, 공간 패턴 분석, 공정 파라미터 상관 분석, 양산 이관). → AI Agent#1~#2.
- **생산** (L1 생산): 생산 계획·일정 관리(수요 기반 Wafer Out 계획→Capa.분석·병목공정 파악: Tool 그룹 가용 Capa. 산출, 공정 부하 계산·Over/Under 파악, 병목 해소 대안·WO 계획 조정 / 긴급 오더·변경 관리) · 전공정(리소그래피 공정→노광 공정 실행: APC 보정값 반영 레시피 확인, 웨이퍼 로딩·노광 실행, 인라인 CD-SEM·Overlay 계측 / 식각·증착·CMP 공정 실행) · 후공정·패키징(Wafer 레벨 테스트→프로브 테스트·빈 분류: 프로브 카드 장착·컨택 확인, ATE 테스트 프로그램 로딩·검증, Wafer 테스트 실행·빈맵 파일 생성 / 다이싱·본딩·몰딩) · 품질 검사·출하(최종 전기 검사→온도 스트레스 테스트: 핸들러 온도 프로파일 셋업, Cold/Hot 조건 테스트 실행·종합 판정 / 출하 승인·고객 납품). → AI Agent#1~#3.
- **물류** (L1 물류): 인바운드(원자재 수입 통관: 수입 신고서 작성·세관 제출(HS 품목·관세율 조회, 과세 가격 산출, 세관 제출·허가 수령) / 국산 부품·소재 입고: 납품 사전 예약·입고 슬롯, 수령·수량/외관 검수(거래명세서/PO 대조), 국산품질 입고검사(IQC), ERP 입고 전표 생성·재고 반영, 반품·불량 협력사 관리) · 창고·재고 관리(자재 출고·공정 라인 공급: MES 자재요청 수신·피킹 지시(EWM 전송 확인), FIFO 재고위치 조회·피킹리스트 생성, RF 단말기 피킹·수량 확인, 순환 재고 조사) · 아웃바운드(고객요청 포장·출하 준비: 포장 실행·라벨 발행(사양 DB 조회·설비 셋업, 수량 자동 계수, 라벨 발행·부착 이력 등록), 수출 통관·운송 관리) · 공급망 리스크 관리(리스크 식별·평가: 유형별 식별(지정학/규제·단일 공급업체·자연재해), 정량 평가, 대응 계획 수립, 모니터링·조기 경보). → AI Agent#1~#3.
- **재무회계** (L1 재무 회계): 예산 관리(연간 사업계획 예산 편성: 가이드라인 배포, 부문별 예산 요청서 수집·1차 검토, 부문 협의·조정 확정, 실적 분석·예산대비 보고) · 원가 관리(공정별 제조원가 집계: 웨이퍼 투입 재료비 집계(MES 자재 불출 ERP 인터페이스), 표준원가 vs 실제원가 차이분석) · 결산·재무 보고(월말 결산 마감: 감가상각비 계산·전표 생성, 자산 마스터 정합성 확인, SAP AA 감가상각 Batch 실행) · 세무 관리(법인세 신고·세액공제: R&D 비용 계정 분류, 세무조정 계산서·세액 추정치 / 투자 특례 세액공제: 신규 Fab 설비 자산 분류, 세액공제액 산출·신청서 / 글로벌 이전가격·국제 조세: 정상가격 산출, BEPS·로컬 파일, 국외 원천소득·원천징수: 로열티 원천징수세율 검토, 외국납부세액공제 / 부가가치세·통관세무: 수입 관세·부가세 처리, 고가 장비 관세 감면, 수입 세금계산서 매입세액 공제, 수출 영세율 신고, 부가세 조기 환급) · 자금 관리(투자자금 조달·집행: Fab 증설 자금 조달 계획, CAPEX 집행·대금 지급 / 글로벌 자금 유동성: Cash Pooling 운영, Cash Flow 예측 / 외환·금리 리스크: 환리스크 포지션·헤지, 금리 변동 리스크). → AI Agent#1~#3.
- **인사** (L1 인사): 인력 계획·채용(인력 수요 예측·채용 계획: 인력 현황 분석·갭 파악(SAP 인력 데이터 추출·정제, 직무별 역량 갭 매트릭스, 채용 우선순위·방법 결정) / 채용 실행: 외부 공고·Q&A 대응(공고문 작성), 지원자 평가·면접(1·2차 스크리닝, 면접 질의서, 면접 실행)) · 인재 육성·교육(직무 교육 과정 설계·운영: 교육 수요조사·과정 개발(수요 설문, 과정 설계, 등록·수강 신청 오픈)) · 성과관리·보상(목표 설정·성과 평가: 조직/개인 KPI 설정, 성과 모니터링·피드백, 다면 평가·1:1 리뷰, 역량 평가·등급 보정 / 연봉 계약·인센티브: 임금 벤치마킹·재원 수립, 등급별 기본급 인상 매핑, 단기/장기 인센티브) · 급여·노무 관리(급여 계산·4대보험: 근태 마감·검증, 고정급·수당 원천징수, 4대보험 취득/상실 신고·정산, 급여 이체·명세서 발급 / 복리후생·근로환경, 노사 관계·리스크) · 퇴직·인력 전환(퇴직 프로세스, 사내 직무 전환·재배치, 전직 지원·아웃플레이스먼트). → AI Agent#1~#3.

### 4.4 M2 수행 방법 상세 (업무 정의 + Agent 대상영역 분석)
1. 우리 조직(부문/본부)의 L2 업무 영역과 하위 L3 업무 활동을 시각화 — Process Level Rule에 맞게 정의(L2 핵심 업무 영역, L3 업무 활동). TIP: "조직 주요 업무는 무엇인가?" 확인, 세분화 수준 다른 항목 동일 수준 조정.
2. AI 구상도 준비 시 Agent가 조직 핵심 업무 영역 중 어디 포함되는지 Check.
3. 구상도 미준비 시 후보 영역 대상 핵심 질문 Double Check:
   | 기준 | 핵심 질문 |
   |------|----------|
   | 전략적 중요도 | 핵심 성과에 직접 기여하는가? 이 영역 성과 저하 시 부서/회사 타격은? |
   | 투입 자원 현황 | 인력·시간이 과도 투입되지 않는가? 반복/단순 작업 비율은? |
   | 개선 임팩트 | AI/자동화로 개선 시 효과는? 처리 속도·정확도·비용 절감·품질 개선 측면 효과 큰가? |

**AI Agent 대상영역 분석 3단계**: ① 후보 Agent 영역을 L4 단계로 세분화 → ② '반드시' 사람 개입이 필요한 일을 제외하고 AI 적용 가능 업무에 로봇 마킹 → ③ 사람 개입 필수 지점 제외한 나머지 전체가 'AI Agent 영역'. TIP: 정의된 AI Agent 영역은 기본 모두 Agent화 가능 → M3 후 Focusing.

**Check & Review (M2)** — 업무 LVL 정의·권장 포스트잇:
| LVL | 정의 | 권장 포스트잇(조별) |
|-----|------|---------------------|
| L1 | 조직이 가치를 창출하는 최상위 업무 집합 (본부/부문 단위 거시적 정체성) | 비즈니스 도메인 1개 |
| L2 | L1 달성 위해 독립 운영되는 주요 직무 기능 (담당자별 R&R 단위) | 핵심 업무 영역 3~5개 |
| L3 | 업무 영역 내 특정 산출물(Output)을 만들기 위한 과업 묶음 (주간 보고서 기재 단위) | 각 L2별 3~5개 |
| L4 | 최소 단위 실행 단계 ('동사' 중심 구체적 업무) | 각 L3별 3~5개 |

가이드 핵심:
- **PLAN·DO·SEE 관점**으로 업무 영역 빠짐없이 작성 — PLAN(계획·기획·목표 설정), DO(실행·운영·수행), SEE(모니터링·점검·피드백·보고) 중 빠진 영역 없는가?
- L4 업무를 "무엇을 어떻게 하는지"까지 구체적으로 기술 — L4 하나 읽었을 때 그 안에 할 일이 두 개 이상 보이지 않는가?(두 개 이상이면 L3 수준). 예) 산업/경쟁사 동향 분석(L3) → 경쟁사 목록 선정 / 기술 로드맵 자료 수집 / 항목별 비교표 작성 / 시사점 정리(각 L4).
- AI·사람 역할 경계 명확히 — L4 중 최종 결과 책임/윤리적 판단 필요 단계에 반드시 사람이 배치되었는가? 예) 협력사 선정: AI는 납기율·품질 점수 등 평가 데이터 산출까지, 최종 계약 결정은 구매 담당자 승인.

**목표 산출물**: L2~3 업무 정의 20분 + Agent 대상영역 분석 20분. 예시 산출물(p60)은 마케팅 「산업/경쟁사 분석 · 고객사 수요/기술 트렌드 분석 · 외부 데이터 통합 · 수요 예측(모델링)」 4영역을 L4까지 분해하고 '고객 협력 Agent' 등 기회를 도출한 형태.

## 5. M3. Pain Point 도출 및 정량화

**수행 방법 4단계**:
1. AI Agent 발굴 대상영역을 새로운 위치로 옮긴다(전략적 중요도·투입 자원 현황·개선 임팩트 관점 핵심 영역 선정).
2. L4 각 업무별 반복 발생하는 큰 어려움을 정의해 업무 옆에 이어서 기술.
3. 구체적 수치/빈도를 함께 기록(예: 부서별 요구사항 수시 변경, 월 평균 3회). 6관점으로 탐색:
   - 속도·지연(예: 승인까지 평균 5일) / 품질·정확성(편차) / 반복·비효율(동일 데이터 여러 번 입력) / 협업·커뮤니케이션 / 데이터·시스템(제때·정확히 안 나옴) / 규정·정책(복잡·모호)
   - 정량화 예: 채널 10곳 이상 × 1건별 10회 × 20건 이상 = 10×20 = 200회
4. 가장 Critical한 지점에 빨간 Check Mark(●). 집중 기준:
   | 기준 | 설명 |
   |------|------|
   | 발생 빈도 | 매일/매주 반복 발생 |
   | 영향 범위 | 여러 팀원·다른 부서에 영향 |
   | 시간 소모 | 상당한 시간/지연 야기 |
   | 품질 영향 | 실수·오류로 재작업 필요 |
   - TIP: Agent 대상 영역 중 Pain Point 많은 영역 중심으로 선택과 집중.

**[참고] Pain Point 도출 원리**: 문제 상황 공감 → 구조적 이해 → 실행 가능 수준으로 정의.
- **판단 오류 3유형**: ① 표현상의 니즈를 문제로 인식 → 비즈니스 '손실' 관점에서 팩트 파악 ② Pain 단위를 크게 잡음 → 측정/관찰 가능한 업무 행동 수준으로 표현 ③ 증상만 보고 처방 → 문제 본질 정의 없이 기술로 즉시 번역하지 않도록 주의.
- **도출 Points 6관점**: Pain & Needs(현장 호소) / 발생 빈도(언제·얼마나 자주) / 프로세스 단계(어떤 흐름에서 반복) / 영향 대상(누가 가장 영향) / 정량·정성적 근거 / 문제 영역(유형: 기술·조직·프로세스 등).

**Check & Review (M3)**:
| 구분 | 정의 | 진행 방식 | 가이드 핵심 |
|------|------|----------|-------------|
| Pain Point | 업무 프로세스 내 반복 발생·조직에 실질 손실 초래하는 문제 | 각 L4당 3개 이내 | 행동+수치 기반 작성 |
| 정량화 | 발생 빈도·소요 시간·영향 범위를 수치로 표현 | 가능한 모든 PP 정량화 | 예) 시간 "하루 1~2시간", 빈도 "월 1~2회", 영향 "한 번 지연 시 관련 부서 3곳 업무 중단" |
| Critical Pain Point | 비즈니스 임팩트 가장 크고 우선 해결 필요한 핵심 문제 | 조별 투표(빨간 스티커, 인당 3개) | 빈도 × 영향도 관점 선택 |

가이드 핵심:
- Critical PP 선별: 발생 빈도·영향 범위·시간 소모·품질 영향 중 1개 이상 기준으로 왜 Critical한지 설명 가능한가?
- 팀·부문 관점: '나만 불편한 것'인가, '반복될수록 팀·부문에 실제 손실(시간·비용·품질)이 쌓이는 것'인가? 예) "내 툴 UI 불편"(X) → "부서별 양식 불일치로 매주 팀 10시간 소요"(O)
- 정량화: 모호한 표현 없이 누구나 심각성 이해 가능한 수치로? 예) "자주 오류로 지연"(X) → "월 평균 3회 오류, 1회당 4시간 소요"(O) / "월 2회로 빈번하진 않지만 1회 발생 시 라인 전체 4시간 중단으로 손실 큼"(O)

**최종 산출물 예시 (마케팅 산업/경쟁사 분석)**: 글로벌 반도체 산업동향 수집 / 주요 경쟁사 기술 로드맵·공급능력 분석 / 주요 이슈 브리핑·내부보고 등 L4 업무별 Pain Point — 산업 뉴스·리포트 수집 수작업 과다 · 고객사별 정보 접근 제한 · 브리핑 주체/주기 불명확 · 제품 카테고리별 비교 포맷 부재 · 공개 자료만으로 Fab 생산능력 정보 불명확 · 조직별 보고 양식 상이·출처 다름 · 이슈 발생 시 긴급 보고서 요청 빈번 · 기관별 점유율 산식 상이 · 리포트/데이터 출처별 기준 불일치. (Critical 1~3개 선정)

**목표 산출물**: 20분 / Pain Point 정량화표(Lv2~Lv4 + Pain Point + Critical dot).

## 6. M4. 근본 원인 분석 및 문제 해결 Ideation

**수행 방법 4단계**:
1. 현상이 일어나는 이유(1st Why) 작성.
2. 그 이유가 발생하는 이유(2nd Why)를 다른 Column에 작성.
3. 두번째 이유가 발생하는 근본 이유(3rd Why)를 또 다른 Column에 작성.
   - 흐름: Pain Point → Why1 → Why2 → Why3 → 가설적 근본 원인. 예) 서류 포맷 불일치 → "표준 양식 없음" → "시스템 미통합" → "각 부서 독자 운영 관행".
4. 3단계 Why에 **6가지 시각** 추가해 근본 원인 기술. (프로세스·시스템 외 외부요인·사람 등으로 확대 권장)

**6가지 관점의 근본 원인 분석**:
| 유형 | 설명 | 예시 |
|------|------|------|
| 프로세스 | 절차·표준화·승인 흐름 등 업무 수행 과정의 구조적 문제 | 승인 절차 길어 처리 지연 / 역할·책임 불명확 |
| 시스템 | 기능 부재·통합 미비·자동화 부족 등 기술적 구조 한계 | 시스템 간 데이터 연동 불가 / 알림·자동화 부재로 누락 |
| 데이터 | 정확성·표준화·접근성 부족 등 데이터 품질·관리 문제 | 부서별 수집 기준 불일치 / 실시간 조회 불가 |
| 사람 | 역량·인식·책임 부재 등 인적 요인 | 담당자 숙련도 부족 / 역할 혼선으로 의사결정 지연 |
| 정책·거버넌스 | 제도·규정·의사결정 체계·리더십 운영상 문제 | 규정-현실 상충 예외처리 다수 / KPI 기준 불명확 |
| 외부요인 | 시장·법규·환경·공급망 등 외부 변화 영향 | 정부 정책 변경으로 절차 추가 / 외부 협력사 시스템 변경으로 지연 |

**문제 해결 Ideation**: 근본 원인 옆에 해결 아이디어 기재(한 유형 또는 여러 유형 결합). **해결 아이디어 6유형**:
① 시스템(신규 구축·기능 추가·통합) ② 프로세스(표준화·승인 절차 변경·병목 제거) ③ 사람(교육·훈련·역량 강화) ④ 데이터(표준 포맷 정의·품질 관리) ⑤ 정책·규정(제도 개선·기준 명확화) ⑥ 해결 불가/유지(외부 규제·환경으로 개선 불가 → 장기 과제 관리).
- 완성 구조: Pain Point > 근본 원인 > 해결 방안의 논리 구조 완결(WHY?→WHY?→WHY?→해결 아이디어). 실습은 시간 감안해 Pain Point 연계 해결 Ideation까지 완결.
- 예: "공고문 표준안·검수 절차 부재 → 채용 시스템 내 표준 템플릿 탑재 + 게시 전 AI 문장 검수 기능".

**Check & Review (M4)**:
| 구분 | 정의 | 권장 포스트잇 | 가이드 핵심 |
|------|------|---------------|-------------|
| 현상(Pain Point) | 해결 필요한 핵심 고충 | 붉은 스티커 1개 이상, PP 3~5개 | Module 3에서 선정한 Critical 문제 |
| 3-Whys | 현상 이유를 추적하는 과정 | 각 Why 단계별 3개(분홍) | "왜?" 3회 반복으로 표면적 이유 너머 탐색 |
| 근본 원인 | 문제의 본질적 이유 | 최종 1개(분홍) | 6개 유형(시스템·데이터·정책·외부·사람·프로세스) 중 식별 |
| 해결 아이디어 | 원인 제거 위한 (AI) 방안 | 원인별 2~4개(노랑) | 기존 방식 넘어 AI 기술 패턴 활용 혁신적 대안 |

가이드 핵심:
- 3-Why 논리 인과관계: 각 답변을 앞뒤로 '왜냐하면'으로 자연스럽게 연결되는가? 예) 보고서 취합 오래 걸림 → "양식 달라 수동 변환 필요" → ("시스템 노후화"는 비약 X) → "양식 표준화 기준 자체가 없음"(O).
- 발생 원인 규명: 근본 원인이 Pain Point를 다른 말로 바꾼 수준이 아니라 6개 유형 중 하나로 명확히 분류되었는가?
- 다양한 해결 아이디어: 하나의 근본 원인에 2개 이상 관점(시스템·프로세스·사람·데이터)에서 고민했는가? 예) 근본 원인 "리포팅 가이드라인·품질관리 체계 부재" → "표준 템플릿 작성"만(X) → 표준 템플릿(Process) + 산업이슈·경쟁사 동향 AI 요약 자동화(Infra) + 품질관리 체계 확립(Org.)(O).

**최종 산출물 예시 (마케팅 산업/경쟁사 분석)**:
| Pain Point | 근본 원인 | 해결 아이디어 |
|------------|-----------|---------------|
| 데이터 출처별 기준 불일치로 내부 해석 혼선 | 내부 통합 표준 없음 / 데이터 관리 주체 부재·단일 참조 기준 미정의 | 단일 참조 기준 제정 + Agent 기반 자동 정합화 / 주요 기관별 단위·정의 Mapping Table 작성 |
| 경쟁사 동향·투자 정보 수집의 수작업 의존 | 소스 분산 but 자동 수집 체계 없음 / 비정형 데이터 개별 탐색·정리 | Market Watcher TF·공유 프로세스 설계 / 자동 스크랩·이슈 트래킹 시스템 구축 (Keyword 기반 Alert) |
| 리포트 업데이트·최신성 관리 부재 | 리포트 버전/수집일 등 메타데이터 없어 최신 리포트 식별 불가 / 콘텐츠 저장 관리체계 미흡 | 리포트 메타데이터 관리 시스템(수집일·출처·버전 자동 태깅) ("Latest Only" 필터로 최신판 자동 표시) |
| 내부 공유 포맷·요약 기준 일관성 부족 | 담당자마다 요약 스타일·분류기준 달라 브리핑 품질 편차 / 리포팅 가이드라인·품질관리 체계 부재 | 리포팅 표준 템플릿·품질관리 체계 확립 / 산업이슈·경쟁사 동향 AI 요약 자동화 |

**목표 산출물**: 20분 / Pain Point → 근본 원인 → 해결 아이디어표.

- **데이터 준비도** (Q7: 데이터 충분히 준비/준비 계획 구체적·실현 가능?)
  - 3점: 시스템 내 데이터 표준화 포함, 체계적 적재/정제 / 2점: 산발적 존재나 추가 정제/표준화 필요 / 1점: 데이터 없거나 품질(오류/중복/비정형) 낮음

- 평가 기준별 1·2·3 중 점수 택일 후 조별 논의로 조정.
- 1점 항목 존재해도 1순위 선정 이유와 극복 방안 설명 가능한가?(추후 로드맵 반영) 예) 데이터 없어 실행 용이성 1점이나 전략적 중요도·시급성 높아 1순위 → 데이터 확보는 1Q 내 IT팀 협의로 해결 예정.
- 조원 간 점수 차이 큰(2점 차) 항목의 원인 파악·조정해 하나의 점수로 합의했는가? 예) 전략적 중요도 A=3·B=1 → "단기 성과보다 중장기 전략 연계 관점이 맞다" 합의해 2점 조정.

**목표 산출물**: 20분 / 평가 결과 기반 영역·범위 조정 검토 → 후보 AI 과제 확정(필요시 범위 변경 후 재평가).

## 8. Day1 Wrap up — Quiz & Facilitation Guide

### 8.1 Quiz (정답: 모두 D… 원문 표기상 정답 마킹 'D'가 보기 A/B/C 외 — 실제 의도는 균형적 선택지) — 상황별 적절한 선택
> 원문에 각 문항 상단 "D" 마킹이 있으나 보기는 A/B/C. 출제 의도는 "정량화 강요/생략 양극단이 아닌 최소 정량화", "사람 탓 배제가 아닌 구조적 확장", "확장 아이디어 삭제가 아닌 단·중장기 구분"인 균형적 답.

1. **Pain Point 정량화 없이 다음 단계로 넘어가려는 경우** (대부분 "비효율적"·"번거롭다" 수준, "정확한 수치 모름")
   - A. 범주 수준 추정값(예: 월 1~2회·약 1시간)으로 최소한의 정량화 유도 ← 권장
   - B. 정량화 안 된 PP는 의미 없으므로 삭제·재작성
   - C. 방향성 중요하니 정량화 생략, 이후 보완
2. **원인이 자꾸 "사람 탓"으로만 흘러가는 경우** (담당자 실수·옆 부서 비협조 등)
   - A. 사람 중심 원인 유지, 실행 단계에서 개선
   - B. 사람 중심 원인을 출발점으로 "왜 이 문제가 반복되는가" 재질문하며 구조적 원인으로 확장 ← 권장
   - C. 사람 원인 제외, 시스템·데이터 관점으로만 재작성
3. **아이디어가 점점 커져 감당할 수 없는 범위가 된 경우** ("본부 전체 자동화"·"타 조직 확장"까지)
   - A. 확장 아이디어 유지, 로드맵에서 우선순위 정리
   - B. 현재 범위 명확히 제한, 확장 아이디어 제외하고 재정의
   - C. 지금 해결할 범위(핵심 PP)와 이후 다룰 확장 아이디어(중장기)를 구분해 각각 정리 ← 권장

### 8.2 Facilitation Guide — 상황별 Don't/Do
| 상황 | Don't | Do (Resolution) |
|------|-------|-----------------|
| Agent 정의가 누구는 한 줄, 누구는 꽉 참 | 산출물 비교 | 사고 흐름 — 완성도보다 생각의 방향. 왜 이렇게 정리했는지 설명 가능하면 충분 |
| 진도의 차이 (A는 다음 모듈까지, B는 문제 정의 중) | 속도 통제 | 목표 최소화·확장 — 빠른 진도는 리스크 점검, 늦은 진도는 PP 하나만 명확해도 충분 |
| "Agent 이름 뭘로?" 질문 | 답 찾기 | 사고 유도 — "이 Agent 하는 일을 한 문장으로?", "없으면 제일 불편한 사람은?", "사람이 꼭 해야 하는 일은 어디까지?" |
| 아이디어 정체·머뭇거림 | 압박 | 심리적 안전 — 정확할 필요 없음, 가설도 OK, 평가 자리 아님 |
| 이해 수준 차이 (AI 용어 익숙/처음) | 비교 | 정상화 — AI 경험 다른 게 정상, 질문 나오는 게 캠프 잘 가는 신호 |

## 10. M7. As-Is 업무 프로세스 정의

**수행 방법**: ① 현재 As-Is 일하는 모습 생각 → ② Input과 Activity > Task를 구분해 프로세스 흐름이 드러나도록 작성 → ③ Agent 추진 시 예상 제약사항 작성.
- 작성 TIP: 각 Task 간 연결 흐름·분기 조건을 화살표로 표현 / 반복·되돌아오는 흐름(반려·재검토) 빠짐없이 / 활용 데이터·시스템·Task 수행 주체를 각 Task에 표현.
- 진행 TIP: "지금 실제로 어떻게 하고 있나요?"로 시작, 이상적 모습 아닌 현실 그대로 / 데이터 접근 권한·시스템 연동·보안 이슈 등 현실적 제약 구체적으로.

**[참고] Process Modeling Rule** (IDEF0형 Task 모델):
- **Task** = 하나의 명확한 Input·Output을 가지는 최소 실행 단위 작업 단계
- **Input** = 선행 단계/외부에서 제공되는 데이터·문서·요청
- **Output** = 업무 결과로 생성되는 결과물·보고서·시스템 반영 값·의사결정 결과
- **Constraint** = Task 수행 조건·규칙·가이드라인·법규 등. Input과 달리 변하지 않고 업무 방향을 결정
- **Mechanism** = Task 수행 주체(담당자)나 인프라(시스템·설비)
- 최소 실행 단위에서 Input·Output·시스템(AI Agent)·제약조건 설계. AI Agent는 기본 L4·L5 최소 실행 단위 자동화 → 단, Function call·API·MCP 활용해 상위 레벨 자동화로 확대.

**BPMN Notation 예시**:
| 구분 | 표시 | 내용 |
|------|------|------|
| Event | 원(단선) | 프로세스 시작 지점 |
| | 원(굵은선) | 프로세스 완료 지점 |
| | 원(이중선) | 진행 중 발생 사건 (예: 고객 수신 대기) |
| Activities | 직사각형 | Task (단일 작업 단위) |
| | + 기호 추가 | Sub Process (복합 활동) |
| Gateways | 마름모 | 의사결정/흐름 분기·합류 (o·+·x로 배타적·병렬적·포괄적 선택 표기) |
| Connecting Object | 실선 화살표 | 작업 진행 순서 |
| | 점선 화살표 | 다른 객체와 주고받는 메시지·정보 흐름 |
| | 점선 | 특정 활동에 연결 (텍스트 주석·데이터) |

**As-Is 정의서 Template**: 선행 Work/Event → Activity Flow → 후행 Work/Event + 제약 사항. 상세표 컬럼: ID / Activity 명 / Activity 설명 / 기준 유무 / 업무 주기 / 소요시간(TAT) / System 명 / Menu 명.

|------|------|-------------|
| 업무/프로세스 | AI 적용 업무 흐름·프로세스 정의 | AI 도입 위해 선행될 업무 기준·표준화 Task 먼저 작성 |
| 데이터 | AI 학습·판단 위한 데이터 환경 구축 | 수집·적재·표준화 순서로 구성, 데이터 소스·규모 구체 명시 |
| AI 모델 | LLM/AI 알고리즘 설계·학습·운영화 | 알고리즘 설계부터 시스템 연동까지 순서대로 |
| Agentic AI Prototype | 구현된 Agent가 실제 업무에서 동작하는 형태 | 단일 Agent 완성 후 타 Agent와 연계·확장 가능성까지 고려 |

> 실행 계획서 = Module 8 To-Be Process & Task 정의서 참고, 위 4구분으로 Task 분류, 분기별 흐름·연결관계를 화살표로 표시.

가이드 핵심:
- 선결 Task의 로드맵 앞단 배치: 각 Task 시작 전 먼저 완료될 Task(데이터 수집·가공·보안)가 앞단에 배치? 예) AI 모델 학습 only(X) → 데이터 수집·적재 → Algorithm 표준화 → AI 모델 학습(O).
- Task 배치 순서 적절성: 동시 진행 가능 Task인데 굳이 순서 나눠 일정 늘어나지 않았나? 예) "보안 검토"와 "Agent 개발" 순차(X) → 의존관계 없으니 병행으로 단축.
- 당장 해결 불가 제약 처리: 현재 쓸 수 있는 데이터·시스템 범위로 좁혀 올해 안에 완성 가능 수준으로? 예) "전사 데이터 플랫폼 구축 후 시작"(X) → "현재 쓸 수 있는 데이터 1~2종으로 먼저 시작, 나머지는 제약사항 칸에 명시"(O).

## 13. Day2 Wrap up

### 13.1 Process 관점에서 AI Agent (Decomposition / Map / SOP × Agentic AI)
> Decomposition으로 "어디에" Agent를 배치할지, Process Map으로 "어떻게" 흐를지, SOP로 "무엇을" 정확히 실행할지 정의.

| 관점 | Agent의 정의 | Agent 설계에 주는 것 | 이것 없이 Agent를 만들면 |
|------|--------------|----------------------|---------------------------|
| Decomposition | 업무 계층의 실행 노드 | Agent의 범위·레벨·계층 구조 (어디에 배치) | 경계 불명확 → 중복·충돌·누락 |
| Process Map | 흐름 위의 지능형 행위자 | Agent의 트리거·흐름·분기·핸드오프 (어떻게 움직이나) | 언제 시작·종료·누구에게 넘길지 모름 |
| SOP | 절차서의 실행자 | Agent의 행동 규칙·판단 기준·안전망 (무엇을 어떻게) | 스스로 규칙 만들어 컴플라이언스 위반 |

**Agentic AI 3관점 매핑**:
- SOP의 절차 순서·판단 기준 → **LLM 행동 지침**
- SOP의 "시스템 조작" → **LLM Tool & Function call**
- SOP 예외처리·제한의 구현 → **LLM Guardrail & Policy**

### 13.2 현재 성과와 향후 작업
- **What We did**: 각자 업무 영역에서 AI Agent 도입 기회를 정의 / AI Agent 구체화(AI 과제 정의서·To-Be Process Map)
- **What We didn't**: 전체 AI Agent 우선순위 반영 Short List(AI 포트폴리오) / 개발 계획(Phased Plan·Resource) / Quick win 또는 Phase 1 대상 서비스 기본 설계

### 13.3 포트폴리오 — Long List → Short List → Phased Plan 우선순위 기준
| 대분류 | 기준 | 질문 |
|--------|------|------|
| 비즈니스 가치 | 전략적 정렬 | 회사 단기/장기 목표(KPI)·핵심 전략과 일치하는가? |
| | 재무적 수익 | 매출 증대·비용 절감·생산성 향상 효과가 명확한가? |
| | 고객/사용자 경험 | 고객 Pain Point를 해결하거나 편의성을 개선하는가? |
| | 시장 경쟁력 | 경쟁사 대비 차별 우위·선점 효과가 있는가? |
| 실행 가능성 | 데이터 준비도 | 모델 학습·추론에 필요한 양질 데이터가 확보되었는가? |
| | 기술적 난이도 | 현재 AI 기술로 구현 가능한가? |
| | 리소스 가용성 | 내부 인력/외부 파트너십으로 수행 가능한가? |
| | 확장성 | PoC 이후 실제 운영 환경 배포·확장이 용이한가? |
| 시급도 | 업무 요구 | 비즈니스 니즈에 따른 긴급 요구인가? |
| | 선행 조건 | 후행 과제 수행 위한 필수 선행 조건인가? |
| 리스크 및 규제 | 윤리 및 보안 | 개인정보 보호·편향성·모델 투명성 이슈 없는가? |
| | 법적 규제 | 산업별 특수 규제·AI 관련 법안을 준수하는가? |
