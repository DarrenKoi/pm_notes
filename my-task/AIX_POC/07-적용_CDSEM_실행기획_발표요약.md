## 장표 2 — STEP 01 · 전략 일관성 (조직목표 → AI후보 Top-Down)

과제가 개인 흥미가 아니라 **조직 목표에서 위→아래로 연역**된 결과임을 1장으로 증명한다.

| L0 조직 목표 | L1 업무 목표 | L2 관련(핵심) 업무 | L3 AI 후보 과제 |
|---|---|---|---|
| **R&D 디바이스 개발 지연 최소화** | ① 장비 운영/관리 최적화<br>② 검/계측 효율화 | ① MI Recipe Solution 제공<br>② 측정 장비 간 Skew Zero<br>③ MI 장비 가동률 극대화 | **Auto Recipe Creation**<br>(1차 PoC = Align 자동 보정) |

> **판정**: L3 과제(Auto Recipe Creation)가 핵심업무 3종→L1→L0로 끊김 없이 거슬러 올라감 → **전략적으로 정합**. 후보 과제는 하나이며, **하나의 VLM Agent**(②~⑥ 연속)가 수행하되 ⑥(Align 자동 보정)을 1차 PoC로 둔다.

> 🎤 **발표 스크립트**
> "첫 단계는 추적성입니다. 이 과제는 'R&D 디바이스 개발 지연 최소화'라는 조직 목표에서 연역됐고, 일관 기준(Skew Zero)과 24h 무중단(가동률↑)으로 장비 운영/관리·검계측을 최적화합니다."

## 장표 3 — STEP 02 · 대상 업무영역 (L2×L3 + AI 대체 판정)

계측(MI) 운영을 L2 4분할로 펼치고 각 L3에 AI 대체 판정을 오버레이 → **1차 대상을 좁힌다**. (L2 4분할 = 엔지니어 확정 2026-06-24)

| L2 영역 | 대표 L3 | 판정 | 결정 |
|---|---|:---:|---|
| **A. Recipe Management ★** | ② **Recipe Creation** (셋업 ②~⑥) | 🤖 | **1차 PoC 대상** — 반복·정형·공수 최대 + SEM 데이터 가용 |
| B. 측정 실행 | ② 자동 측정 Run | ⚙️ | 이미 자동화 — 범위 밖 |
| C. 데이터·모니터링 | ② 안정성 모니터링 ◇ | 🤝 | **차기(2차) 대상** — 운영 공백 |
| D. 장비 유지보수 | ① PM · ③ 캘리브레이션 | 🧑 | 물리 작업 — 범위 밖 |

> 범례 🤖 AI대체 · 🤝 부분(HITL) · 🧑 사람 · ⚙️ 기존 자동 // **모든 칸에 AI를 넣지 않는다** — 반복성·정형성·공수·데이터 가용성이 큰 지점만 선별.

> 🎤 **발표 스크립트**
> "AI를 모든 업무에 넣지 않습니다. 4개 영역 중 반복성·공수가 최대이고 데이터가 이미 있는 Recipe Creation을 1차로, 야간/주말 공백이 큰 안정성 모니터링을 차기로 정했습니다."

## 장표 4 — STEP 03 · Pain Point 정량화 (L4 ①~⑦)

1차 대상 Recipe Creation을 As-Is 셋업 ①~⑦(L4)로 분해하고, **빈도·심각성·주요성 3축**으로 점수화. (점수는 τ placeholder, ⑥만 가설 5/5/5)

| L4 활동 | Pain Point | 빈 | 심 | 주 |
|---|---|:-:|:-:|:-:|
| ③ Align 등록 *(병목)* | 패턴 인식 템플릿 수동·숙련 의존 | τ | τ | τ |
| ④ 좌표·박스 *(병목)* | EP 좌표·측정박스 수동·숙련 의존 | τ | τ | τ |
| **⑥ 시험측정→오인식 수정** | 좌표 shift·align 실패·오인식 → 수정 **반복** (**최대 병목**) | **5** | **5** | **5** |
| ⑦ 저장·검증 | 검증 누락 시 오류가 양산 Run으로 전파 | τ | τ | τ |

> **무게중심 = ⑥**. 셋업마다 거의 매번 발생(빈도)하고 오측·재셋업으로 직결되며(심각성) 리드타임의 지배적 비중(주요성). 이 점수가 #05 적정성·#10 기대효과의 백데이터.

> 🎤 **발표 스크립트**
> "어느 단계를 자동화할지는 감이 아니라 점수로 정합니다. ①~⑦ 전 스텝을 같은 자로 재면 ⑥ 오인식 수정 반복이 최대 병목으로 떠오릅니다."

## 장표 5 — STEP 04 · 근본원인 (3-Why) → 해결 아이디어

⑥ 최대 병목을 "왜?"로 세 번 파고들면 셋업 병목이 **두 근본원인으로 수렴**한다.

```
⑥ 오인식 수정 반복
 └ 왜? 미리 만든 템플릿·좌표가 실측 시 안 맞는다
    └ 왜? 실측 시작 시 전체 좌표가 shift / align 실패
       └ 왜? (a) 패턴 인식 실패   (b) 측정박스·좌표 오류
```

| Pain (L4) | 해결 아이디어 | 수렴 |
|---|---|:---:|
| ⑥ 오인식 수정 반복 | VLM이 SEM 이미지로 **실시간 좌표 재정합·오인식 자동 수정** | **(a)(b)** |
| ③④ 패턴·좌표 설정 | VLM이 일관 기준으로 등록 (⑥과 **같은 Agent**) | **(a)(b)** |

> 셋업 병목 해결책이 전부 (a)(b)로 수렴 → 별개 기능이 아니라 **단일 VLM 재정합 Agent**(②~⑥). (a)(b)는 #09 KSF#1·05 PoC가 검증할 리스크가 된다.

> 🎤 **발표 스크립트**
> "Pain은 증상이지 원인이 아닙니다. 3-Why로 내려가면 모든 셋업 병목이 '패턴 인식 실패'와 '좌표 오류' 둘로 수렴하고, 그래서 해결책도 하나의 Agent로 묶입니다."

## 장표 6 — STEP 05 · 과제 적정성 + 가치검증 (Validation 흡수)

7문항 × 1~3점 스코어카드로 **착수 게이트**를 통과시킨다. 비즈니스 성과 칸이 Validation(ROI)을 흡수.

| 카테고리 | 문항 | 점수 |
|---|---|:---:|
| 전략적 중요도 | 본원 경쟁력 · KPI 연계성 | 3 · 3 |
| **비즈니스 성과 (Validation)** | 수익성(ROI) · 확장성 · 시급성 | τ · τ · τ |
| 실행 용이성 | 데이터 준비(강점) · PoC 존재(강점) | 3 · 3 |

```
Σ = (3+3) + (τ₃+τ₄+τ₅) + (3+3) = 12 + Σ_B(τ) / 21
ROI = ΔS × N,   ΔS = s − s′  (⑥ 절감이 대부분),   s·s′·N = placeholder
착수 게이트: Σ ≥ τ_gate → #10 Execution 착수, 미달 시 재설계
```

> **12점은 가설 확정**(전략 직결 + 데이터 강점 + PoC 존재), 나머지 비즈니스 성과 3문항이 캠프 확정 몫. 별도 Validation 문서 없이 스코어카드 안에 ROI를 박았다.

> 🎤 **발표 스크립트**
> "발굴됐다고 착수해도 좋다는 뜻은 아닙니다. 7문항으로 점수화해 게이트를 통과시키고, 수익성·확장성 칸이 곧 ROI 검증입니다. 강점은 데이터가 이미 있고 PoC 설계가 잡혀 있다는 점입니다."

## 장표 7 — STEP 06 · AI 과제 정의서 (1매)

Discovery 5단계 결과를 **한 장으로 압축**. 각 칸이 앞 단계에서 상속된다(추적성 도착지).

| 정의서 칸 | 내용 | 상속 |
|---|---|:---:|
| 업무 목표 | 장비 운영/관리 최적화 · 검/계측 효율화 | #01 |
| 관련(핵심) 업무 | MI Recipe Solution 제공 · 측정 장비 간 Skew Zero · MI 장비 가동률 극대화 | #01 |
| 핵심 업무 구조화 | 계측>Recipe Management>Recipe Creation (①~⑦) | #02 |
| 추진 배경(Pain) | ⑥ 오인식 수정 반복 = 최대 병목 | #03 |
| 근본원인(Big Why) | (a) 패턴 인식 실패 · (b) 좌표 오류 | #04 |
| **AI Agent** | 대상 **②~⑥** / SEM 판독 + GUI 제어 + 실시간 재정합 / ref layer 우선 확장 | #03·#04 |

> **HITL 박제**: AI는 ②~⑥만. ①(부족정보 플래깅)·⑦(최종 검증·승인)·임계 초과 예외는 사람. 적정성 점수 Σ/21은 정의서를 써도 되는지 판정하는 **착수 게이트**.

> 🎤 **발표 스크립트**
> "흩어진 Discovery 결과를 한 장으로 모았습니다. AI가 하는 일은 ②~⑥로 한정되고, 의뢰서 검토와 최종 승인은 사람이 합니다. 정의서의 모든 칸은 앞 단계에서 상속돼 추적성이 끊기지 않습니다."

## 장표 8 — STEP 07 · As-Is 프로세스 (현재 — 전부 사람)

셋업 ①~⑦ 전 구간이 🧑 사람 수작업. ⑥에 **반복 루프**가 걸려 소요시간을 지배한다.

```
🧑①확인 → 🧑②생성·로드 → 🧑③패턴등록◀병목 → 🧑④좌표·박스◀병목
        → 🧑⑤이미징 → 🧑⑥시험측정→오인식수정 ◀최대병목 → 🧑⑦저장·검증
                          │ 통과 미달 → 수정 후 재측정 ↺ 반복 루프
```

| 단계 | 소요 | 시스템 |
|---|:---:|---|
| ③ 패턴 등록 *(병목)* | s₃ | CD-SEM GUI |
| ④ 좌표·박스 *(병목)* | s₄ | CD-SEM GUI |
| **⑥ 시험측정·오인식 수정 *(최대)*** | **s₆ (최대)** | CD-SEM GUI |
| **합계 (1건당 리드타임)** | **ΣS** | — |

> 소요시간 `s₁..s₇`는 실측 전 placeholder. ⑥ 반복 루프가 `s₆`를 최대로 만들며, 이 컬럼이 #08 before 값·#05 ROI의 ΔS·#10 기대효과의 근거가 된다.

> 🎤 **발표 스크립트**
> "현재는 일곱 단계 모두 사람이 합니다. 특히 ⑥은 시험측정과 수정을 통과할 때까지 반복해서, 셋업 시간을 가장 많이 잡아먹습니다."

## 장표 9 — STEP 08 · To-Be 프로세스 (🤖/🧑 레인)

같은 ①~⑦ 축에서 **②~⑥을 VLM(🤖)이 연속 수행**, ①·⑦은 사람, 임계 초과 예외만 핸드오프.

```
🧑①        ┌──── 🤖 AI Agent (VLM, computer-use) ────┐        🧑⑦
의뢰서      │ ②생성 ▶③패턴 ▶④좌표 ▶⑤이미징 ▶⑥실시간 │        최종
플래깅      │ VLM이 SEM 화면 판독 + GUI 직접 제어,      │ ─게이트▶ 검증·
            │ ⑥에서 좌표 shift·오인식 실시간 재정합     │        승인
            └──────────────┬───────────────────────────┘
              예외(θ·τ 미달 / 재시도 r 초과) → 🧑 엔지니어 핸드오프
```

| 단계 | 적용 AI 기술 (사내 LLM) | 운영 기준 |
|---|---|---|
| ②~⑤ GUI 제어 | **Qwen3-VL** computer-use | 화면매칭 θ 이상 |
| ⑥ 재정합 *(구 최대병목)* | **Qwen3-VL-30B** 정밀 비전 | 정확도 τ + 재시도 r 이내 |
| ⑦ 승인 | 사람 + VLM 자가점검 리포트 | **승인 게이트** |

> 외부 LLM API 미사용 — 전부 사내 엔드포인트(Kimi-K2.5·Qwen3-VL-8B/30B·BGE-M3). 운영 기준 θ/τ/r은 #09 KSF·05 PoC와 **같은 기호**.

> 🎤 **발표 스크립트**
> "To-Be의 핵심 변화는 ②~⑥이 사람에서 VLM으로 넘어가는 것입니다. 단, θ·τ·r이라는 운영 기준 안에서만 자동이고, 벗어나면 즉시 사람에게 넘깁니다. 모델은 전부 사내 LLM을 씁니다."

## 장표 10 — STEP 09 · KSF · 제약 (4관점)

To-Be가 작동하려면 **선결돼야 할 조건**을 4관점으로 박제.

| 관점 | 핵심 성공요소(KSF) | 선결 조건 |
|---|---|---|
| 업무 프로세스 | ②~⑥ 연속 + 예외 핸드오프 | 임계 τ·재시도 r + ⑦ 승인 게이트 |
| 데이터 | SEM 이미지·ref·라벨 충분 | 숙련자 라벨 + 스크린샷→VLM 추출(99% DRM) |
| 시스템·인프라 | GUI 자동제어 안정 + 자원 | 사내 Qwen3-VL 엔드포인트·GPU·섀도우·인터록 |
| 조직 문화 | 엔지니어 신뢰·현장 수용 | **⑦ 사람 최종승인 유지** + 단계적 도입 |

```
선결 우선순위:  #1 좌표 재정합 정확도 (= Phase0 PoC, 최대 리스크)
            → #2 GUI 자동제어 안정 → #3 핸드오프 기준(τ/r) → #4 신뢰·수용
```

> ⚠️ 장비 직접 제어 과제 → **안전 인터록·롤백 / 사고 책임소재 / 보안 승인·감사 로그** 필수. 장애 요인 컬럼은 #10 리스크 게이트(🚦)의 입력.

> 🎤 **발표 스크립트**
> "설계가 좋아도 선결 조건이 안 되면 무너집니다. 가장 먼저 깨질 곳은 좌표 재정합 정확도라서, 이걸 1순위로 PoC에서 단독 검증합니다. 장비를 직접 제어하므로 인터록·롤백·감사 로그를 필수로 박았습니다."

## 장표 11 — STEP 10 · 개발 일정 (1년 분기 칸반 + 게이트)

KSF 우선순위를 분기 칸반으로 환산하고 **분기말 Stage-Gate**로 자원 투입을 통제.

| 테마 | Q1 PoC | Q2 단일 layer | Q3 ⑥ 본격 | Q4 확장·이관 |
|---|---|---|---|---|
| 프로세스 | θ/τ/r 정의 | ②~⑤ 구현 | ⑥ 루프 + 핸드오프 | 운영 이관·SOP |
| 데이터 | 라벨·hard set | 추출 파이프라인 | 피드백 반영 | 재학습셋 |
| AI 모델 | 재정합 PoC(8B vs 30B) | computer-use 통합 | 30B 정밀+룰엔진 | 일반화·경량화 |
| Test | 섀도우·프로토콜 | E2E 오프라인 | 수용 게이트 | 파일럿(실장비) |

```
🚦 G0 (Q1말, Stage-Gate): 재정합 P95 오차 ≤ τ AND recall ≥ r
    통과→Q2 GUI 통합 │ 미달→과제 재설계 (자원 투입 전 중단, No-Regret)
👤 변화관리(전 구간 상시): 운영팀 협업 · 엔지니어 신뢰 · HITL 승인 UI
Kill 기준: 게이트 재설계 누적 2회 초과 → 과제 중단 (좀비 과제 방지)
```

> 테마 레인 = #09 KSF 4관점과 1:1 정렬 → 일정이 성공조건을 빠짐없이 덮는다. 소요·인력은 placeholder.

> 🎤 **발표 스크립트**
> "1년 안에 끝내는 일정입니다. 핵심은 Q1 말 G0 게이트 — 좌표 재정합이 기준을 못 넘으면 GUI 개발에 돈을 쓰기 전에 과제를 재설계합니다. 먼저 깨질 곳을 먼저 칩니다."
