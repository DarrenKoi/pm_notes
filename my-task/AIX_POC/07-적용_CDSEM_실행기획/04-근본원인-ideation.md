# 04 · 근본원인 분석 + 문제해결 Ideation (Step5+6 / Discovery)

> [#03 Pain Point 도출·정량화](./03-painpoint-도출-정량화.md)에서 받은 L4(=As-Is ①~⑦) Pain 중 **M3에서 빨간 dot으로 찍힌 Critical Pain 3개(②·⑥·①)** 에 집중해, **근본원인(3-Why)** 까지 파고들어 **6유형 분류 라벨**을 달고 각각을 **해결 아이디어와 1:1로 페어링**한다. 이 단계의 결론 — **집중 근본원인 (A)=(a) 패턴 인식 실패 [시스템] · (B)=(b) 측정박스/좌표 오류 [시스템]** — 이 단일 **"VLM 실시간 재정합 Agent"** 로 수렴하며, 이후 [#09 KSF](./09-ksf-제약-정의.md)와 [05 PoC 리스크](../05-적용_CDSEM_PoC실험설계.md)의 출처가 된다. (상류 데이터 갈래 Pain①은 (A)(B)에 포함되지 않아 **후순위 과제로 분리**.)

## 왜 필요한가 (Why)

#03에서 Pain Point를 단계별로 망라하고 정량화했지만, Pain은 *증상*이지 *원인*이 아니다. 증상만 보고 해결책을 붙이면 표면 대증요법에 그친다. New AI Design Camp 방법론은 이 단계를 "개선 필요성이 높은 Pain Point를 유발하는 **근본 원인**을 분석하고, 이를 해결할 수 있는 방안을 **아이디어 차원**에서 논의"하는 구간으로 정의한다 — 그래야 다음의 실행 Process 설계에서 문제 해결 방법을 더 명확하게 설정할 수 있다 (출처: [원문 08](../source/08-final-output-example.md)).

이번 M4 세션에서는 #03 L4 Pain 전체 중 **M3에서 빨간 dot으로 표시된 Critical Pain 3개(②·⑥·①)** 를 집중 대상으로 확정하고, 각 근본원인에 **6유형 분류 라벨**을 부여했다 (브레인스토밍 합의 2026-06-29). 나머지 L4 Pain(③④·야간공백·⑦)은 *주변 보완 행*으로 기존 확정 사실을 그대로 보존한다.

이 단계가 하는 일은 세 가지다.

1. **증상 → 원인 분리 + 유형 라벨링**: "오계측 보정이 오래 걸린다"는 증상을, "왜?"를 세 번 물어 *진짜 원인*까지 내려가고, 그 원인이 **사람·프로세스·시스템·데이터·표준/룰·환경 6유형** 중 어디인지 라벨을 단다. (마지막 Why는 *사람 탓*이 아니라 시스템/데이터 층위에서 종결한다.)
2. **원인 → 아이디어 수렴**: 흩어진 셋업 Pain의 근본원인이 결국 **(a)=(A) 패턴 인식 실패 · (b)=(B) 측정박스/좌표 오류** 두 갈래(둘 다 [시스템] 유형)로 수렴함을 드러내, Pain②·⑥의 해결 아이디어를 하나의 핵심 Agent(**VLM 실시간 재정합**)로 묶는다. Pain①(의뢰서) 갈래는 여기 포함되지 않아 상류 데이터 후순위로 분리한다.
3. **다음 단계의 씨앗 제공**: 여기서 확정한 집중 근본원인 (A)(B)가 바로 #09 KSF의 1순위 선결조건과 05 PoC가 검증할 리스크 변수가 된다. 즉 이 표는 *과제의 기술 리스크를 미리 박제하는* 문서다.

> 본 단계는 #03 Phase의 "근본 원인(3-Why) → 해결 아이디어표"(03 Phase 3)를 받아 **M3 빨간 dot Critical Pain 중심으로 심화**한 상세층이다 (03 인용). 03이 집중 Pain만 4행으로 다뤘다면, 여기서는 Critical 3행 + 주변 3행으로 펼치되 *집중은 Critical 3개*에 둔다.

## 핵심 산출 (What)

**Critical Pain ↔ 근본원인(3-Why·6유형) ↔ 해결 아이디어 1:1 페어링 표** 1매. 컬럼은 방법론 양식(출처: [원문 08](../source/08-final-output-example.md))에 **유형(6분류)** 컬럼을 더한다.

| 항목 | 의미 |
|------|------|
| Critical Pain | #03 L4 Pain 중 **M3 빨간 dot** 으로 찍힌 집중 대상(②·⑥·①) |
| 근본원인 (왜① → 왜② → 왜③ 근본) | "왜?"를 반복해 증상 아래의 구조적 원인까지 내려감 (증상 재진술·사람 탓 종결 금지) |
| 유형 (6분류) | 근본원인이 **사람 · 프로세스 · 시스템 · 데이터 · 표준/룰 · 환경** 중 무엇인지 라벨 (필수) |
| 해결 아이디어 | 근본원인을 직접 겨냥하는 To-Be 방향, **원인별 2개 이상·2관점 이상** (아이디어 수준, 설계 아님) |
| 수렴 근본원인 | 그 아이디어가 (A)(B) 중 어디로 수렴하는지 표시 |

산출물의 핵심 메시지: **Critical Pain②·⑥의 해결 아이디어가 (A) 패턴 인식 실패 · (B) 측정박스/좌표 오류 두 [시스템] 근본원인으로 수렴**하므로, 별개 기능이 아니라 **하나의 VLM 실시간 재정합 Agent**로 묶인다. **Pain①(의뢰서)** 은 (A)(B)에 들지 않는 [데이터·프로세스] 갈래라 **상류 후순위 과제로 분리**한다.

## 작성 결과 (How)

### 1. 3-Why 드릴 — M3 Critical Pain 3개 (②·⑥·①)

M3에서 빨간 dot으로 찍힌 3개를 각각 "왜?"를 세 번 물어 내려가고, 마지막 Why(근본)에 **6유형 라벨**을 단다. 작성 규칙: ① 근본원인 칸에 증상을 다시 쓰지 않는다, ② "사람이 실수해서"로 종결하지 않고 한 단계 더 내려가 시스템/데이터·프로세스 층위에서 멈춘다.

**Critical Pain② — 좌표 shift & 공정 변화로 align 인식 실패** (As-Is ③ 등록·⑥ 실측에서 발현, **PoC 핵심**)
```
증상 : 좌표 shift·공정 변화로 align/패턴 인식이 실패한다 (M3 🔴)
  │
  └─ 왜① ─▶ 미리 만든 인식 템플릿이 Wafer 실측 시점의 화면과 다르다
        │
        └─ 왜② ─▶ 공정 변화 & 실측 시작 시 전체 좌표가 shift 된다
              │
              └─ 왜③ (근본) ─▶ 룰·템플릿 매칭이 공정 변화·좌표 shift에 강건하지 않다
                              (적응형 인식 체계 부재)            [유형: 시스템]
```

**Critical Pain⑥ — 오계측 시 보정 반복** (As-Is ⑥, **셋업 최대 병목**)
```
증상 : 시험측정(Try) 후 오계측·실패 지점 보정이 반복돼 셋업이 길어진다 (M3 🔴)
  │
  └─ 왜① ─▶ 측정 박스·좌표가 실제 패턴과 어긋난다
        │
        └─ 왜② ─▶ 그 어긋남을 사람이 눈으로 보고 수동 보정한다 (반복·숙련 의존)
              │
              └─ 왜③ (근본) ─▶ 오인식·측정박스 어긋남을 자동으로 검출·보정하는
                              체계가 없다                        [유형: 시스템]
```

**Critical Pain① — 의뢰서 정보 부족·표준 미흡** (As-Is ①, **상류 데이터 갈래·PoC 범위 밖 = 후순위**)
```
증상 : 의뢰서 정보가 부족·불명확해 셋업 시 재문의·추측이 생긴다 (M3 🔴)
  │
  └─ 왜① ─▶ 부서·엔지니어별 의뢰서 작성 수준이 제각각이다
        │
        └─ 왜② ─▶ 표준 필수 항목(반드시 채울 칸)이 정의돼 있지 않다
              │
              └─ 왜③ (근본) ─▶ 의뢰서 작성·검수 거버넌스(프로세스)가 부재하다
                                                                [유형: 데이터·프로세스]
```

→ Pain②의 근본은 **(a)=(A) 패턴 인식 실패**, Pain⑥의 근본은 **(b)=(B) 측정박스/좌표 오류** 로 귀결되며 **둘 다 [시스템] 유형**이다. 이 둘이 본 과제의 **집중 근본원인**이고, 이후 셋업 해결 아이디어가 이 둘을 겨냥한다. Pain①은 [데이터·프로세스] 유형의 상류 갈래로 (A)(B)와 다른 축이다 (03 Phase 3 인용 + 브레인스토밍 합의 2026-06-29).

### 2. Critical Pain ↔ 근본원인(3-Why·6유형) ↔ 해결 아이디어 페어링 표

#### (집중) M3 빨간 dot Critical Pain 3행

| Critical Pain (M3 🔴) | 근본원인 (왜① → 왜② → 왜③ 근본) | 유형 | 해결 아이디어 (2관점 이상) | 수렴 |
|---|---|:---:|---|:---:|
| **② 좌표 shift·공정 변화로 align 인식 실패** (As-Is ③/⑥, PoC 핵심) | 템플릿이 실측 화면과 다름 → 공정 변화·좌표 shift → **룰·템플릿 매칭이 공정변화·shift에 강건하지 않음(적응형 인식 체계 부재)** | **시스템** | (시스템) VLM 기반 실시간 패턴 재인식·좌표 자동 재정합 · (사람) 수동 보정·숙련 의존 제거 | **(a)=(A)** |
| **⑥ 오계측 시 보정 반복** (As-Is ⑥, 최대 병목) | 측정박스·좌표가 실제 패턴과 어긋남 → 사람이 눈으로 수동 보정(반복·숙련 의존) → **오인식·측정박스 어긋남을 자동 검출·보정하는 체계 없음** | **시스템** | (시스템) 오계측 실시간 모니터링 후 자동 보정 루프 · (프로세스) 사람은 임계 초과 예외만 확인 | **(b)=(B)** |
| **① 의뢰서 정보 부족·표준 미흡** (As-Is ①, 상류·PoC 범위 밖) | 부서·엔지니어별 작성 수준 상이 → 표준 필수 항목 없음 → **의뢰서 작성·검수 거버넌스(프로세스) 부재** | **데이터·프로세스** | (데이터) 의뢰서 표준화 및 작성 자동화 · (프로세스) 제출 시 AI 필수 항목 검수 | (상류·후순위) |

#### (주변) 비-critical 보완 Pain — 기존 확정 사실 보존(라벨만 추가)

> 아래 행은 #03 L4 Pain 중 M3에서 빨간 dot이 찍히지 *않은* 주변 보완 Pain이다. 기존 확정 해결 아이디어를 그대로 유지하고 유형 라벨만 추가한다.

| 주변 Pain (#03 L4) | 근본원인 | 유형 | 해결 아이디어 | 수렴 |
|---|---|:---:|---|:---:|
| **③④ 패턴 등록·EP 좌표/박스 설정** (병목) | 수작업·숙련 의존 → 표준 판단기준 부재 → 같은 패턴도 사람마다 다르게 등록(숙련도 편차) | 시스템 | VLM이 일관 기준으로 패턴 등록·좌표 설정 (②~⑥를 ⑥과 같은 Agent로 묶음) | (a)(b) |
| **B 야간/주말 셋업 공백** | 셋업 처리량 상한 존재 → 사람 작업시간이라는 물리적 제약 | 프로세스(운영) | VLM 24h 무중단 셋업, 사람은 예외만 개입(HITL) | (운영측) |
| **⑦ 저장·등록·검증 단계 검증 누락** | 셋업 부실이 ⑦에서 안 걸러져 후속 측정·양산 이관 후로 전파 → 표준 검증 체크리스트 부재 | 프로세스(검증) | 사람 최종 검증 유지 + AI 자가점검 리포트(신뢰도·재정합 이력 자동 첨부) | (검증측) |

> **표 읽는 법** — 집중 3행 중 ②·⑥의 해결 아이디어가 모두 **(A)(B)로 수렴**한다(둘 다 [시스템]). 즉 셋업 병목 해결책은 별개 기능이 아니라 **하나의 VLM 실시간 재정합 Agent**(②~⑥ 연속 수행)로 묶인다 (03 Phase 3 인용). Pain①은 [데이터·프로세스] 유형의 상류 갈래라 (A)(B)에 들지 않고 **후순위로 분리**한다. 주변 3행(③④·야간공백·⑦)은 (A)(B)를 보강하거나 다른 축(운영·검증)을 받친다.

### 3. 집중 근본원인 수렴 (이 단계의 결론)

```
집중 근본원인 (A)=(a) 패턴 인식 실패        [유형: 시스템]   ← Critical Pain② 수렴
집중 근본원인 (B)=(b) 측정박스/좌표 오류     [유형: 시스템]   ← Critical Pain⑥ 수렴

→ Pain②·⑥의 해결책이 (A)(B)로 수렴 → 별개 기능이 아니라
   단일 "VLM 실시간 재정합 Agent" 하나로 묶인다 (②~⑥ 연속 수행).

→ Pain① 의뢰서 갈래는 (A)(B)에 포함되지 않는다 [데이터·프로세스]
   → 상류 데이터 후순위 과제로 분리 (부족정보 플래깅·의뢰서 표준화, MIDAS 연계 선).

→ (A)(B)는 다음 단계로 흐른다:
   ├─ 05 PoC 실험 설계 : Phase 0 = (A)(B) 좌표 재정합 정확도 1순위 검증 (τ/r placeholder)
   └─ #09 KSF · 제약   : 선결 우선순위 #1 = (A)(B) 해소 = 좌표 재정합 정확도
```

- **#09로**: (A)(B)는 #09 KSF 매트릭스의 **데이터 관점**(SEM 이미지·reference·라벨 충분성)과 **업무 프로세스 관점**(⑥ 재정합 잔존·예외 모호 = 장애요인)의 직접 입력이다. #09 선결 우선순위 **#1 = 좌표 재정합 정확도**가 곧 (A)(B)의 해소 여부다.
- **05 PoC로**: (A)(B)는 회피 대상이 아니라 **Phase 0 PoC로 1순위 검증**할 리스크다 (03 Phase 4·5 인용). 검증 임계는 재정합 정확도 τ, 재시도 횟수 r 로 두되 절대 수치는 캠프에서 엔지니어가 확정한다 (placeholder).
- **Pain①(의뢰서)은 분리**: (A)(B)와 같은 Agent로 묶지 않는다. 상류 데이터 거버넌스(의뢰서 표준화·작성 자동화·AI 필수항목 검수)는 MIDAS 연계 선까지의 별도 후순위 트랙으로 둔다 (SPEC §2 — MIDAS 전산화는 외부 변수).

> ⑥의 구현 난이도(정밀 좌표 재정합 정확도)는 03에서 "남는 리스크 = ⑥의 구현 난이도 → KSF#1·Phase 0 PoC로 1순위 검증"으로 이미 식별됐다 (03 Phase 4 인용). 본 단계는 그 리스크의 **근본원인 (A)(B)** 를 [시스템] 유형으로 명시 분리해 후속 문서에 넘긴다.
