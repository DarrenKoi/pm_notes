# 04 · 근본원인 분석 + 문제해결 Ideation (Step5+6 / Discovery)

> [#03 Pain Point 도출·정량화](./03-painpoint-도출-정량화.md)에서 받은 L4(=As-Is ①~⑦) Pain을 **근본원인(3-Why)** 까지 파고들어 각각을 **해결 아이디어와 1:1로 페어링**한다. 이 단계의 결론 — **집중 근본원인 (a) 패턴 인식 실패 · (b) 측정박스/좌표 오류** — 가 이후 [#09 KSF](./09-ksf-제약-정의.md)와 [05 PoC 리스크](../05-적용_CDSEM_PoC실험설계.md)의 출처가 된다.

## 왜 필요한가 (Why)

#03에서 Pain Point를 단계별로 망라하고 정량화했지만, Pain은 *증상*이지 *원인*이 아니다. 증상만 보고 해결책을 붙이면 표면 대증요법에 그친다. New AI Design Camp 방법론은 이 단계를 "개선 필요성이 높은 Pain Point를 유발하는 **근본 원인**을 분석하고, 이를 해결할 수 있는 방안을 **아이디어 차원**에서 논의"하는 구간으로 정의한다 — 그래야 다음의 실행 Process 설계에서 문제 해결 방법을 더 명확하게 설정할 수 있다 (출처: [원문 08](../source/08-final-output-example.md)).

이 단계가 하는 일은 세 가지다.

1. **증상 → 원인 분리**: "오인식 수정이 오래 걸린다"는 증상을, "왜?"를 세 번 물어 *진짜 원인*까지 내려간다.
2. **원인 → 아이디어 수렴**: 흩어진 셋업 Pain들의 근본원인이 결국 **(a) 패턴 인식 실패 · (b) 측정박스/좌표 오류** 두 갈래로 수렴함을 드러내, 해결 아이디어를 하나의 핵심 Agent(VLM 실시간 재정합)로 묶는다.
3. **다음 단계의 씨앗 제공**: 여기서 확정한 집중 근본원인 (a)(b)가 바로 #09 KSF의 1순위 선결조건과 05 PoC가 검증할 리스크 변수가 된다. 즉 이 표는 *과제의 기술 리스크를 미리 박제하는* 문서다.

> 본 단계는 #03 Phase의 "근본 원인(3-Why) → 해결 아이디어표"(03 Phase 3)를 받아 **L4 Pain 전체로 확장**한 상세층이다 (03 인용). 03이 집중 Pain만 4행으로 다뤘다면, 여기서는 #03이 정량화한 L4 Pain을 모두 받아 5행으로 펼친다.

## 핵심 산출 (What)

**Pain Point ↔ 근본원인(3-Why) ↔ 해결 아이디어 1:1 페어링 표** 1매. 컬럼은 방법론 양식(출처: [원문 08](../source/08-final-output-example.md))을 그대로 따른다.

| 항목 | 의미 |
|------|------|
| Pain Point | #03 L4(As-Is ①~⑦) Pain 리스트에서 그대로 받음 (03 인용) |
| 근본원인 (Why → Why → Why) | "왜?"를 반복해 증상 아래의 구조적 원인까지 내려감 |
| 해결 아이디어 | 근본원인을 직접 겨냥하는 To-Be 방향 (아이디어 수준, 설계 아님) |
| 수렴 근본원인 | 그 아이디어가 (a)(b) 중 어디로 수렴하는지 표시 |

산출물의 핵심 메시지: **셋업 병목(③④⑥) 해결 아이디어가 전부 (a) 패턴 인식 실패 · (b) 측정박스/좌표 오류 두 근본원인으로 수렴**하므로, 별개 기능이 아니라 **하나의 VLM 실시간 재정합 Agent**로 묶인다.

## 작성 결과 (How)

### 1. 3-Why 드릴 — ⑥ 오인식 수정 반복 (최대 병목)

가장 깊게 파야 할 최대 병목 ⑥은 "왜?"를 세 번 물어 내려간다. ⑥의 As-Is 정의 = "시험측정(Try) → 오인식/실패 지점 수정 (반복)" (출처: SPEC §2 As-Is ⑥).

```
증상 : 시험측정(Try) 후 오인식·측정실패 지점을 수정하는 작업이 반복돼 셋업이 길어진다 (⑥, 최대 병목)
  │
  └─ 왜? ─▶ 미리 만든 패턴 인식 템플릿·좌표가 실측 시작 시 그대로 맞지 않는다
        │
        └─ 왜? ─▶ 실측 시작 시 웨이퍼/스테이지 기준으로 전체 좌표가 shift 되거나 align 이 실패한다
              │
              └─ 왜? ─▶ (a) SEM 화면의 실제 패턴을 템플릿이 못 알아본다  = 패턴 인식 실패
                        (b) 인식하더라도 측정박스·EP 좌표가 실제 패턴과 어긋난다 = 측정박스/좌표 오류
                        ↳ 현재는 이 (a)(b)를 사람이 화면 보며 눈으로 잡아 수동 보정 → 반복·숙련 의존
```

→ ⑥의 근본원인은 결국 **(a) 패턴 인식 실패 · (b) 측정박스/좌표 오류** 두 갈래다. 이 둘이 본 과제의 **집중 근본원인**이고, 이후 모든 셋업 해결 아이디어가 이 둘을 겨냥한다 (03 Phase 3 인용).

### 2. Pain ↔ 근본원인(3-Why) ↔ 해결 아이디어 페어링 표

#03 L4 Pain 전체를 받아 1:1로 페어링한다. 각 행의 Pain은 #03에서, 근본원인 구조와 해결 아이디어 방향은 03 Phase 3 + 브레인스토밍 합의(2026-06-24)에서 온다.

| Pain Point (L4, #03 받음) | 근본원인 (왜 → 왜 → 왜) | 해결 아이디어 | 수렴 근본원인 |
|---------------------------|--------------------------|----------------|:---:|
| **⑥ 시험측정 후 오인식·실패 지점 수정 반복** (최대 병목) | 템플릿을 미리 만들어도 → 실측 시작 시 전체 좌표 shift·align 실패 발생 → **(a) 패턴 인식 실패 / (b) 측정박스·좌표 오류** (위 3-Why 드릴) | **VLM이 SEM 이미지를 보고 실시간으로 좌표 재정합·재align, 오인식 자동 수정** (핵심 아이디어) | **(a)(b)** |
| **③ Alignment/Addressing 포인트 등록**, **④ EP 좌표·측정박스·알고리즘 설정** (병목) | 수작업·숙련 의존 → 표준 판단기준 부재 → 같은 패턴도 사람마다 다르게 등록(숙련도 편차) | **VLM이 일관된 기준으로 패턴 등록·좌표 설정 수행** (③④를 ⑥과 같은 Agent로 묶음) | **(a)(b)** |
| **① 측정 대상 레이어/패턴 확인** 단계의 의뢰정보 부족 (의뢰서 정보 품질) | 셋업 시 재문의·추측 발생 → 단일 참조 기준 부재 + 의뢰자가 의뢰서를 명확히 작성하지 않음 | **VLM이 의뢰서 부족정보를 플래깅** (MIDAS 연계 선까지 — MIDAS 전산화는 외부 변수) | (입력측) |
| **B 전반 야간/주말 셋업 공백** | 셋업 처리량 상한 존재 → 사람 작업시간이라는 물리적 제약 | **VLM 24h 무중단 셋업 수행, 사람은 예외만 개입(HITL)** | (운영측) |
| **⑦ Recipe 저장·등록·검증 단계 검증 누락** | 셋업 부실이 ⑦에서 안 걸러져 양산에 전파 → 표준 검증 체크리스트 부재 | **사람 최종 검증 유지 + AI 자가점검 리포트**(신뢰도·재정합 이력 자동 첨부)로 검증 보조 | (검증측) |

> **표 읽는 법** — 위 두 행(⑥, ③④)의 해결 아이디어가 모두 **(a)(b)로 수렴**한다. 즉 셋업 병목 해결책은 별개 기능이 아니라 **하나의 VLM 실시간 재정합 Agent**(②~⑥ 연속 수행)로 묶인다 (03 Phase 3 인용). 아래 세 행(①·야간공백·⑦)은 (a)(b)와 다른 축(입력·운영·검증)을 보완하는 주변 아이디어다.

### 3. 집중 근본원인이 다음 단계로 흐르는 경로

이 단계의 산출이 단순 분석으로 끝나지 않고 후속 문서의 입력이 됨을 명시한다.

```
집중 근본원인 (a) 패턴 인식 실패 · (b) 측정박스/좌표 오류
   │
   ├─▶ #09 KSF · 제약 정의 : 선결 우선순위 #1 = "좌표 재정합 정확도"
   │      → 데이터 관점 KSF(SEM 이미지·ref·라벨 충분), 업무 프로세스 KSF(⑥ 재정합 잔존 = 장애요인)
   │
   └─▶ 05 PoC 실험 설계 : Phase 0 PoC = "KSF#1 좌표 재정합" 정확도 검증
          → 검증 변수 = 재정합 정확도 임계 τ, 재시도 r (placeholder, 엔지니어 확정)
```

- **#09로**: (a)(b)는 #09 KSF 매트릭스의 **데이터 관점**(SEM 이미지·reference·라벨 충분성)과 **업무 프로세스 관점**(⑥ 재정합 잔존·예외 모호 = 장애요인)의 직접 입력이다. #09 선결 우선순위 **#1 = 좌표 재정합 정확도**가 곧 (a)(b)의 해소 여부다.
- **05 PoC로**: (a)(b)는 회피 대상이 아니라 **Phase 0 PoC로 1순위 검증**할 리스크다 (03 Phase 4·5 인용). 검증 임계는 재정합 정확도 τ, 재시도 횟수 r 로 두되 절대 수치는 캠프에서 엔지니어가 확정한다 (placeholder).

> ⑥의 구현 난이도(정밀 좌표 재정합 정확도)는 03에서 "남는 리스크 = ⑥의 구현 난이도 → KSF#1·Phase 0 PoC로 1순위 검증"으로 이미 식별됐다 (03 Phase 4 인용). 본 단계는 그 리스크의 **근본원인 (a)(b)** 를 명시적으로 분리해 후속 문서에 넘긴다.
