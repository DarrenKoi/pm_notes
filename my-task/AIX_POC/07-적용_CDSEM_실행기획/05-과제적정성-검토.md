# 05 · 과제 적정성 검토 (Step7 + Validation 흡수)

> 선정 후보 **VLM GUI-제어 기반 CD-SEM Recipe Creation 자동화**를 7문항 × 1~3점 스코어카드로 평가해, 종합 Σ/21을 **#10 착수(Execution) 게이트의 정량 근거**로 확정하는 단계.

## 왜 필요한가 (Why)

Discovery(#01~#04)는 "이 과제가 조직목표에서 연역됐고(#01), 대상 업무가 명확하며(#02), Pain이 정량화됐고(#03), 근본원인까지 파고들었다(#04)"를 보였다. 그러나 **발굴됐다는 사실이 곧 착수해도 좋다는 뜻은 아니다.** 자원이 한정된 이상, 후보를 객관적 잣대로 점수화해 "지금 이 과제를 실행할 가치가 있는가"를 게이트로 통과시켜야 한다.

- 과제 선정 3-STEP의 STEP 3은 **"고통의 정도(빈도·심각성·시급성/중요성) · 구현 난이도 · 비용 효용성을 고려해 최종 AI 과제를 선정한다"** 고 못박는다 (출처: [원문 01](../source/01-kakao-3step-pain-point.md)). 이 단계가 바로 그 STEP 3의 정량 게이트다.
- 방법론 4 Step 중 **Step 3 Validation(가치 검증)** = "투자 효용성 및 재무적 효과 산출 / 비즈니스 임팩트 정량화 / 재무적 타당성 산출" (출처: [원문 02](../source/02-ax-consulting-methodology.md)). 본 문서의 **'비즈니스 성과' 카테고리(수익성·확장성·시급성)** 가 이 Validation을 흡수한다 — 별도 Validation 문서를 두지 않고 적정성 스코어카드 안에 ROI 검증을 박는다.
- #03의 Pain 정량 컬럼(빈도·심각성·주요성)과 #04의 근본원인이 본 스코어카드의 **백데이터**다. 점수는 여기서 새로 지어내는 것이 아니라 앞 단계 산출물에서 **연역**된다(추적성).

## 핵심 산출 (What)

**3카테고리 × 7문항 스코어카드** (각 문항 1~3점, 종합 Σ/21):

| # | 카테고리 | 문항 | 흡수 단계 |
|---|----------|------|-----------|
| 1 | 전략적 중요도 | 본원적 경쟁력 (조직 본원적 활동 연계성) | Step7 |
| 2 | 전략적 중요도 | 사업전략·목표·KPI 연계성 | Step7 |
| 3 | **비즈니스 성과** | **수익성 (ROI = 절감공수 ΔS × 처리량 N)** | **Validation** |
| 4 | **비즈니스 성과** | **확장성** | **Validation** |
| 5 | **비즈니스 성과** | **시급성** | **Validation** |
| 6 | 실행 용이성 | 데이터 준비 (SEM 이미지·ref recipe 가용성) | Step7 |
| 7 | 실행 용이성 | 준비계획 구체성 (PoC 실험설계 존재) | Step7 |

> **점수 척도(τ placeholder)**: 1점=미흡/불확실, 2점=보통, 3점=우수/확실. 본 문서의 점수는 **브레인스토밍 가설값**이며, 가중치·최종 확정 점수는 **캠프에서 엔지니어·Champion이 정량 확정**한다 (절대 수치 아님 — τ placeholder). 1·2번은 전략 직결 항목이라 가설 만점(3점) 예시로 둔다.

## 작성 결과 (How: 스코어카드 + Validation)

### 스코어카드

#### A. 전략적 중요도 (Σ_A / 6)

| 문항 | 점수 | 근거 |
|------|:---:|------|
| **1. 본원적 경쟁력** (조직 본원적 활동 연계) | **3** (가설 만점) | 계측(MI) Recipe Creation은 신규 공정 셋업 리드타임을 좌우하는 **본원적 활동**이다. 과제가 이 활동의 최대 병목(⑥ 오인식 수정 반복)을 정조준한다 (출처: 03 Phase 1·4 인용). 주변부 효율화가 아니라 핵심 가치사슬상의 과제이므로 가설 만점. |
| **2. 사업전략·목표·KPI 연계성** | **3** (가설 만점) | #01 Top-Down 연계표에서 `조직목표(신규 공정 셋업 리드타임 단축) → 업무목표(Recipe Setup 처리량 확대·숙련편차 제거·24h 무중단) → 핵심업무(Recipe Creation) → 본 과제`로 직접 연역됨 (출처: [#01 전략일관성](./01-전략일관성-확인.md), 03 §1 인용). KPI(리드타임·처리량)에 1:1 매핑되므로 가설 만점. |

#### B. 비즈니스 성과 (Σ_B / 9) — **Validation/가치검증 흡수 지점**

| 문항 | 점수 | 근거 |
|------|:---:|------|
| **3. 수익성** (ROI = 절감공수 ΔS × 처리량 N) | **τ** (placeholder) | ROI = (As-Is 셋업공수 s − To-Be 공수 s′) × 연간 처리량 N. ⑥이 최대 병목이므로 ΔS의 대부분이 ⑥ 절감에서 나온다 (출처: 03 Phase 4 인용). **s·s′·N은 #07 As-Is / #08 To-Be 소요시간 실측 전이라 placeholder**, 점수 τ는 캠프 확정. ROI 산식은 아래 '가치·재무 정량화' 참조. |
| **4. 확장성** | **τ** (placeholder) | ref recipe를 보유한 layer로 수평 확장 가능. 1차 단일 layer 검증 후 동일 VLM Agent를 타 layer·타 장비군으로 일반화 (출처: 03 Phase 4 "②~⑥ 연속수행" + #06 적용범위 예정). 확장 layer 수가 N을 키워 ROI를 증폭시키는 레버 — 점수 τ는 가용 layer 수 확정 후. |
| **5. 시급성** | **τ** (placeholder) | 야간/주말 셋업 공백 = 처리량 상한 제약이 현재 진행형(출처: 03 Phase 2 인용). 신규 공정 셋업 수요 증가 시 병목이 즉시 리드타임으로 전이되므로 시급. 점수 τ는 셋업 적체 현황 수치 확정 후. |

#### C. 실행 용이성 (Σ_C / 6)

| 문항 | 점수 | 근거 |
|------|:---:|------|
| **6. 데이터 준비** (SEM 이미지·ref recipe 가용성) | **3** (가설 — 강점) | **이 과제의 핵심 강점.** SEM 화면 이미지·reference recipe가 **이미 가용한 데이터**라 추측 설계가 아니다 (출처: 03 Why "데이터 중심" 인용; 원문 02 Data-Centric 원칙). 단, 라벨링 공수·배율/노이즈 편차·DRM은 #09 데이터 KSF에서 다룸 — 그래도 데이터 '존재' 자체는 확보됐으므로 가설 3점. |
| **7. 준비계획 구체성** (PoC 실험설계 존재) | **3** (가설 — 강점) | 최대 리스크(⑥ 좌표 재정합 정확도)를 회피하지 않고 **KSF#1 · Phase0 PoC 실험설계로 선검증**하는 구체 계획이 이미 존재 (출처: [PoC 실험설계 05](../05-적용_CDSEM_PoC실험설계.md), 03 Phase 4 "남는 리스크" 인용). 막연한 의지가 아니라 검증 프로토콜(τ/s/r/N 변수)이 잡혀 있어 가설 3점. |

### 종합 점수 (Σ / 21)

```
Σ = Σ_A + Σ_B + Σ_C
  = (3 + 3)  +  (τ₃ + τ₄ + τ₅)  +  (3 + 3)
    └전략 6┘    └Validation: placeholder┘  └실행 6┘
  = 12 + Σ_B(τ)   /  21
```

- 전략적 중요도 6/6 + 실행 용이성 6/6 = **12점이 가설 확정(전략 직결·데이터 강점·PoC 존재)**.
- 나머지 **비즈니스 성과 3문항(τ₃·τ₄·τ₅)** 이 캠프에서 확정되면 Σ가 완성된다.
- **착수 게이트 규칙(placeholder)**: Σ ≥ τ_gate(캠프 확정 임계, 예: 14~15/21) → #10 Execution 착수. 미달 시 재설계 또는 보류. 이 종합점수가 **#10 개발일정 착수 게이트의 정량 근거**다.

### 가치·재무 정량화 (Validation)

본 단계가 4 Step의 **Step 3 Validation(가치 검증)** 을 흡수한다는 점을 명시한다. 위 스코어카드의 **3.수익성 · 4.확장성 칸이 곧 ROI 검증**이며, 별도 Validation 문서 대신 적정성 게이트 안에 재무 타당성을 박는다 (출처: [원문 02](../source/02-ax-consulting-methodology.md) Step3 "투자 효용성 및 재무적 효과 산출").

**ROI 산식 (placeholder 변수만 — 수치는 캠프 확정):**

```
연간 절감 가치  ΔValue = ΔS × N
  ΔS = s − s′      (셋업 1건당 공수 절감)
       s  = As-Is 셋업 소요시간   → #07 As-Is 프로세스에서 실측 (s placeholder)
       s′ = To-Be 셋업 소요시간   → #08 To-Be 프로세스에서 실측 (s′ placeholder)
       (ΔS의 대부분 = ⑥ 오인식 수정 반복 절감, 최대 병목 — 출처 03 Phase 4)
  N  = 연간 처리량 (셋업 건수)     → 확장 layer 수에 비례 (N placeholder)
       + 24h 무중단 → 처리량 상한 자체 상향 (단순 효율화 초과 — 출처 03 Why)

ROI ≈ ΔValue / (구현·운영 투자)   → 투자측 placeholder는 #10 인력·일정에서 산정
```

- **수치 금지 원칙**: s·s′·r·N·τ는 전부 placeholder다. 실측은 #07/#08, 검증 프로토콜은 [PoC 실험설계 05](../05-적용_CDSEM_PoC실험설계.md), 일정·인력 투자측은 #10에서 확정한다. 본 문서는 **산식과 게이트 구조만 박제**하고 빈칸은 캠프 확정 몫으로 남긴다 ("브레인스토밍 합의(2026-06-24)").
- **확장성 = ROI 레버**: 확장 layer 수가 N을 키워 ΔValue를 증폭하므로, 4번 확장성 점수는 수익성과 곱셈 관계로 묶인다.
