# 02 · 대상 업무영역 정의 (Step2 Scope Analysis / Discovery)

> 계측(MI) 운영 전체를 L2 컬럼×L3 프로세스 플로우로 가로로 펼치고, 각 L3에 AI 대체 판정을 오버레이해 **1차 PoC 대상 = Recipe Creation(②~⑥)** 을 좁혀 확정한다.

## 왜 필요한가 (Why)

[01-전략일관성-확인](./01-전략일관성-확인.md)에서 "CD-SEM Recipe Creation"이 조직목표로부터 연역된 핵심업무임을 추적성으로 증명했다. 다만 그 핵심업무가 **계측 운영이라는 더 큰 업무 지도의 어디에 위치하는지, 그리고 그 주변 업무 중 무엇을 AI로 대체할지/말지**를 가리지 않으면 PoC 범위가 흐려진다.

- **"모든 칸에 AI를 넣지 않는다"**: AI Agent는 **반복성·정형성·공수·데이터 가용성**이 큰 지점에만 선별 배치한다 (출처: [원문 06](../source/06-sales-ai-opportunity.md) 해석 포인트, [원문 01 STEP3](../01-기획문서_AX서비스기획.md)).
- **개입 최소 단위는 L5(Task)**: 업무를 L1~L5로 분해해야 AI를 어디에 배치할지 정밀하게 식별할 수 있다 (출처: [원문 05](../source/05-process-level-definition.md) 해석 포인트). 본 문서는 L1~L3까지 가로로 펼쳐 **1차 대상 L3를 선별**하고, L4~L5 분해와 정량화는 [03-painpoint-도출-정량화](./03-painpoint-도출-정량화.md)로 넘긴다.
- **PoC 경계의 합의**: 1차 대상과 차기(2차) 대상을 명시적으로 분리해, 이후 모든 산출물(과제 정의·To-Be·일정)이 동일한 범위 위에서 작성되게 한다.

## 핵심 산출 (What)

1. **L1~L3 가로 레이아웃 다이어그램** — L1=계측(MI) 운영, L2 컬럼 4개, 각 L2 아래로 L3 프로세스 플로우. (포맷 출처: [원문 05](../source/05-process-level-definition.md)의 L1~L5 가로 위계 표)
2. **AI 대체 판정 오버레이 표** — 각 L3에 🤖/🤝/🧑/⚙️ 판정 + 근거(반복성·정형성·공수·데이터가용성 4기준).
3. **범위 결정** — 1차 대상 L3와 차기 대상 L3를 명문화.

> ✅ **L2 4분할은 엔지니어 확정 항목이다 (2026-06-24).** 아래 L2-A~D 구획과 각 L3 목록은 도메인 엔지니어 검토를 거쳐 **계측(MI) 운영의 프로세스 위계로 확정**되었다. 셋업(②~⑥, B 영역)은 As-Is로 확정돼 있고, L2/L3 경계도 동일하게 확정. (출처: [03 §2 L1~L5](../03-적용_CDSEM기획.md) + 엔지니어 확정 2026-06-24)

## 작성 결과 (How: L2×L3 플로우 + 대체 판정)

### A. L1~L3 가로 레이아웃 (Process Level Map)

L1 = **계측(MI) 운영** 하나. 그 아래 L2 4개 컬럼을 좌→우로 두고, 각 컬럼 안에서 L3가 위→아래(또는 좌→우) 흐름을 갖는다. (포맷: [원문 05](../source/05-process-level-definition.md) / 배치 마킹 개념: [원문 06](../source/06-sales-ai-opportunity.md))

```
L1 (Mega Process) :  계측(MI) 운영                                                    (엔지니어 확정 2026-06-24)
================================================================================================================
                                                                                      │ 1차 PoC 대상 ★
   ┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────────┐
L2 │ A. Recipe           │   │ B. 측정 실행        │   │ C. 데이터·모니터링  │   │ D. 장비 유지보수    │
   │    Management   ★   │   │   (Measurement Run) │   │   (Data & Monitor)  │   │   (Maintenance)     │
   └─────────┬───────────┘   └─────────┬───────────┘   └─────────┬───────────┘   └─────────┬───────────┘
             │ (L3 flow)               │ (L3 flow)               │ (L3 flow)               │ (L3 flow)
             ▼                         ▼                         ▼                         ▼
L3   ① 의뢰 접수             ① 로트 접수               ① 데이터 적재            ① 정기 PM
        │                       │                         │                        │
        ▼                       ▼                         ▼                        ▼
     ② Recipe Creation ★★    ② 자동 측정 Run          ② 안정성 모니터링 ◇      ② 상태 점검
        │  (= 셋업 ②~⑥)       │                         │                        │
        ▼                       ▼                         ▼                        ▼
     ③ 검증·등록             ③ 결과 수집·1차 판정      ③ 이상 알람·리포트       ③ 캘리브레이션
        │
        ▼
     ④ 유지·개정

  ★  = 1차 PoC 대상 L2
  ★★ = 1차 PoC 대상 L3 (Recipe Creation, 내부 셋업 ②~⑥)
  ◇  = 차기(2차) 대상 L3 (안정성 모니터링)
```

- L2-A의 L3②(Recipe Creation)는 03의 SOP에서 **[B] Recipe 셋업 ①~⑦**에 해당하며, 그중 AI 수행 범위는 **②~⑥**이다 (출처: [03 §2](../03-적용_CDSEM기획.md), 셋업 ①~⑦ 중 병목 ③④⑥·최대병목 ⑥; AI 수행=②~⑥, 사람=① 검토·⑦ 최종승인).
- L2-C의 L3②(안정성 모니터링)는 03의 **[D] 모니터링·이상 대응**에 대응하며, 사람 주기점검·야간/주말 의존으로 **운영 공백**이 큰 구간이다 (출처: [03 §2 Phase1](../03-적용_CDSEM기획.md)).
- L2-B 양산 Run은 등록된 Recipe로 자동 실행되어 *수작업 비중이 낮다* → 자동화 우선순위가 낮다 (출처: [03 §2](../03-적용_CDSEM기획.md), "[C] Run은 자동 실행되어 수작업 비중 낮음").

### B. AI 대체 판정 오버레이

**판정 범례**

| 아이콘 | 의미 |
|--------|------|
| 🤖 | AI 대체 (반복·정형·고공수, AI가 주수행) |
| 🤝 | 부분 대체 (AI 보조 + 사람 판단 병행, HITL) |
| 🧑 | 사람 (판단·승인·물리작업, AI 부적합) |
| ⚙️ | 기존 자동 (이미 장비/시스템이 자동화, 신규 AI 불필요) |

**판정 기준** = 반복성 · 정형성 · 공수 · 데이터 가용성 (출처: [원문 06](../source/06-sales-ai-opportunity.md) "반복성·정형성·공수가 큰 Task 지점 선별"). 아래 표의 L2 구획·L3 목록은 엔지니어 확정(2026-06-24).

#### L2-A. Recipe Management ★

| L3 | 판정 | 근거 (반복성·정형성·공수·데이터가용성) |
|----|:----:|------|
| ① 의뢰 접수 | 🤝 | 의뢰서 정보 충분성 검토는 정형성 중간·공수 중간. **부족정보 플래깅**만 AI 보조, 최종 접수 판단은 사람. MIDAS 전산화는 외부 변수 (출처: [03 §2](../03-적용_CDSEM기획.md), MIDAS는 별도 프로젝트). |
| ② **Recipe Creation ★★** (셋업 ②~⑥) | 🤖 | **반복성 高·정형성 高·공수 最大·데이터 가용 高**(SEM 이미지 이미 축적). ③④=병목, ⑥=최대병목. SEM 화면 판독+GUI 제어가 반복적·정형적 → **1차 PoC 대상** (출처: [03 §2 병목 ③④⑥·최대 ⑥](../03-적용_CDSEM기획.md), 데이터 가용성은 [원문 06] 선별기준). |
| ③ 검증·등록 | 🤝 | Recipe 저장·등록은 정형적이나, **최종 검증·승인은 사람**(⑦) 유지. AI는 자가점검 리포트 보조. |
| ④ 유지·개정 | 🧑 | 공정 변경·드리프트 기반 개정 *판단*은 비정형·맥락 의존 → 사람. (보정 *실행*은 ②로 회귀 시 AI 재사용) |

#### L2-B. 측정 실행 (Measurement Run)

| L3 | 판정 | 근거 |
|----|:----:|------|
| ① 로트 접수 | ⚙️ | 로트 스케줄링·큐잉은 기존 시스템이 자동 처리. |
| ② 자동 측정 Run | ⚙️ | 등록 Recipe로 장비가 자동 측정 → **이미 자동화**, 신규 AI 불필요 (출처: [03 §2](../03-적용_CDSEM기획.md), "Run 자동 실행·수작업 낮음"). |
| ③ 결과 수집·1차 판정 | 🤝 | 정상/이상 1차 판정 일부는 룰/임계로 자동, 경계 사례는 사람. AI 보조 여지 있으나 1차 범위 밖. |

#### L2-C. 데이터·모니터링 (Data & Monitoring)

| L3 | 판정 | 근거 |
|----|:----:|------|
| ① 데이터 적재 | ⚙️ | 측정 결과 DB 적재는 기존 파이프라인이 자동. |
| ② **안정성 모니터링 ◇** | 🤝 | 재현성·드리프트·장비상태 추세 감시. 반복성·데이터가용성 高이나 이상 *원인 분석*은 사람 판단 병행 → **차기(2차) 대상**(운영 공백: 사람 주기점검·야간/주말 의존) (출처: [03 §2](../03-적용_CDSEM기획.md), D-1 모니터링 공백). |
| ③ 이상 알람·리포트 | 🤝 | 알람 트리거는 룰 자동화 가능, 원인 구분(오인식 vs 장비 vs 공정)은 사람 협업. |

#### L2-D. 장비 유지보수 (Maintenance)

| L3 | 판정 | 근거 |
|----|:----:|------|
| ① 정기 PM | 🧑 | 물리적 점검·부품 교체 → 사람(현장 작업). AI 부적합. |
| ② 상태 점검 | ⚙️ | 장비 자체 진단 로그·인터록이 기존 자동. |
| ③ 캘리브레이션 | 🧑 | 기준 시료 기반 물리 보정·승인 → 사람. |

### C. 범위 결정 (Scope Decision)

```
1차 PoC 대상  =  L2-A › L3② Recipe Creation  (내부 셋업 ②~⑥)
                 └ 근거: 반복성·정형성·공수 最大 + SEM 이미지 데이터 이미 가용
                 └ 최대 병목 ⑥(오인식 수정 반복) 집중 → VLM GUI-제어 + 실시간 좌표 재정합

차기(2차) 대상 =  L2-C › L3② 안정성 모니터링
                 └ 근거: 운영 공백(사람 주기점검·야간/주말 의존), 데이터 가용성 高
                 └ 1차 안정화 후 Passive→Active 확장 단계에서 착수

범위 밖(이번)  =  L2-B(이미 자동) · L2-D(물리 작업) · A①/A③/A④(사람 판단·승인)
```

- **1차 대상 = L2-A › L3② Recipe Creation(②~⑥)**: AI(🤖)가 SEM 화면을 판독하고 GUI를 직접 제어해 Recipe를 생성하며, 실측 시작 시 좌표 shift·align 실패·오인식을 실시간 재정합한다. ①(검토)·⑦(최종승인)은 사람(🧑) 유지 — **완전자동화 아님, Human-in-the-loop** (출처: [SPEC §0](./_internal/SPEC.md) 고정 과제 정의).
- **차기 대상 = L2-C › L3② 안정성 모니터링**: 1차 셋업 자동화가 안정화된 뒤, 운영 공백(야간/주말 모니터링)을 메우는 2차 확장 후보.
- L2 4분할과 각 L3 경계는 **엔지니어 확정(2026-06-24)** — 계측(MI) 운영 프로세스 위계로 고정. 이후 산출물(과제 정의·To-Be·일정)은 이 범위 위에서 작성된다.
