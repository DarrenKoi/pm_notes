# 10 · 개발 일정 수립 (Step12 + Execution 흡수)

> 1년 내(2027) 구현을 목표로, KSF 우선순위를 분기 칸반 보드로 환산하고 분기말 Stage-Gate와 전 구간 변화관리를 박은 **실행 로드맵**.

## 왜 필요한가 (Why)

[04 Step12](../04-적용_CDSEM기술.md)는 Phase 0~3을 한 줄짜리 흐름으로만 잡아 두었다. 본 문서는 그 골격을 **분기 단위로 풀어, 누가·무엇을·어떤 순서로 시작하고, 어디서 멈춰 판단하는지**까지 채운 실행 가능한 일정이다. 단계별 한 줄을 일정으로 옮기지 못하면 "내년 실행 가능"이 성립하지 않는다.

세 가지를 명시적으로 박는다:

- **Stage-Gate 안전장치** — Q1 말 Phase 0(좌표 재정합 PoC, KSF#1)이 수용 기준을 못 넘으면 GUI 제어 개발에 자원을 쓰기 전에 과제를 재설계한다. 먼저 깨질 곳을 먼저 친다(No-Regret). (05 PoC 판단 게이트 인용)
- **변화관리의 상시화** — 운영팀 협업·엔지니어 신뢰·HITL 승인 UI는 특정 분기 작업이 아니라 전 구간 상시 활동이다. 이것이 Execution(실행/변화관리) 층의 흡수다. (04 Step11 조직 제약 인용)
- **KSF 4관점과의 정렬** — 칸반 테마 레인이 [#09 KSF 4관점](./09-ksf-제약-정의.md)(업무 프로세스 / 데이터 / 시스템·인프라 / 조직 문화)과 1:1로 맞아, 일정이 성공조건을 빠짐없이 덮는다. (SPEC §3 합의)

## 핵심 산출 (What)

| 산출물 | 내용 | 형식 |
|--------|------|------|
| **분기 칸반 보드** | Q1~Q4 × 테마(프로세스·데이터·AI 모델·Test) 셀 채움 | 마크다운 표 |
| **🚦 게이트 레인** | 분기말 의사결정 게이트 = 다음 분기 착수조건 (Q1말=Stage-Gate) | 칸반 행 + 표 |
| **👤 변화관리 레인** | 운영팀 협업·신뢰·HITL UI = 전 구간 상시 (Execution 흡수) | 칸반 행 |
| **실행 로드맵 노트** | 의사결정 게이트·오너십(워크스트림 리드) | Execution 층 |

- 분기 정의(04 Step12 인용): **Q1=Phase 0 PoC**(좌표 재정합 KSF#1) / **Q2=Phase 1** 단일 layer(②~⑤+⑥ 부분) / **Q3=Phase 2** ⑥ 본격+핸드오프(KSF#3) / **Q4=Phase 3** layer 확장·운영 이관(KSF#4).
- 소요·인력은 캠프에서 엔지니어가 확정하는 **placeholder**다(τ/s/r/N, `<담당 임원>`, 워크스트림 리드). 본 문서는 일정의 *형식·순서·게이트*만 고정한다. (SPEC §1.5)

## 작성 결과 (How) — 칸반 보드 + 게이트 + 변화관리

### A. 분기 칸반 보드

각 테마 레인은 #09 KSF 4관점과 1:1로 정렬된다(괄호 안 매핑).

| 테마 (레인) | **Q1 — Phase 0 PoC**<br>(좌표 재정합·KSF#1) | **Q2 — Phase 1**<br>(단일 layer ②~⑤+⑥부분) | **Q3 — Phase 2**<br>(⑥ 본격+핸드오프) | **Q4 — Phase 3**<br>(layer 확장·운영 이관) |
|---|---|---|---|---|
| **프로세스**<br>(↔ KSF 업무 프로세스) | As-Is/To-Be 흐름 확정 · 임계값(θ/τ/r) 정의·합의 | ②~⑤ 자동 수행 구현 · ⑦ 사람 승인 게이트 적용 | ⑥ 재정합 루프 + 예외 핸드오프(임계 초과→🧑) 도입 | 운영 이관 · SOP 정립 · 의뢰서 입력 표준화 연계(MIDAS, 외부변수) |
| **데이터**<br>(↔ KSF 데이터) | PoC용 SEM 이미지 수집 · 숙련자 라벨링 · hard set 구성 | 캡처+VLM 추출 파이프라인 구축 · BGE-M3 인덱스 | 데이터셋 확장 · 오인식/핸드오프 피드백 수집·반영 | 운영 데이터 축적 · 재학습셋 구성 |
| **AI 모델**<br>(↔ KSF 시스템·인프라) | 좌표 재정합 PoC 파이프라인 (Qwen3-VL-8B vs 30B 비교) | computer-use 통합(②~⑤) · GUI 자동제어 연동 | Qwen3-VL-30B 정밀(⑥) · 신뢰도 스코어·룰엔진 | 일반화(배율·노이즈) · 8B 경량화 · layer 확장 |
| **Test**<br>(↔ 검증 전반) | 섀도우 환경 구축 · 측정 프로토콜 확정(τ/s/r/N) | 단일 layer E2E 오프라인 검증 | 예외 핸드오프 검증 · 수용 게이트(recall/false-alarm) | 제한 파일럿(실 장비) · 회귀 테스트 |

> 범례: 모델 레인 ②~⑥은 [#08 To-Be](./08-to-be-프로세스.md) 적용범위와 일치(Q2=②~⑤, Q3=⑥). 불일치 시 설계오류. **완전자동화 아님** — ⑦ 사람 승인은 Q2부터 끝까지 유지.

### B. 🚦 게이트 레인 (분기말 의사결정 = 다음 분기 착수조건)

| 게이트 | 시점 | 판정 기준 | 통과 시 | 미달 시 |
|---|---|---|---|---|
| **🚦 G0 (Stage-Gate)** | Q1 말 | Phase 0 PoC: P95 오차 ≤ τ **그리고** 오인식 recall ≥ r (05 게이트 인용) | Q2 착수 (GUI 제어 통합) | **과제 재설계** — fine-tune / 전용 vision / hybrid 검토 (자원 투입 전 중단) |
| **🚦 G1** | Q2 말 | 단일 layer ②~⑤ E2E 오프라인 통과 · ⑦ 승인 게이트 동작 | Q3 착수 (⑥ 본격) | 적용 layer/패턴 좁혀 진행 + 원인 분석 (05 CONDITIONAL) |
| **🚦 G2** | Q3 말 | ⑥ 재정합 + 예외 핸드오프 수용 게이트 충족(recall/false-alarm 기준) | Q4 착수 (확장·이관) | 핸드오프 임계(τ/r) 재조정 후 재검증 |
| **🚦 G3** | Q4 말 | 제한 파일럿 회귀 통과 · 운영 SOP 정립 | 운영 이관 완료·확장 의사결정 | 파일럿 범위 유지·운영팀과 안정화 |

> **G0가 핵심 안전장치다.** 좌표 재정합(KSF#1)이 안 되면 GUI 제어가 완벽해도 ⑥ 자동화는 성립하지 않으므로, Q1 끝에서 싸고 빠르게 판단한다. (05 §5 판단 게이트 인용)

### C. 👤 변화관리 레인 (전 구간 상시 — Execution 흡수)

| 항목 | 전 구간(Q1~Q4) 상시 활동 |
|---|---|
| **운영팀 협업** | 장비 운영팀과 섀도우 환경·실장비 검증 일정 합의, 안전 인터록 운영 (04 조직/정책 제약 인용) |
| **엔지니어 신뢰** | 숙련 엔지니어의 라벨·피드백 루프 유지, 설명가능 리포트 공유, 단계적 도입으로 거부감 완화 |
| **HITL 승인 UI** | ⑦ 사람 최종 승인·예외 핸드오프를 받는 Human-in-the-loop 승인 인터페이스 — Q2 도입 후 전 구간 운영·개선 |

> 변화관리는 분기 작업이 아니라 모든 분기에 걸친 상시 레인이다. #09 조직 문화 KSF(⑦ 사람 승인 유지·운영팀 협업·단계적 도입)와 동일 출처. (04 Step11 / #09 인용)

### D. 실행 로드맵 노트 — 게이트·오너십 (Execution 층)

- **의사결정 게이트**: 각 분기는 직전 게이트 통과를 착수 전제로 한다(G0→Q2, G1→Q3, G2→Q4, G3→이관). 게이트는 일정이 아니라 **자원 투입 결정점**이다. Champion = `<담당 임원>` (미정).
- **오너십(워크스트림 리드 — placeholder)**: 테마 레인별 단일 책임자를 둔다.
  - 프로세스 리드 `<리드>` · 데이터 리드 `<리드>` · AI 모델 리드 `<리드>` · Test/검증 리드 `<리드>` · 변화관리 리드(운영팀 접점) `<리드>`
  - 각 리드는 자기 레인의 분기 산출과 게이트 근거 데이터를 책임진다.
- **RACI (책임 매트릭스 — placeholder)**: 오너십을 리드 지정에서 한 단계 더 굳힌다.
  - **R(실행)** 워크스트림 리드(프로세스/데이터/AI모델/Test) · **A(승인)** Champion `<담당 임원>` · **C(자문)** 데이터 오너·모델 오너·**보안 승인자**·장비 운영팀 · **I(공유)** 현업 엔지니어.
  - 특히 **보안 승인자·운영팀**은 #09 안전·보안 보강(인터록·롤백·감사로그)의 게이트 통과 책임을 진다.
- **Kill 기준 (Exit) — sunk-cost 방지**: 게이트 재설계가 **누적 2회**를 넘으면 '재설계'가 아니라 **과제 중단(Kill)** 을 판정한다. Kill 결정 주체 = Champion `<담당 임원>`. ('재설계'로만 회피하면 좀비 과제가 자원을 잠식 — 04/05 게이트 보강.)
- **소요·인력**: 분기별 공수·인원은 캠프에서 확정(placeholder). 본 문서는 순서·게이트·오너십 구조만 고정.
