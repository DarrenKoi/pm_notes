## 이 디렉터리가 무엇인가

SK Hynix "New AI Design Camp" 방법론을 **표준**으로 고정하고, 그 위에 부문 AIX POC 과제를 **프로젝트 폴더 단위**로 쌓는 **문서 저장소**. 코드·빌드·테스트는 없다(예외: `tools/` md→pptx 변환기) — 산출물은 전부 마크다운이다.

## 3축 구조 (가장 먼저 이해할 것)

```
lectures/                  [표준]  New AI Design Camp V2.7 완본 덱 + 편집 Template + captures/(캡처 10장 전사)
                                   → 팀을 가이드하는 기준점. 모든 틀·적용 문서의 1차 근거.
_가이드/                   [가이드] 재사용 방법론 틀 (01 기획 · 02 기술) + 00-템플릿(도메인 중립 키트)
프로젝트_<이름>/           [프로젝트] 틀을 실제 과제로 채운 폴더. 현재: 프로젝트_smart_align_agent (CD-SEM Align Fail 자동화)
tools/                             md → pptx 변환기
```

## 문서 세트의 목적

이 산출물의 최종 목표는 **내년(2027)에 실제로 실행할 수 있는 기획 문서**다. 따라서:

- **기획 문서가 중심**, **기술 문서가 뒷받침**한다. 기획(왜·무엇을·어떻게 실행)을 두껍게, 기술(To-Be 설계·PoC 실험 설계)은 근거로 받친다.
- **테스트 실행은 범위 밖이다.** 실제로 PoC를 돌려 결과를 내지 않는다 — 설계·계획 수준까지만 쓴다. 실측 수치·실험 결과를 지어내지 말고 엔지니어 확정 플레이스홀더(예: `<담당 임원>`, τ/s/r/N)로 둔다.
- "내년 실행 가능"이 되려면 4 Step 중 **Validation(가치 검증)·Execution(실행 로드맵)** 까지 채워져야 한다. Discovery만으로는 실행 문서가 아니다.

## 콘텐츠 아키텍처 (가장 중요)

세 계층이 명확히 분리되어 있고, 이 경계를 유지하는 것이 핵심이다.

```
lectures/captures/01~10 + lectures 완본 덱  (원문 전사·표준, 불변)
      │  faithful transcription — 캡처 1장 = 파일 1개, 내용 가공 금지
      ▼
_가이드/01·02 (방법론 "틀")     ← 프레임워크. 일반론. 재사용 가능.
      │  01=기획(Track A Discovery), 02=기술(Track B To-Be 설계)
      ▼
프로젝트_<이름>/03·04… (적용 "사례")  ← 틀을 실제 과제로 채운 인스턴스 (예: CD-SEM = Smart Align Agent)
```

- **`lectures/captures/`** = 원문 전사. 캡처 이미지를 그대로 옮긴 것이며 **재해석·요약·수정하지 않는다.** frontmatter에 원본 파일명(`source:`)과 `type: source-transcription`을 단다. `lectures/`가 상위·완본 출처다.
- **`_가이드/01·02`** = New AI Design Camp **방법론 틀**. 일반적·재사용 가능한 프레임워크. 특정 과제 내용을 여기에 섞지 않는다.
- **`프로젝트_<이름>/`** = 그 틀을 **실제 ITC AIX 과제로 채운 사례**. 01·02의 구조(Why→What→How, 산출물 체크리스트)를 **거울처럼** 따르되 모든 칸을 실제 값으로 채운다. 적용 문서 파일명은 일관성을 위해 `NN-적용_<과제>기획.md` / `NN-적용_<과제>기술.md` 형식(소문자-하이픈 규칙의 예외).

## 신규 프로젝트 추가 규칙

새 AIX 과제는 **`프로젝트_<이름>/` 폴더**로 분리한다.

1. `lectures/`(표준)와 `_가이드/01`(기획)·`_가이드/02`(기술)를 먼저 읽고 섹션 구조를 미러링한다 — 새 구조를 발명하지 않는다.
2. 폴더 안에 `README.md`(프로젝트 인덱스: 정의·문서 목록·읽는 순서)를 둔다.
3. 표준·틀은 모든 프로젝트가 공유하고, 과제별 내용은 `프로젝트_*/` 안에만 둔다 — 경계를 섞지 않는다.

## 방법론 골격 (문서가 따르는 사고 흐름)

```
Baseline 3원칙(가치·데이터·인간 중심)
  → 4 Step: Discovery → Design → Validation → Execution
    → 12-Step Design Camp
        Track A. Idea 도출(1~7)   = _가이드/01 기획문서가 다룸
        Track B. 실행 구체화(8~12) = _가이드/02 기술문서가 다룸
```

핵심 산출물 3종이 모든 적용 문서에 반복 등장한다: **조직목표↔과제 연계표**, **L1~L5 프로세스 분해표**, **Pain Point → 근본원인(3-Why) → 해결 아이디어표**. Track B는 여기에 **AI 과제 정의서(1과제=1매)**, **To-Be Swimlane(🤖 AI / 🧑 Human 구분)**, **KSF·제약(기술/조직/비용/정책·보안 4측면)** 을 더한다.

## 작업 규칙

- **기획 우선**: 기획 문서를 중심으로 두껍게 쓰고, 기술 문서는 그 근거로 받친다. 목표는 내년 실행 가능한 문서이지 실험 보고서가 아니다.
- **테스트 실행 금지**: 실제 PoC를 돌려 결과를 내지 않는다. 설계·계획까지만 쓰고, 미확정 수치는 플레이스홀더로 둔다.
- **새 적용 문서 작성 시**: 대응하는 틀 문서(기획→`_가이드/01`, 기술→`_가이드/02`)를 먼저 읽고 섹션 구조를 그대로 미러링한다.
- **출처 표기**: 적용 문서의 모든 사실은 `lectures/captures/NN`을 인라인으로 인용하거나(`(출처: [원문 NN](../lectures/captures/NN-*.md))`) 브레인스토밍 합의 결과임을 명시한다.
- **회사 기밀 제외**: 구체 장비 수치·내부 시스템 상세는 넣지 않고 방법론 적용 골격만 남긴다 (상위 CLAUDE.md ⚠️ 주의사항).
- **문서 완료 후**: 해당 `프로젝트_*/README.md`와 루트 `README.md`의 구성 표를 갱신한다.
- **사내 LLM 제약** (기술 문서에 반영): 외부 LLM API 전면 차단. 사내 OpenAI-호환 엔드포인트만 사용 — Kimi-K2.5(텍스트), Qwen3-VL-8B/30B(비전), BGE-M3(임베딩). 99% 사내 문서가 DRM이라 스크린샷+VLM 추출이 기본 파이프라인.
